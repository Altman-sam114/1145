# v3.6（AI高价值海军护航门槛）

## 版本号

v3.6

## 版本分配依据

- 当前 `update_log.md` 和 `md/prompt/v3（海战反潜）/` 最新版本为 v3.5「收紧 AI 侦察反潜目标」。
- v3.0-v3.5 已围绕反潜 HUD、AI ASW、战场反馈、声呐覆盖和 AI 侦察公平性完成多轮小步。
- v3.6 延续 v3 海战反潜 / 海军 AI 阶段，聚焦 AI 编队比例和高价值海军保护，只调整 Red 主攻 attack-move 波次成员筛选。

## 背景

当前 Red AI 在 `commandEnemyAttackers(_:)` 中把所有 idle combat unit 汇总后，按距离和 `formationPriority(for:)` 取一批单位进入主攻 attack-move。Carrier 和 Battleship 虽然成本、战略价值、射程和生产意义都较高，但当前没有护航门槛，可能作为孤立单位被常规主攻波次拉走。

这会削弱 Desert Stormfront 风格 RTS 中「高价值海军需要护航推进」的战术感。v3.6 要做一个小而可验收的 AI 行为增量：Carrier / Battleship 仍可生产、开火、防守、被维修、作为目标评分对象，但在 routine attack-move 主攻波次中，缺少护航或波次规模不足时暂不加入，留待后续波次或本地防守。

## 目标

1. Red routine attack-move 主攻波次加入高价值海军护航门槛。
   - 高价值海军定义为 Red 的 `.carrier` 和 `.battleship`。
   - 它们不应在缺少护航时作为孤立单位进入 `issueFormationMove(... attackMove: true)` 的主攻 wave。
2. 门槛只作用于 `commandEnemyAttackers(_:)` 的 routine attack-move wave。
   - 不影响玩家 `AMOV`。
   - 不影响 Red 旗点防守响应、基地防守响应、出厂 tactical target、普通战斗索敌、生产或维修。
3. 可接受的加入条件。
   - 若 provisional wave 中已有足够非高价值可战斗单位，可允许 high-value naval 加入。
   - 或 high-value naval 附近一定距离内存在足够非高价值且同属本轮 provisional wave 的护航单位，可允许加入。
   - Carrier 门槛应不低于 Battleship；建议 Carrier 至少需要 2 个 escort 或足够大 wave，Battleship 至少需要 1 个 escort 或足够大 wave。
4. 缺护航时的行为。
   - 高价值海军保持 idle，不设置 `attackMoveDestination`，不清空现有 target，不强行移动。
   - 后续 AI 周期如果护航满足，可以加入下一波。
5. 同步文档。
   - `README.md` 的 AI 功能描述补充高价值海军需要护航或足够波次才加入 routine attack-move。
   - `md/flow/flow.md` 的 AI 小节说明该门槛。
   - 必要时 `md/flow/flowchart.md` 的 AI 节点补充「高价值海军护航门槛」。

## 非目标

- 不改单位/建筑数值、成本、血量、速度、射程、伤害、冷却或视野。
- 不改 `aiBuildPattern()`、`queueBuild(...)`、`productionSource(...)` 或任何生产逻辑。
- 不改 `issueFormationMove(...)` 的共享玩家 / AI 编队执行逻辑。
- 不改玩家 `issueAttackMoveOrder(...)` 或任何 HUD 操作。
- 不改 `tacticalTarget(for:)`、`targetScore(for:target:)`、`attackDestination(for:target:)`、`firingPosition(...)`。
- 不改 `updateMovement(dt:)`、`updateCombat(dt:)`、`fire(...)`、`canAttack(...)`、迷雾、声呐、潜艇检测或支援技能。
- 不引入外部依赖、素材、XCTest target、长期 AI 状态、PR 流或非 `main` 分支。
- 不更新 `update_log.md`，由 Agent C 在 artifact 验收通过后追加正式记录。

关键源码事实：

- `updateAI(dt:)` 在生产、占点和支援后，满足 `shouldAttack` 时调用 `commandEnemyAttackers(enemyUnits)`。
- `commandEnemyAttackers(_:)` 是 routine attack-move 主攻波次入口；当前先生成 `idleAttackers`，然后有 objective 时计算 `waveSize`、排序、取 `wave`，最后调用 `issueFormationMove(to:objective, units:Array(wave), showMarkers:false, showFeedback:false, attackMove:true)`。
- `issueFormationMove(...)` 是玩家和 AI 共享的移动 / attack-move 执行函数，不应放入 Red 专属门槛。
- `formationPriority(for:)` 只负责编队排序，不等于是否允许入波。
- `triggerEnemyControlPointDefenseIfNeeded(...)` 和基地防守响应是防守逻辑，不是 routine attack-move 波次。
- `tacticalTarget(for:)` 与 `targetScore(...)` 用于目标选择，不应用来决定高价值海军是否加入主攻 wave。

## 实现步骤

1. 开始前同步 `main`：

```sh
git fetch origin
git switch main
git pull --ff-only origin main
git status --short
git rev-parse HEAD
git rev-parse origin/main
```

要求当前分支为 `main`，目标远端为 `origin/main`，工作区无无关改动。

2. 在 `commandEnemyAttackers(_:)` 附近增加小 helper，不新建平行 AI 系统。
   - 推荐 helper：
     - `private func isHighValueNavalAssaultUnit(_ unit: GameEntity) -> Bool`
     - `private func enemyAssaultWave(from provisionalWave: [GameEntity]) -> [GameEntity]`
     - `private func canJoinEnemyAssaultWave(_ unit: GameEntity, provisionalWave: [GameEntity]) -> Bool`
     - `private func enemyEscortCount(near unit: GameEntity, in candidates: [GameEntity]) -> Int`
   - helper 名称可调整，但必须表达清楚。
3. 形成 provisional wave 后再过滤。
   - 先保留现有排序和 `waveSize` 计算。
   - 对 `idleAttackers` 先排序并取 `waveSize` 作为 provisional wave。
   - 从 provisional wave 中过滤 high-value naval：满足护航门槛则保留，不满足则剔除。
   - 非 high-value combat unit 保持原有入波行为。
   - 如果过滤后 wave 为空，直接 return，不强行派出高价值海军。
4. 设计护航门槛。
   - Escort 必须是 Red、alive、非结构、operational、有伤害、非占点 reservation、非 high-value naval。
   - Escort 必须来自本轮 provisional wave，避免高价值海军因为附近有不会随行的 idle 单位而孤立出击。
   - 建议 escort 半径保守，例如 520-640 world units，避免全图护航。
   - Carrier 推荐需要至少 2 个 escort；Battleship 推荐至少 1 个 escort。
   - 可加入 wave-size bypass：如果 provisional wave 中非 high-value combat unit 已达到 `aiDifficulty.attackGroupSize`，允许 high-value naval 加入，避免大规模进攻时永远缺席。
5. 保持旧行为边界。
   - 不改变 `shouldAttack`。
   - 不改变 no-objective fallback 中 direct tactical target 逻辑，除非明确只对 routine wave 有 objective 的分支。
   - 不改变 `issueFormationMove(...)`、`formationPriority(for:)`、`setDestination(...)`、combat / fog / support / production。
6. 同步文档。
   - README 更新 AI 功能描述。
   - flow 更新 AI 小节。
   - flowchart 的 AI 节点可简短补充高价值海军护航门槛。
   - 不更新 `md/test/test.md`。
   - 不更新 `update_log.md`。
7. 可以使用 `git diff`、`git diff --name-only`、`git diff --stat` 自查改动范围，但不要把它们写成验收通过。

## 验收标准

- `commandEnemyAttackers(_:)` 的 objective attack-move wave 不会让孤立 Carrier / Battleship 单独进入 routine assault。

- 玩家 `AMOV`、Red 防守响应、出厂 tactical target、生产、寻路、战斗、迷雾、支援、AI ASW / 防空逻辑没有语义变化。

## 风险和禁止项

- 风险：把门槛放到 `issueFormationMove(...)`，会影响玩家命令和其他 AI 用途。禁止这样做。
- 风险：把门槛放到 `shouldAttack`，会阻断整个 AI 主攻。禁止这样做。
- 风险：把 Carrier / Battleship 永久排除出进攻，导致海军永远不参战。必须允许满足 escort / wave 条件时加入。
- 风险：把 production、target score、combat 或 pathfinding 一起改掉，扩大范围。禁止这样做。
- 禁止修改单位数值、AI build pattern、支援技能、迷雾、潜艇检测、玩家命令、Xcode project 或 workflow。
- 禁止使用非 `Altman-sam114` 的 GitHub 账号伪装 push、CI 或 artifact 验收。
