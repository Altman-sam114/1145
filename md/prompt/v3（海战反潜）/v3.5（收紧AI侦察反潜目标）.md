# v3.5（收紧AI侦察反潜目标）

## 版本号

v3.5

## 版本分配依据

- 当前 `md/prompt/` 和 `update_log.md` 最新海战反潜阶段为 v3.4「声呐覆盖选择反馈」。
- v3.0-v3.4 已补反潜 HUD、AI 已知潜艇压力补 ASW、ASW 命中反馈和声呐覆盖选择反馈。
- v3.5 继续 v3 海战反潜阶段，只收紧 Red AI `SCAN` 反潜选点的认知边界，不改变 sonar、潜艇暴露、支援半径/时长、生产或单位数值。

## 背景

当前 `GameScene.swift` 中 `enemyReconSupportPoint()` 会筛选玩家潜艇并返回目标坐标。现状问题是：函数直接选择 Red 未知的玩家潜艇真实坐标，用 `SCAN` 去扫该点。这会让 Red AI 读取本不应知道的隐藏潜艇位置，绕过 v3.2 已建立的 Red 当前认知边界。

项目当前 Red 对玩家潜艇的合法认知边界：

- `isKnownToFaction(entity, observer: .enemy)` 负责潜艇是否被 Red 侦测。
- `isPointKnownToEnemySensors(entity.node.position)` 负责该点是否在 Red 存活单位、Red 已完工建筑或 Red 已占旗点视野边界内。
- v3.2 的 `isKnownPlayerSubmarineThreat(_:)` 已要求玩家潜艇同时满足上述两个条件，才计入 Red 已知潜艇威胁。

本轮要把 Red `SCAN` 的反潜选点收紧到同一套 Red 合法认知上：已知潜艇可以作为精确目标，未知潜艇不能被读取真实坐标。若没有已知潜艇，Red 仍可基于自己已占旗点和己方 operational sonar sensor 附近的海岸/水域热点做巡扫。

## 目标

1. 修复 `enemyReconSupportPoint()` 读取未知玩家潜艇真实坐标的问题。
   - Red `SCAN` 不得再选择 `!isKnownToFaction(entity, observer: .enemy)` 的玩家潜艇坐标。
   - 禁止用隐藏潜艇实体位置作为 fallback、排序依据或热点生成依据。
2. Red `SCAN` 反潜选点只允许基于 Red 合法认知。
   - 第一优先级：已知玩家潜艇。
   - 已知玩家潜艇必须同时满足 `entity.faction == .player`、`entity.kind == .submarine`、`entity.isAlive`、`isKnownToFaction(entity, observer: .enemy)` 和 `isPointKnownToEnemySensors(entity.node.position)`。
   - 该判断应复用或对齐 `isKnownPlayerSubmarineThreat(_:)`。
3. 若没有已知玩家潜艇，Red 可选择合法巡扫热点。
   - 巡扫热点只能来自 Red 已占旗点、Red 己方 operational sonar sensor 或其他 Red 自身可感知资产附近。
   - 推荐优先：Red 已占旗点附近的海岸/水域、己方 operational Sonar Buoy / Battleship / Carrier / Helicopter / Submarine sonar sensor 附近的海岸/水域。
   - 热点必须通过地图/地形规则生成，不得从未知玩家潜艇坐标反推。
4. 保持范围小。
   - 只改 AI recon support 选点 helper 及必要的小 helper。
   - 同步 `README.md`、`md/flow/flow.md` 和必要的 `md/flow/flowchart.md`。
   - 不更新 `update_log.md`，由 Agent C 在云端 artifact 验收通过后追加正式记录。
5. 实现后直接 push `main` 触发 GitHub Actions，由 Agent C 下载 artifact 验收。

- 当前 `enemyReconSupportPoint()` 筛选玩家潜艇且要求 `!isKnownToFaction(entity, observer: .enemy)`，然后直接返回该隐藏潜艇的真实坐标。
- `knownPlayerSubmarineThreatCount()` 和 `isKnownPlayerSubmarineThreat(_:)` 已定义 Red 已知潜艇语义，必须同时检查 `isKnownToFaction(..., observer: .enemy)` 和 `isPointKnownToEnemySensors(...)`。

- `EntityKind.hasSonar` 包含 Battleship、Submarine、Carrier、Helicopter、Sonar Buoy；Fighter 是 ASW attacker，但不是 sonar sensor。
- `isSubmarineDetected(...)` 已处理 `revealedUntil` 与 sonar sensor 检测，结构 sonar 必须 operational。
- `SCAN` 当前仍是 8 秒区域侦察暴露，不应在本轮改动效果本身。

2. 改造 `enemyReconSupportPoint()`。
   - 删除或替换当前选择 `!isKnownToFaction(..., observer: .enemy)` 玩家潜艇真实坐标的逻辑。
   - 第一优先级改为选择 Red 已知玩家潜艇，可直接筛选 `entities.values.filter(isKnownPlayerSubmarineThreat)`。
   - 若存在已知玩家潜艇，返回该已知潜艇当前位置作为 `SCAN` 点是允许的，因为它已在 Red 合法认知内。
3. 增加 fallback 巡扫热点。
   - 在没有已知潜艇时，`enemyReconSupportPoint()` 可从 Red 自身 anchor 生成巡扫点。
   - 合法 anchor 建议包括 Red 已占控制点和 Red 己方 active sonar sensor。
   - 从 anchor 周边采样候选点时，只能使用地图 tile、地形和 anchor 坐标。
   - 候选点应偏向水域或海岸，采样半径保持保守。
   - 如找不到合适水域/海岸热点，可返回 `nil`，让 AI 不放 `SCAN`；不要回退到未知潜艇坐标。
4. 尽量复用现有 helper，减少新状态。
   - 优先复用 `isKnownPlayerSubmarineThreat(_:)`、`isPointKnownToEnemySensors(_:)`、`tile(at:)`、`tileCenter(...)`、地形判断和现有距离工具。
   - 如新增 helper，命名需清晰，例如 `enemyReconPatrolHotspot()`、`enemyReconHotspotCandidates(around:)`、`isActiveEnemySonarSensor(_:)`。
   - 不引入新的全局状态、缓存、随机种子或长期 AI 记忆。
5. 保持支援执行链不变。
   - `tryEnemySupportPower()` 仍只调用 `enemyReconSupportPoint()` 并执行 `.reconSweep`。
   - 不修改 `executeSupportPower(...)`、`revealSupportTargets(...)`、support cost/cooldown/radius/duration。
6. 更新文档。
   - `README.md`：补充 Red `SCAN` 不读取未知潜艇坐标，侦察 fallback 基于已占旗点/己方 sonar 附近巡扫。
   - `md/flow/flow.md`：AI 小节说明 Red `SCAN` 反潜选点优先已知玩家潜艇；没有已知潜艇时只从 Red 已占旗点或 operational sonar sensor 周边生成海岸/水域巡扫热点。
   - `md/flow/flowchart.md`：AI 节点简短同步合法认知 `SCAN` 巡扫。
   - 不更新 `update_log.md`。
7. 最后可使用 `git diff`、`git diff --name-only`、`git diff --stat` 查看改动范围。这些命令只用于自查，不得写成验收通过。

## 验收标准

- `enemyReconSupportPoint()` 不再读取或返回 Red 未知玩家潜艇的真实坐标。
- Red `SCAN` 第一优先级只会瞄准已知玩家潜艇；已知条件必须同时满足 `isKnownToFaction(..., observer: .enemy)` 和 `isPointKnownToEnemySensors(...)`。
- 无已知玩家潜艇时，Red `SCAN` fallback 只基于 Red 已占旗点或己方 operational sonar sensor 等 Red 合法 anchor 周边的海岸/水域热点。
- fallback 巡扫热点不依赖隐藏玩家潜艇坐标、距离、存在性或排序。
- 找不到合法热点时可以不释放 `SCAN`，不得作弊回退到未知潜艇。
- `sonarRange(for:)`、`isSubmarineDetected(...)`、`revealedUntil`、`SCAN` 半径/持续时间/费用/冷却、AI ASW 生产、伤害、单位数值、玩家 `SCAN` 行为均无改动。
- README、flow、flowchart 与真实行为一致。
- 本轮提交已 push 到 `origin/main`，并触发 GitHub Actions。
- Agent C 能下载最新 `origin/main` run 的 artifact，并核对 manifest / JUnit / xcodebuild.log / failure summary / xcresult。

## 风险和禁止项

- 风险：为了让 AI 更积极使用 `SCAN`，不小心继续使用未知潜艇坐标作为 fallback。
- 风险：只检查 `isKnownToFaction(..., observer: .enemy)` 不检查 `isPointKnownToEnemySensors(...)`，会与 v3.2 的 Red 已知潜艇压力语义不一致。
- 风险：把 Fighter 当成 sonar sensor 生成热点。Fighter 是 ASW attacker，但不是 sonar sensor。
- 风险：把未完工 Sonar Buoy 当成 operational sonar sensor。结构类 sonar 必须 operational。
- 风险：改动 `SCAN` 支援效果本身，导致玩家 `SCAN` 或支援平衡变化。
- 禁止读取未知玩家潜艇坐标、距离或数量来决定 Red `SCAN` 精确落点。
- 禁止修改 `sonarRange(for:)`、`isSubmarineDetected(...)`、`revealedUntil`、`hasSonar`、`canAttack(...)`、支援技能数值、AI ASW 生产、单位/建筑数值或玩家命令。
- 禁止绕过 `GameScene.swift` 现有 AI / fog / support 状态流另建平行系统。
- 禁止运行本地测试、本机构建或本地 static check 作为验收依据。
