# v3.7（AI编队比例混编进攻）

## 版本号

v3.7

## 版本分配依据

- 当前 `update_log.md` 和 `md/prompt/v3（海战反潜）/` 最新版本为 v3.6「AI 高价值海军护航门槛」。
- v3.0-v3.6 已围绕海战 / 反潜可读性、AI ASW、公平侦察和 Red 高价值海军主攻保护完成多轮小步。
- v3.7 延续 v3 的 AI 主攻波次质量方向，聚焦 Red routine attack-move 波次的混编选择，不做完整 AI 角色系统。

## 背景

当前 Red AI 在 `commandEnemyAttackers(_:)` 中从 idle attackers 里按目标距离和 `formationPriority(for:)` 取一段最近单位作为 provisional wave，再套 v3.6 Battleship / Carrier 护航过滤。这能保证简单进攻，但当某一类单位离目标更近时，routine assault 容易形成单一兵种波次，降低 Desert Stormfront 风格 RTS 中「陆空海混编推进」的战术感。

v3.7 要做一个小而可验收的行为增量：Red 主攻波次仍然走同一个 attack-move 和 v3.6 护航链路，但 provisional wave 先尝试从可用 idle attackers 中带上前排、防空、远程、空军和海军支援，再用最近单位补齐。这样 AI 在拥有混合兵种时更容易发起混编进攻；在兵种不足时仍可按现有单位出击，不能卡死。

## 目标

1. Red routine attack-move 主攻波次优先混编。
   - 有 objective 时，`commandEnemyAttackers(_:)` 不再只按距离截取最近单位。
   - 先从 idle attackers 中优先挑选少量不同角色单位，例如前排陆军、防空、远程火力、空军支援、海军支援。
   - 再用按距离 / `formationPriority(for:)` 排序的剩余单位补齐 wave size。
2. 保留既有出击弹性。
   - 某类单位不存在、缺生产来源或暂时未造出时，不阻断主攻。
   - 如果只有单一兵种可用，仍允许该兵种按现有规则进攻。
3. 保留 v3.6 高价值海军护航门槛。
   - Battleship / Carrier 仍必须经过 `enemyAssaultWave(from:)` / `canJoinEnemyAssaultWave(...)` 过滤。
   - v3.7 只改变 provisional wave 的组成偏好，不绕过护航过滤。
4. 轻调 `aiBuildPattern()`。
   - 只允许调整固定轮转顺序 / 重复项，让 Easy / Normal / Hard 更稳定地产生陆军核心、AA、空军和海军支援。
   - 不绕过 `canQueueBuild(...)`、`queueBuild(...)` 或 `productionSource(...)`。
5. 同步文档。
   - `README.md` 的 AI 功能描述补充 routine assault 会优先混编。
   - `md/flow/flow.md` 的 AI 小节说明混编 provisional wave。
   - `md/flow/flowchart.md` 的 AI 节点简短补充「混编主攻波次」。

## 非目标

- 不新增单位、建筑、AI 长期角色 reservation 或全局战术状态。
- 不改单位 / 建筑成本、HP、速度、射程、伤害、冷却、视野或 XP。
- 不改动态防空 / ASW 生产逻辑，只允许常规 `aiBuildPattern()` 的小顺序调整。
- 不改 `issueFormationMove(...)` 的共享玩家 / AI 编队执行逻辑。
- 不改玩家 `issueAttackMoveOrder(...)`、HUD 操作、控制组或选择逻辑。
- 不改 `tacticalTarget(for:)`、`targetScore(for:target:)`、`attackDestination(for:target:)`、`firingPosition(...)`。
- 不改 `updateMovement(dt:)`、`updateCombat(dt:)`、`fire(...)`、`canAttack(...)`、迷雾、声呐、潜艇检测、支援技能或防守响应。
- 不引入外部依赖、素材、XCTest target、PR 流或非 `main` 分支。
- 不更新 `update_log.md`，由 Agent C 在 artifact 验收通过后追加正式记录。

关键源码事实：

- `updateAI(dt:)` 在生产、占点和支援后，满足 `shouldAttack` 时调用 `commandEnemyAttackers(enemyUnits)`。
- `aiBuildPattern()` 返回 Easy / Normal / Hard 的常规生产轮转；实际排产仍由 `nextAvailableAIBuildKind(...)`、`canQueueBuild(...)`、`queueBuild(...)` 和 `productionSource(...)` 限制。
- `commandEnemyAttackers(_:)` 是 routine attack-move 主攻波次入口；当前有 objective 时生成 `idleAttackers`、排序取 `provisionalWave`，再通过 `enemyAssaultWave(from:)` 过滤后调用 `issueFormationMove(... attackMove: true)`。
- v3.6 已新增 `enemyAssaultWave(from:)`、`canJoinEnemyAssaultWave(...)`、`isHighValueNavalAssaultUnit(...)`、`isEnemyAssaultEscort(...)` 和 `enemyEscortCount(...)`。v3.7 应保留这些 helper 并在其前面调整 provisional wave 组成。
- `issueFormationMove(...)` 是玩家和 AI 共享的移动 / attack-move 执行函数，不应放入 Red 专属混编逻辑。

## 实现步骤

1. 开始前同步 `main`：

```sh
git fetch origin
git switch main
git pull --ff-only origin main
git status --short
git rev-parse HEAD
git rev-parse origin/main
```

要求当前分支为 `main`，目标远端为 `origin/main`，工作区无无关改动。

2. 在 `commandEnemyAttackers(_:)` 附近增加小 helper，不新建平行 AI 系统。
   - 推荐 helper：
     - `private func enemyProvisionalAssaultWave(from candidates: [GameEntity], toward objective: CGPoint, waveSize: Int) -> [GameEntity]`
     - `private func sortedEnemyAssaultCandidates(_ candidates: [GameEntity], toward objective: CGPoint) -> [GameEntity]`
     - `private func appendEnemyAssaultRole(from candidates: [GameEntity], to selected: inout [GameEntity], limit: Int, where matches: (GameEntity) -> Bool)`
   - 也可使用等价 helper 名称，但必须保持局部、可读、无长期状态。
3. 用 helper 替换 `commandEnemyAttackers(_:)` 里原先的排序 + `prefix(waveSize)`。
   - 先得到按距离 / `formationPriority(for:)` 排序的 candidates。
   - 优先追加少量角色：
     - 前排陆军：Humvee / Tank。
     - 防空：AA Truck / Fighter。
     - 远程火力：Artillery / Battleship。
     - 空军支援：Helicopter / Fighter。
     - 海军支援：Submarine / Battleship / Carrier。
   - 每个角色只挑 1 个，前排可挑 2 个；具体数量可小幅调整，但不能超过 wave size。
   - 最后用剩余 ordered candidates 补齐到 wave size。
4. 继续调用 `enemyAssaultWave(from:)`。
   - v3.6 Battleship / Carrier 护航过滤必须继续生效。
   - 如果过滤后 wave 为空，保持现有 `return` 行为，不强行派出高价值海军。
5. 轻调 `aiBuildPattern()`。
   - Easy / Normal / Hard 的轮转可以更明确地穿插陆军核心、AA、空军、海军。
   - 不允许改队列来源、资金扣除、生产时间或直接生成单位。
6. 保持旧行为边界。
   - 不改变 `idleAttackers` 初筛条件。
   - 不改变 no-objective fallback 中 direct tactical target 逻辑。
   - 不改变 `shouldAttack`。
   - 不改变 `issueFormationMove(...)`、`formationPriority(for:)`、`setDestination(...)`、combat / fog / support / defense / production。
7. 同步文档。
   - README 更新 AI 功能描述。
   - flow 更新 AI 小节。
   - flowchart 的 AI 节点简短补充混编主攻波次。
   - 不更新 `md/test/test.md`。
   - 不更新 `update_log.md`。
8. 可以使用 `git diff`、`git diff --name-only`、`git diff --stat` 自查改动范围，但不要把它们写成验收通过。

## 验收标准

- `commandEnemyAttackers(_:)` 的 objective attack-move provisional wave 会优先尝试混入前排、防空、远程、空军和海军支援，而不是只取最近单位。

- 风险：角色规则太硬导致 AI 因缺少某类单位而不进攻。必须 fallback 到可用 units。

- 风险：绕过 v3.6 高价值海军护航过滤。禁止这样做。
- 风险：改 production source、combat、fog 或 support，扩大范围。禁止这样做。
- 禁止修改单位数值、动态 AA / ASW 逻辑、支援技能、迷雾、潜艇检测、玩家命令、Xcode project 或 workflow。
