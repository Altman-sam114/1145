# v3.8（AI高价值老兵主攻保护）

## 版本号

v3.8

## 版本分配依据

- 当前 `update_log.md` 和 `md/prompt/v3（海战反潜）/` 最新版本为 v3.7「AI 编队比例混编进攻」。
- v3.6-v3.7 已在 Red routine assault wave 中加入高价值海军护航和混编 provisional wave。
- v1.0 已完成老兵经验系统，Red 与玩家共享 XP、等级、有效伤害、冷却和视野加成。
- v3.8 延续 AI 主攻波次质量和高价值单位保护方向，聚焦受损 Veteran / Elite 单位是否参加 routine 主攻。

## 背景

Red AI 当前会通过直接火力击杀获得 Veteran / Elite 单位。这些单位拥有更高伤害、更快冷却和更远视野，是战场上的高价值资产。v3.7 的 mixed provisional wave 会主动挑选多类单位进入主攻，但没有区分老兵单位的当前 HP。低血 Veteran / Elite 如果被 routine attack-move 波次反复拉走，容易在没有必要的情况下损失。

本轮只做一个保守小增量：受损严重的 Red Veteran / Elite 机动战斗单位不参加 objective routine assault，留在 idle 状态，等待既有 mechanic 自动维修或 Red `REPR` 支援自然恢复；Red `REPR` 评分会轻微偏向这类单位，但不改变维修量、费用、冷却或玩家手动支援。HP 恢复到阈值以上后，它们可以按现有规则重新加入下一波。

## 目标

1. Red routine attack-move 主攻波次加入受损老兵保护。
   - 保护对象：Red、alive、非结构、operational、有伤害、Veteran 或 Elite，且 HP 比例低于阈值的机动战斗单位。
   - 建议阈值：`hp / maxHP < 0.55`。
2. 门槛只作用于 `commandEnemyAttackers(_:)` objective routine wave 的最终入波判断。
   - 不影响玩家 `AMOV`。
   - 不影响 no-objective fallback tactical target。
   - 不影响 Red 基地防守、旗点防守、普通战斗索敌、生产、维修、支援技能或迷雾。
3. 与 v3.6 / v3.7 兼容。
   - v3.7 mixed provisional wave 仍可先挑选该单位。
   - `enemyAssaultWave(from:)` / `canJoinEnemyAssaultWave(...)` 再过滤掉受保护老兵。
   - Battleship / Carrier 仍受 v3.6 护航门槛；若同时是低血 Veteran / Elite，也必须先被老兵保护过滤。
4. 受保护老兵行为。
   - 不进入本轮 `issueFormationMove(... attackMove: true)`。
   - 不设置 `attackMoveDestination`、`destination` 或 `attackTarget`。
   - 后续 AI 周期中 HP 恢复到阈值以上后，可重新加入主攻。
5. Red `REPR` 支援评分偏向受损老兵。
   - `supportRepairScore(...)` 在 `faction == .enemy` 且覆盖低血 Veteran / Elite 作战单位时增加少量价值评分。
   - 不改变 `SupportPower.fieldRepair` 的维修量、费用、冷却、半径、资产需求或玩家手动 `REPR` 行为。
6. 同步文档。
   - `README.md` 的 AI 功能描述补充低血 Veteran / Elite 不参加 routine assault。
   - `md/flow/flow.md` 的 AI 小节说明该门槛和 Red `REPR` 评分偏向。
   - `md/flow/flowchart.md` 的 AI 节点简短补充「受损老兵保护」。

关键源码事实：

- `VeterancyRank` 定义 Recruit / Hardened / Veteran / Elite，`GameEntity.veterancyRank` 由 `veterancyXP` 动态计算。
- `awardVeterancyKill(to:target:)` 只给存活、非结构、有伤害、直接火力击杀单位增加 `killCount` 和 XP，玩家与 AI 共享。
- `updateRepair(dt:)` 让 mechanic 自动维修附近同阵营受损实体；`bestEnemySupportRepair()` 让 Red 可用 `REPR` 修复高价值受损资产。
- `supportRepairScore(...)` 是 `REPR` 目标评分 / 合法性评分；v3.8 只可在 `faction == .enemy` 时给受损 Veteran / Elite 作战单位加分，不改变实际维修效果。
- `commandEnemyAttackers(_:)` 是 routine attack-move 主攻波次入口；有 objective 时先生成 mixed provisional wave，再通过 `enemyAssaultWave(from:)` 过滤后调用 `issueFormationMove(... attackMove: true)`。
- v3.6 已在 `canJoinEnemyAssaultWave(...)` 中加入 Battleship / Carrier 护航门槛；v3.8 应复用同一个最终入波过滤点。
- `issueFormationMove(...)` 是玩家和 AI 共享的移动 / attack-move 执行函数，不应放入 Red 专属老兵保护逻辑。

## 实现步骤

1. 开始前同步 `main`：

```sh
git fetch origin
git switch main
git pull --ff-only origin main
git status --short
git rev-parse HEAD
git rev-parse origin/main
```

要求当前分支为 `main`，目标远端为 `origin/main`，工作区无无关改动。

2. 在 `commandEnemyAttackers(_:)` 附近扩展 v3.6/v3.7 的 wave filter。
   - 推荐新增 helper：
     - `private func isProtectedEnemyVeteranCombatUnit(_ unit: GameEntity) -> Bool`
     - `private func healthRatio(of unit: GameEntity) -> CGFloat`
   - helper 名称可调整，但必须表达清楚。
3. 在 `canJoinEnemyAssaultWave(_:, provisionalWave:)` 中加入受损老兵保护判断。
   - 若 `isProtectedEnemyVeteranCombatUnit(unit)` 为 true，直接返回 `false`。
   - 建议该判断放在 Battleship / Carrier 护航判断之前，确保低血 Veteran / Elite 高价值海军也不会因护航充足而出击。
   - Recruit / Hardened、HP 不低的 Veteran / Elite、非作战单位保持原有行为。
4. 保护对象建议条件：
   - `unit.faction == .enemy`
   - `unit.isAlive`
   - `!unit.kind.isStructure`
   - `unit.kind.damage > 0`
   - `unit.isOperational`
   - `unit.veterancyRank == .veteran || unit.veterancyRank == .elite`
   - `healthRatio(of: unit) < 0.55`
5. 保持旧行为边界。
   - 不改变 `idleAttackers` 初筛条件。
   - 不改变 `enemyProvisionalAssaultWave(...)` 的混编选择。
   - 不改变 no-objective fallback 中 direct tactical target 逻辑。
   - 不改变 `shouldAttack`。
   - 不改变 `issueFormationMove(...)`、`formationPriority(for:)`、`setDestination(...)`、combat / fog / support / defense / production。
6. 在 `supportRepairScore(...)` 中加入 Red 专属评分偏向。
   - 只在 `faction == .enemy` 且修复范围内实体满足 `isProtectedEnemyVeteranCombatUnit(...)` 时加分。
   - 推荐加分基于 `strategicValue(of:)` 的小比例，例如 `0.20` 到 `0.30`。
   - 不改变 `applySupportRepair(...)`、`SupportPower.fieldRepair`、`bestEnemySupportRepair()` 的资金 / 冷却 / 资产边界。
7. 同步文档。
   - README 更新 AI 功能描述。
   - flow 更新 AI 小节。
   - flowchart 的 AI 节点简短补充受损老兵保护。
   - 不更新 `md/test/test.md`。
   - 不更新 `update_log.md`。
8. 可以使用 `git diff`、`git diff --name-only`、`git diff --stat` 自查改动范围，但不要把它们写成验收通过。

## 验收标准

- `commandEnemyAttackers(_:)` 的 objective attack-move wave 不会把 HP 低于阈值的 Red Veteran / Elite 机动战斗单位拉入 routine assault。
- 受保护老兵保持 idle，不被设置 `attackMoveDestination`、`destination` 或 `attackTarget`。
- 受保护老兵 HP 恢复到阈值以上后可按既有规则重新加入后续主攻。
- Battleship / Carrier 仍受 v3.6 护航门槛约束；低血 Veteran / Elite Battleship / Carrier 不会因护航满足而带伤出击。
- Red `REPR` 评分会优先覆盖低血 Veteran / Elite 作战单位，但维修量、费用、冷却、半径和玩家手动 `REPR` 行为不变。
- Recruit / Hardened、未低血 Veteran / Elite、普通非 high-value combat unit 的 routine attack-move 行为基本保持原样。
- 玩家 `AMOV`、Red 防守响应、no-objective tactical target、动态 AA / ASW 生产、生产来源、维修、寻路、战斗、迷雾、支援逻辑没有语义变化。
- README、flow、flowchart 和 v3.8 prompt 与真实行为一致。
- 本轮提交 push 到 `origin/main` 后，GitHub Actions artifact 验收通过。

## 风险和禁止项

- 风险：把老兵保护放到 `issueFormationMove(...)`，会影响玩家命令和其他 AI 用途。禁止这样做。
- 风险：让所有 Veteran / Elite 永久不进攻，会削弱 AI。只保护明显低血的 Veteran / Elite。
- 风险：改 XP、伤害、冷却、维修或支援技能，会扩大范围。禁止这样做。
- 风险：改 no-objective fallback、基地防守或旗点防守，会削弱 AI 防守。禁止这样做。
- 禁止修改单位数值、AI build pattern、动态 AA / ASW 逻辑、支援技能、迷雾、潜艇检测、玩家命令、Xcode project 或 workflow。
- 禁止使用非 `Altman-sam114` 的 GitHub 账号伪装 push、CI 或 artifact 验收。
