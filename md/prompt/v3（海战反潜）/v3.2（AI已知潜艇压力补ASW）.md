# v3.2（AI已知潜艇压力补ASW）

## 版本号

v3.2

## 版本分配依据

- 当前 `update_log.md` 和 `md/prompt/` 最新海战反潜阶段为 v3.1“反潜攻击 HUD 信息”。
- v3.1 已区分 ASW attack capability 与 sonar sensor；v3.2 继续在同一阶段做小型 AI 增量。
- 本轮只让 Red 在自己已知玩家潜艇威胁时保守补 ASW 生产订单，不改变潜艇侦测、战斗、单位数值或建筑规则。

## 背景

当前 AI 已有 `queueEnemyAirDefenseIfNeeded()`，会基于 Red 自己认知的玩家空军压力，复用 `canQueueBuild(...)`、`queueBuild(...)` 和 `productionSource(...)` 额外排产 AA Truck / Fighter。玩家潜艇已有隐身、`revealedUntil`、sonar detection 和 ASW 攻击单位，但 Red 尚未对已知玩家潜艇压力做对称生产响应。

## 目标

1. 统计 Red 当前已知玩家潜艇威胁。
   - 目标必须是存活玩家 Submarine。
   - 必须同时满足 `isKnownToFaction(..., observer: .enemy)` 和 Red 单位 / 已完工建筑 / 已占旗点视野边界。
   - 不得使用玩家 `visibleTiles` / `exploredTiles`，不得全图统计玩家潜艇。
2. 统计 Red 现有和已排队 ASW 攻击资产。
   - ASW 候选包含 Helicopter、Fighter、Submarine、Battleship、Carrier。
   - Fighter 可计入 ASW attack capability，但不是 sonar sensor。
   - Sonar Buoy 是 sonar sensor，不是 ASW attacker。
3. 当已知潜艇威胁至少为 1 且 committed ASW 不足时，每个 AI 指挥周期最多额外尝试排产一个 ASW 单位。
   - 通过 `canQueueBuild(...)` 和 `queueBuild(...)` 排队。
   - 缺生产来源、来源未完工或资金不足时自然跳过。
4. 接入点放在 `queueEnemyAirDefenseIfNeeded()` 之后、常规 `aiBuildPattern()` 扫描之前。
5. 同步 README、`md/flow/flow.md` 和 `md/flow/flowchart.md`。

## 非目标

- 不改变 `canAttack(...)`、`hasSonar`、`sonarRange(for:)`、`isSubmarineDetected(...)`、`revealedUntil`、`SCAN`、`AIRS` 或 `BARR`。
- 不改变潜艇隐身、声呐检测、支援命中暴露、迷雾合法性、目标搜索、战斗、单位数值、建筑规则或 HUD 输入。
- 不新增单位、建筑、科技树、升级、支援技能、外部素材或外部依赖。
- 不让 AI 全图知道玩家潜艇，不直接生成单位，不让 ASW 响应阻断常规生产。
- 不修改 Xcode project、CI workflow、`md/test/test.md` 或 `update_log.md`。
- 按人工要求，本轮不得运行本地测试、本机构建或本地静态检查作为验证。

关键源码事实：

- `updateAI(dt:)` 当前在常规 `aiBuildPattern()` 扫描前调用 `queueEnemyAirDefenseIfNeeded()`。
- `queueEnemyAirDefenseIfNeeded()` 已提供动态威胁响应模板：统计 Red 已知威胁、统计现有 / 已排队资产、候选列表自然排队。
- `isPointKnownToEnemySensors(...)` 用 Red 存活单位、Red 已完工建筑和 Red 已占旗点视野判断 Red 认知边界。
- `isKnownToFaction(..., observer: .enemy)` 对潜艇会走 `isSubmarineDetected(...)`。
- `EntityKind.canAttack(.submarine)` 当前允许 Submarine、Battleship、Helicopter、Fighter、Carrier。
- `EntityKind.hasSonar` 当前允许 Battleship、Submarine、Carrier、Helicopter、Sonar Buoy，不包含 Fighter。
- `productionSource(for:faction:)` 要求来源存活且 operational；`queueBuild(...)` 扣钱并追加 `BuildOrder`，不会直接生成单位。

## 风险和禁止项

- 风险：只用 `isKnownToFaction(... observer: .enemy)` 可能把没有 Red 视野支撑的临时状态当作全图认知；必须叠加 Red sensor / flag 视野判断。
- 风险：误用玩家 `visibleTiles` 会让 AI 共享玩家视野。
- 风险：不统计 `buildOrders` 会导致每周期重复堆 ASW。
- 风险：把 Fighter 写成 sonar 或把 Sonar Buoy 计入 ASW 会破坏 v3.1 区分。
- 禁止直接生成单位、绕过生产链路、改变潜艇侦测 / 暴露 / 战斗 / 迷雾规则、改变单位数值或建筑规则。
