# v2.5（AI空军威胁防空响应）

## 版本号

v2.5

## 版本分配依据

- 人工未指定本轮版本；`update_log.md` 和 `md/prompt/` 当前最新正式玩法版本为 v2.4“引入声呐浮标”。
- v2.4 已通过 `origin/main` GitHub Actions artifact 验收，并追加正式记录。
- 本轮继续沿 `v2（地图控制）` 阶段推进，在 SAM Site、AA Truck、Sonar Buoy 已存在后，让 AI 能根据已知玩家空军压力动态补防空，因此递增为 v2.5。

## 背景

当前项目已有多类防空来源：

- Guard Tower / SAM Site：已完工静态防御 / 防空结构。
- AA Truck：War Factory 生产的移动陆地防空单位。
- Fighter：Airfield / Carrier 生产的空中单位，可攻击 air / naval / structure。

v2.3 已让 AI 的固定 build pattern 纳入 AA Truck，但该 pattern 仍是静态轮转，不会根据玩家当前空军规模主动补防空。v2.5 做一个保守 AI 增量：当 Red 通过自身视野或已占旗点视野知道玩家空军压力较高，且当前防空资产和已排队防空不足时，AI 在常规生产前优先尝试排一个 AA Truck 或 Fighter。

## 目标

1. 新增 AI 空军威胁评估。
   - 统计 Red 当前已知的玩家空中单位数量。
   - 统计对象必须是 `entity.faction == .player`、`entity.kind.domain == .air`、`entity.isAlive`。
   - 因当前 `isKnownToFaction(... observer: .enemy)` 对非潜艇目标基本不做 Red 视野过滤，本轮必须额外限制为 Red 存活单位、Red 已完工建筑或 Red 已占旗点当前视野内的玩家空军，避免全图作弊。
2. 新增 Red 防空覆盖评估。
   - 统计 Red 存活且可用的防空资产：`.aaTruck`、`.fighter`、`.guardTower`、`.samSite`。
   - 静态结构必须 `isOperational` 后才计入。
   - 移动单位按存活计入，不要求 idle。
   - 统计 Red 已排队的 `.aaTruck` 和 `.fighter`，避免每个 AI 周期重复堆同类订单。
3. 新增优先补防空逻辑。
   - 当已知玩家空军数量达到阈值且 Red 防空数量不足时，AI 在常规 `aiBuildPattern()` 扫描前尝试排一个防空单位。
   - 防空目标建议：`max(1, min(4, (knownAirThreat + 1) / 2))`。
   - 候选优先级为 `.aaTruck`，其次 `.fighter`。
   - 必须继续走 `canQueueBuild(...)` / `queueBuild(...)`，缺 War Factory、缺 Airfield、未完工来源或资金不足时自然跳过。
4. 保持常规 AI 生产。
   - `queueEnemyAirDefenseIfNeeded()` 成功排队后，本周期后续常规生产仍继续由资金、来源和 build pattern 自然决定。
   - 每个 AI 指挥周期最多额外尝试一个防空订单。
5. 不改变玩家行为。
   - 不新增 HUD 按钮、单位、建筑、支援技能或手动命令。
   - 不改变 AA Truck、Fighter、SAM Site、Guard Tower 的 `canAttack(...)` 或数值。
6. 不改变潜艇 / 声呐链路。
   - 新增 AI 评估不得把 AA Truck、Fighter、SAM Site 或 Guard Tower 变成 sonar。
   - 不改变 `hasSonar`、`sonarRange(for:)`、`revealedUntil`、`isSubmarineDetected(...)`。
7. 文档同步。
   - README Enemy AI bullet 增加“known player air pressure 时优先补 AA Truck / Fighter”这类当前真实行为。
   - `md/flow/flow.md` AI 小节同步该动态防空响应，并强调 Red 视野过滤和共享生产链路。
   - `md/flow/flowchart.md` AI 节点加入“空军压力补防空”短语。
8. 不更新 `update_log.md`。正式 v2.5 记录由 Agent C 在最新 `origin/main` artifact 验收通过后追加。

## 非目标

关键源码事实：

- `queueBuild(kind:faction:showFeedback:)`、`canQueueBuild(kind:faction:)` 和 `productionSource(...)` 是玩家与 AI 共享生产链路。

### 2. 新增 AI helper

建议在 AI helper 区域新增：

- `knownPlayerAirThreatCount() -> Int`
- `isKnownPlayerAirThreat(_ entity: GameEntity) -> Bool`
- `isPointKnownToEnemySensors(_ point: CGPoint) -> Bool`
- `enemyAntiAirAssetCount() -> Int`
- `queuedEnemyAntiAirCount() -> Int`
- `desiredEnemyAntiAirCount(for knownAirThreat: Int) -> Int`
- `queueEnemyAirDefenseIfNeeded() -> Bool`

Red sensor / flag 视野规则：

- Red 存活单位贡献 `effectiveVisionTiles(for:)` 半径。
- Red 存活且已完工结构贡献 `effectiveVisionTiles(for:)` 半径。
- Red 已占旗点贡献 `controlPointVisionTiles` 半径。
- 视野判断可按 tile 半径实现，使用 `tile(at:)` 将目标点和 sensor 点转为 `TileCoord`，再用 `hypot(...)` 判断。
- 不需要引入持久 enemy visible set；本轮只做动态 AI 决策 helper。

可接受的保守策略：

```swift
let knownAirThreat = knownPlayerAirThreatCount()
guard knownAirThreat >= 2 else { return false }
let desiredAntiAir = desiredEnemyAntiAirCount(for: knownAirThreat)
guard enemyAntiAirAssetCount() + queuedEnemyAntiAirCount() < desiredAntiAir else { return false }
for kind in [EntityKind.aaTruck, .fighter] where canQueueBuild(kind: kind, faction: .enemy) {
    return queueBuild(kind: kind, faction: .enemy, showFeedback: false)
}
return false
```

`desiredEnemyAntiAirCount` 建议：

```swift
max(1, min(4, (knownAirThreat + 1) / 2))
```

这样 2 个已知空军触发 1 个防空目标，3-4 个触发 2 个，最多 4 个，避免 AI 过度刷防空。
