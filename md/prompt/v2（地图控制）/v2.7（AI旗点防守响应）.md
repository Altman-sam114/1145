# v2.7（AI旗点防守响应）

## 版本号

v2.7

## 版本分配依据

- 人工未指定本轮版本；`update_log.md` 和 `md/prompt/` 当前最新正式玩法版本为 v2.6“AI 占点任务长期保留”。
- v2.6 已让 Red AI 的占油 / 占旗 runner 在目标完成前不被常规主攻波次抢走。
- 本轮继续沿着地图控制和旗点争夺做一个小闭环：玩家开始夺取 Red 已占旗点时，Red 能调附近可用作战单位防守，因此递增为 v2.7。

## 背景

当前 Red 会周期性派 runner 占油 / 占旗，也会在基地单位或建筑被攻击时触发紧急防守。但玩家站上 Red 已占旗点时，旗点只推进占领进度；如果附近 Red 作战单位没有刚好进入普通目标搜索或主攻波次，Red 不会立即针对抢点单位做防守响应。

本轮目标是补齐“旗点正在被抢”的小型 AI 反应，让地图控制更接近 RTS 争夺点位的节奏，同时继续保持 v2.6 的占点 reservation 不被破坏。

## 目标

1. 当玩家单位开始夺取 Red 已占控制点时触发 Red 旗点防守响应。
   - 触发位置在 `updateControlPointCapture(dt:)` 中玩家单独靠近、旗点当前归 Red、旗点尚未归玩家的分支。
   - 只在 Red 能通过自己的单位、已完工建筑或已占旗点视野确认该玩家单位时触发，不做全图作弊。
2. Red 防守响应调动附近非 reservation 作战单位。
   - 单位必须是 Red、存活、非结构、有伤害、能攻击抢点单位。
   - 必须跳过 `isEnemyCaptureReserved(...)` 为 true 的 runner，避免破坏 v2.6 占点长期保留。
   - 限制响应半径和最多响应数量，避免全图拉兵。
3. 防守响应必须节流。
   - 新增独立 cooldown 字段，避免每帧或每个 capture tick 重复重派。
   - `SKRM` / `restartSkirmish()` 必须重置该 cooldown。
4. 防守单位应使用现有战斗 / 移动链路。
   - 直接设置 `attackTarget` 指向抢点单位。
   - 若不在攻击距离内，使用现有 `attackDestination(for:target:)` 和 `setDestination(for:near:)` 接近。
   - 不新增平行战斗系统、寻路系统或 AI 角色系统。
5. 可见反馈保持轻量。
   - 若玩家当前能看到被抢旗点，可复用既有 enemy defense ping。
   - 不新增 HUD 按钮、调试 UI、外部素材或新图标。
6. 文档同步。
   - README Enemy AI 能力说明同步 Red 会响应玩家抢 Red 旗点。
   - `md/flow/flow.md` AI 小节同步该行为、Red 视野条件、跳过 reservation 和 `SKRM` 重置。
   - `md/flow/flowchart.md` AI 节点加入“旗点防守响应”等简短描述。

## 非目标

- 不改变旗点占领半径、占领速度、收入、视野、build coverage、任务阶段或胜负条件。

关键源码事实：

- `updateControlPointCapture(dt:)` 每帧判断玩家 / Red 是否有单位在旗点占领半径内，并推进 `captureProgress`。
- `isPointKnownToEnemySensors(_:)` 已按 Red 存活单位、Red 已完工建筑和 Red 已占旗点视野判断玩家空军威胁；本轮可复用该 Red 认知边界来避免全图作弊。
- `triggerEnemyDefenseIfNeeded(target:attacker:)` 已展示了 Red 防守调兵的现有风格。
- `isEnemyCaptureReserved(_:)` 标识 v2.6 保护中的占油 / 占旗 runner；本轮防守响应必须跳过这些单位。
- `setDestination(for:near:)`、`attackDestination(for:target:)`、`updateCombat(dt:)` 是当前共享移动和攻击链路。
- `restartSkirmish()` 已重置 AI accumulator、defense cooldown 和 `enemyCaptureReservations`。

- 如果 `git status --short` 显示任何未提交、未跟踪或冲突文件，B 必须停止并确认归属。
- 本提示词文件 `md/prompt/v2（地图控制）/v2.7（AI旗点防守响应）.md` 属于本轮 Agent A 产物。

### 2. 新增 AI 旗点防守 cooldown

- 在 `GameScene` 状态字段中新增 `lastEnemyControlPointDefenseResponseTime`，初始值与现有防守 alert/cooldown 风格一致。
- 在 `restartSkirmish()` 中重置该字段。

### 3. 在旗点占领流程中触发防守

- 在 `updateControlPointCapture(dt:)` 的玩家单独抢点分支中，当 `point.faction == .enemy` 时调用新 helper。
- helper 建议命名为 `triggerEnemyControlPointDefenseIfNeeded(point:intruder:)`。

### 4. 实现 helper 行为

建议逻辑：

- guard 条件：
  - `point.faction == .enemy`
  - `intruder.faction == .player`
  - `intruder.isAlive`
  - `isPointKnownToEnemySensors(intruder.node.position)` 为 true
  - cooldown 已过
- Red 确认感知抢点单位后刷新本次防守尝试 cooldown，即使暂时没有合格 defender，也避免每帧重复扫描。
- 筛选 defenders：
  - Red、存活、非结构、有伤害、能攻击 intruder
  - 距离 flag 不超过有限半径
  - `!isEnemyCaptureReserved(unit)`
- 对距离最近的前几名 defenders：
  - `attackTarget = intruder`
  - 如果超出攻击距离，用 `setDestination(for:near:)` 接近 `attackDestination(for:target:)`
- 如果玩家能看到旗点，复用 `showEnemyDefensePing(at:)`。

### 5. 文档更新

- README 的 Enemy AI bullet 增加“玩家抢 Red 旗点时会拉附近防守单位响应”。
- `md/flow/flow.md` AI 小节增加同等事实。
- `md/flow/flowchart.md` AI 节点简短加入“旗点防守响应”。
- 不更新 `update_log.md`。正式 v2.7 记录由 Agent C 在最新 `origin/main` artifact 验收通过后追加。

## 验收标准

- 玩家单位开始夺 Red 已占旗点时，Red 在自己传感 / 旗点视野确认后会调附近可攻击该玩家单位的作战单位防守。
- 防守响应有独立节流，不会每帧重复重派。
- v2.6 占油 / 占旗 reservation runner 不会被旗点防守响应抢走。
- Red 不会对自己不可感知的抢点行为全图作弊。
- `SKRM` 重开后新 cooldown 状态重置。
- README、flow、flowchart 与真实代码行为一致。
- 最新 `origin/main` GitHub Actions artifact 通过，manifest / run / commit 对应一致。

- 风险：如果不跳过 reservation，v2.6 的占点 runner 保护会退化。
- 风险：如果不使用 Red 视野判断，会形成全图作弊。
- 禁止修改占领数值、收入、建筑覆盖或玩家命令语义。
