# v2.8（AI优先反夺旗点）

## 版本号

v2.8

## 版本分配依据

- 人工未指定本轮版本；`update_log.md` 和 `md/prompt/` 当前最新正式玩法版本为 v2.7“AI 旗点防守响应”。
- v2.7 已让玩家抢 Red 已占旗点时触发 Red 的局部防守响应。
- 本轮继续沿着旗点争夺做一个更小的 AI 目标选择增量：Red 在选择下一个旗点目标时，优先反夺玩家已占或正在被玩家占领的旗点，再用距离作为次级排序，因此递增为 v2.8。

## 背景

当前 `enemyControlPointTarget(excludingReserved:)` 会在所有非 Red 旗点中选择离 Red 基地最近的点。这个策略适合初期扩张，但当玩家已经占住远一点的旗点，Red 仍可能先去最近的中立点，导致地图控制争夺感不足。

v2.8 做一个保守增量：不改变旗点收益、占领速度或单位行为，只调整 Red 选择旗点目标的候选和评分，让玩家已占旗点、玩家正在占领的旗点，以及 Red 已占但正在被玩家夺取的旗点获得更高优先级。该函数同时服务 AI 占旗 runner 和主攻波次，因此能让 Red 更像 RTS 对手一样反夺关键地图点。

## 目标

1. 调整 Red 旗点目标选择优先级。
   - `enemyControlPointTarget(excludingReserved:)` 应考虑非 Red 旗点，以及 Red 已占但正在被玩家占领的旗点。
   - 玩家已占旗点应优先于中立旗点。
   - 玩家正在占领的旗点应获得额外优先级。
   - Red 已占但正在被玩家占领的旗点也应进入候选，并获得最高或接近最高的反夺优先级。
   - 距离 Red 基地仍作为次级排序，避免过度追逐远端目标。
2. 保持 v2.6 reservation 行为。
   - `excludingReserved == true` 时仍必须跳过已有 `.controlPoint(point.id)` reservation 的目标。
   - 不允许同一目标被多个占旗 runner 因本轮改动而重复分配。
3. 同时影响占旗 runner 和主攻波次。
   - `updateAI(dt:)` 的 flag runner 调用继续使用 `enemyControlPointTarget(excludingReserved: true)`。
   - `enemyWaveObjective()` 的 attack-move 波次继续使用默认 `enemyControlPointTarget()`，但目标排序会随新评分变化。
4. 保持既有地图控制规则。
   - 不改变旗点占领半径、占领速度、收入、视野、build coverage、任务阶段或胜负条件。
   - 不改变玩家命令、AI 生产、AI 防守响应、支援技能、迷雾、潜艇或单位 / 建筑数值。
5. 文档同步。
   - README Enemy AI 能力说明同步 Red 会优先反夺玩家控制或正在争夺的前线旗点。
   - `md/flow/flow.md` AI 小节同步目标选择评分行为。
   - `md/flow/flowchart.md` AI 节点加入“反夺旗点优先级”等简短描述。

## 非目标

关键源码事实：

- `BattlefieldControlPoint` 保存 `faction`、`captureProgress` 和 `capturingFaction`。
- `updateControlPointCapture(dt:)` 在玩家或 Red 单独进入旗点半径时推进占领，并在完成后调用 `setControlPointFaction(...)`。
- `enemyControlPointTarget(excludingReserved:)` 当前筛选非 Red 旗点，并按离 `enemyBaseAnchorPoint()` 的距离取最近目标。
- `enemyCaptureTargetNeedsCapture(_:)` 当前对 `.controlPoint` 只看 `controlPoint.faction != .enemy`，若要让 Red 已占但被玩家争夺的旗点成为 reservation 目标，也必须同步改为共享旗点是否需要行动的判断。
- `updateAI(dt:)` 的 flag runner 使用 `enemyControlPointTarget(excludingReserved: true)`，已有 reservation 目标不能重复分配。
- `enemyWaveObjective()` 先调用默认 `enemyControlPointTarget()`，因此主攻波次也会受到该排序影响。
- `isEnemyCaptureTargetReserved(_:)` 和 `enemyCaptureReservations` 是 v2.6 的占点目标去重机制。

### 2. 调整旗点候选与评分

建议新增私有 helpers：

```swift
private func isPlayerPressuringControlPoint(_ point: BattlefieldControlPoint) -> Bool
private func enemyControlPointNeedsAction(_ point: BattlefieldControlPoint) -> Bool
private func enemyControlPointPriority(_ point: BattlefieldControlPoint) -> CGFloat
```

建议候选语义：

- `enemyControlPointNeedsAction(_:)` 对非 Red 旗点返回 true。
- Red 已占旗点若 `isPlayerPressuringControlPoint(_:)` 为 true，也返回 true。
- `enemyCaptureTargetNeedsCapture(.controlPoint)` 和 `enemyControlPointTarget(excludingReserved:)` 应共享这个判断，避免 reservation 目标和战略目标分叉。

建议评分语义：

- `point.capturingFaction == .player` 且 `captureProgress > 0`：最高或接近最高权重。
- `point.faction == .player`：高于中立旗点。
- `point.faction == .neutral`：低于玩家已占旗点，但仍是有效扩张目标。
- 距离 Red 基地越远，分数小幅降低。

实现方式不限，但必须保持行为易读、局部、可维护。可以继续用 `.max` 比较分数，也可以用清晰的 tuple-like 排序逻辑；不要引入全局状态。

### 3. 替换目标选择排序

- `enemyControlPointTarget(excludingReserved:)` 的 filter 改为：
  - `enemyControlPointNeedsAction(point)`
  - `excludingReserved` 为 true 时跳过 `isEnemyCaptureTargetReserved(.controlPoint(point.id))`
- 将 `.min` 距离排序改为按新优先级选择。
- 同分时应保留距离近者优先；再同分可用 `id` 保持稳定。
- `enemyCaptureTargetNeedsCapture(.controlPoint)` 也应改用 `enemyControlPointNeedsAction(...)`，让 Red-held 但 player-contested 的旗点可以维持 reservation。

### 4. 文档更新

- README Enemy AI bullet 增加 Red 会优先反夺玩家控制或正在争夺的前线旗点。
- `md/flow/flow.md` AI 小节说明 `enemyControlPointTarget(...)` 按玩家控制 / 玩家正在占领 / 距离评分选择旗点目标，且 Red 已占但正在被玩家夺取的旗点也会进入候选。
- `md/flow/flowchart.md` AI 节点加入“反夺旗点优先级”。
- 不更新 `update_log.md`。正式 v2.8 记录由 Agent C 在最新 `origin/main` artifact 验收通过后追加。

## 验收标准

- Red 选择旗点目标时，玩家已占旗点优先于中立旗点。
- 玩家正在占领的旗点获得额外优先级，包括 Red 已占但正在被玩家夺取的旗点。
- 距离 Red 基地仍影响排序，避免完全忽略近端目标。
- `excludingReserved: true` 仍跳过已有 reservation 的旗点目标。
- 占旗 runner 和主攻 attack-move 波次都通过同一个目标排序受益。
- 不改变旗点占领、收入、视野、build coverage、玩家命令、AI 防守响应或生产规则。
- README、flow、flowchart 与真实代码行为一致。
- 最新 `origin/main` GitHub Actions artifact 通过，manifest / run / commit 对应一致。
