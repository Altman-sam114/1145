# v4.65 / HOLD 航母警戒缺口反馈

## 版本号

v4.65

## 版本分配依据

v4.64 已完成 `HOLD` 按钮 Carrier guard wing 点击前上下文提示，并通过 `origin/main` 云端 artifact 验收。v4.64 遗留事项建议继续细化 Carrier guard wing 命令 UI。本轮选择点击后的结果反馈作为最小可验证增量：玩家对 Carrier 下达 `HOLD` 后，顶部消息同步显示最终 guard wing 数量、HEL/JET 组成和缺口，版本递增为 v4.65。

## 背景

当前玩家选中 Carrier 点击 `HOLD` 时，若成功绑定附近 HEL/JET，会显示 `Guard wing n Hn/Jn.`；若没有绑定任何 wing，则完全不显示 guard wing 结果。选择面板能在后续展示 `GW x/2 Need n`，但点击后的即时反馈不能解释“没有飞机被绑定”还是“没有触发 Carrier guard wing”。v4.64 已补点击前 subtitle，本轮补点击后的顶部结果消息，让玩家更快判断是否需要生产或调近 HEL/JET。

## 目标

- 当本次 `HOLD` 命令包含玩家 Carrier 时，顶部成功消息总是显示 Carrier guard wing 结果。
- 结果消息基于命令执行后的真实绑定状态：
  - 若满编，显示 `Guard wing 2 H1/J1.` 这类现有组成信息。
  - 若缺员，显示 `Guard wing 1 H1 Need 1.` 或 `Guard wing 0 Need 2.`。
  - 多 Carrier 同选时，按当前选中 Carrier 聚合总需求、总绑定数和去重 HEL/JET 组成。
- 没有选中 Carrier 的 ordinary `HOLD` 仍保持原有消息，只在必要时显示 `CV guard released n Hn/Jn.`。
- 只读复用 `boundCarrierGuardWing(for:)`、`carrierAirWingCompositionSuffix(for:)` 和 `carrierGuardWingRequirement`。
- 同步 README 和 `md/flow/flow.md` 当前事实说明。

## 非目标

- 不改变 `assignCarrierGuardWing(for:)` 的候选筛选、排序、最多 2 架绑定规则或返回值。
- 不改变 `issueHoldPositionOrder(units:)` 的移动单位 HOLD 顺序、释放旧 anchor、刷新选择、甲板 cue 或 `showMessage(...)` 时机。
- 不新增按钮、命令模式、CAP、巡逻、截击、自动护航状态机或战斗加成。
- 不改变移动、攻击、AMOV、AI Carrier guard、战斗、迷雾、生产、支援、任务、胜负或 `SKRM` 重置。
- 不更新 `md/test/test.md`，本轮不改变测试流程。

## 当前架构依据

- `issueHoldPositionOrder(units:)` 是玩家 `HOLD` 命令的统一执行入口。
- `assignCarrierGuardWing(for:)` 在 Carrier HOLD 后写入最终 `guardAnchorCarrierID`，其后可以通过 `boundCarrierGuardWing(for:)` 读取真实绑定结果。
- `carrierAirWingCompositionSuffix(for:)` 已统一生成 `Hn/Jn` 组成后缀，应该复用，避免新口径。
- `carrierGuardWingRequirement` 是单 Carrier guard wing 需求数，多 Carrier 聚合需求应使用 `holdCarriers.count * carrierGuardWingRequirement`。
- 现有 `carrierGuardReleaseSuffix(for:)` 已负责 ordinary HOLD / 移动 / AMOV / 直接攻击释放反馈，本轮不改变该链路。

## 实现步骤

1. 同步 main：
   - `git fetch origin`
   - `git switch main`
   - `git pull --ff-only origin main`
   - 确认工作区没有无关改动。
2. 修改 `DesertFrontline/GameScene.swift`：
   - 在 `issueHoldPositionOrder(units:)` 中替换当前只在 `guardWingCount > 0` 时生成的 `guardWingSuffix`。
   - 新增或内联一个只读汇总逻辑：
     - 若 `holdCarriers.isEmpty`，`guardWingSuffix = ""`。
     - 否则读取 `holdCarriers.flatMap { boundCarrierGuardWing(for: $0) }` 并按 entity id 去重。
     - `requiredWingCount = holdCarriers.count * carrierGuardWingRequirement`。
     - `boundCount = min(uniqueWing.count, requiredWingCount)`。
     - `missing = max(0, requiredWingCount - boundCount)`。
     - 组成后缀继续用 `carrierAirWingCompositionSuffix(for:)`。
     - 生成 ` Guard wing \(boundCount)\(compositionSuffix).`，若 `missing > 0`，在句号前追加 ` Need \(missing)`。
   - 保持甲板 pulse 仍只在实际绑定 wing 时显示，不为 `0 Need n` 新增 pulse。
   - 不写入任何新的实体字段。
3. 修改 `README.md`：
   - 在 Carrier / HOLD 控制说明中补充：Carrier HOLD 顶部反馈会在 guard wing 缺员时显示 `Need n`，没有绑定时也会显示 `Guard wing 0 Need n`。
4. 修改 `md/flow/flow.md`：
   - 在输入 / 命令或 HUD 段补充：Carrier HOLD 成功消息只读汇总最终绑定 wing 数、HEL/JET 组成和缺口。
5. 不需要修改 `md/flow/flowchart.md`，除非实际图中需要同步提到顶部消息缺口。
