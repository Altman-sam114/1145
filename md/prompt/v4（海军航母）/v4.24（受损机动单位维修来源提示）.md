## 背景

当前 `DesertFrontline/GameScene.swift` 已有以下事实：

- `GameScene` 私有常量区已有：
  - `private let mechanicRepairRange: CGFloat = 95`
  - `private let mechanicRepairPerSecond: CGFloat = 22`
- `updateRepair(dt:)` 每帧遍历所有存活 Mechanic，寻找同阵营、存活、非自身、`hp < kind.maxHP` 的最近受损目标；当距离 `< mechanicRepairRange` 时，按 `CGFloat(dt) * mechanicRepairPerSecond` 修复，并偶发 `showRepairSpark(at:)`。
- `damagedRepairTargetCount(for:)` 已为 Mechanic 单选反馈统计范围内同阵营、存活、非自身、受损目标数量，口径与 `updateRepair(dt:)` 候选一致。
- `singleSelectionInfo(for:)` 是单选面板入口。非结构单位默认第 4 行显示 `veterancyProgressLine(for:)`；当 `mobileStatusLine(for:)` 返回状态时，会把 compact veterancy 追加到第 3 行，并把第 4 行替换为该状态。
- `mobileStatusLine(for:)` 当前优先显示：
  - Submarine stealth / temporary detected 状态。
  - Mechanic 的 `Repair 95  Damaged n`。
  - Carrier 专属状态。
  - `Attack-moving  Seek ...`。
  - `Holding position  Guard ...`。
- v4.23 已明确 Mechanic 维修范围圈和受损目标统计都是只读反馈，不改变 `updateRepair` 范围 95、维修量 22/s、最近目标选择、AI、Field Repair、移动、战斗、迷雾、任务、胜负或 `SKRM`。

当前问题是：玩家单选一个已经受损的己方机动单位时，只能看到 HP，但看不到它附近有没有 Mechanic 可以自动维修。玩家需要一个短提示判断是否已经处于维修来源范围内，或最近 Mechanic 大约还有多远。

## 目标

1. 玩家单选受损、存活、非结构、非 Mechanic 的己方机动单位时，选择面板显示短维修来源提示。
   - 触发条件必须同时满足：
     - `entity.faction == .player`
     - `entity.isAlive`
     - `!entity.kind.isStructure`
     - `entity.kind != .mechanic`
     - `entity.hp < entity.kind.maxHP`
   - 推荐文案之一：
     - 最近 Mechanic 距离 `< mechanicRepairRange`：`MECH in range` 或 `MECH in range  72`
     - 有 Mechanic 但不在范围：`Need MECH  184`、`MECH 184 away` 或等价短句
     - 没有存活己方 Mechanic：`Need MECH`
   - 文案必须短，适合现有 selection panel 的单行宽度。
   - 距离若显示，必须来自最近己方存活 Mechanic 到该受损单位的 `node.position.distance(to:)`。
2. 提示只读，只回答“最近维修来源在哪里 / 是否在范围内”。
   - 不改变 `updateRepair(dt:)`。
   - 不改变 `mechanicRepairRange = 95`。
   - 不改变 `mechanicRepairPerSecond = 22`。
   - 不改变维修目标选择、范围比较 `< mechanicRepairRange`、repair spark、HP 结算或任何维修对象口径。
3. 明确状态优先级，避免覆盖关键战术状态。
   - `Submarine` 的 stealth / detected 状态优先级高于维修来源提示。
   - `Mechanic` 自身的 `Repair 95  Damaged n` 优先级高于维修来源提示。
   - `Carrier` 的 deck / rally / escort 面板逻辑不得被维修来源提示覆盖。
   - `AMOV` / `HOLD` 状态优先级高于维修来源提示。
   - 只有当单位没有上述更重要状态时，才显示受损单位维修来源提示。
4. 同步玩家可见文档和核心流程文档，说明这是选择面板只读提示。

### 2. 新增最近己方 Mechanic 查询 helper

推荐新增一个纯查询 helper，例如：

```swift
private func nearestFriendlyMechanicDistance(for entity: GameEntity) -> CGFloat? {
    entities.values
        .filter { mechanic in
            mechanic.faction == entity.faction &&
                mechanic.isAlive &&
                mechanic.kind == .mechanic
        }
        .map { $0.node.position.distance(to: entity.node.position) }
        .min()
}
```

也可以直接返回最近 Mechanic 实体与距离：

```swift
private func nearestFriendlyMechanic(for entity: GameEntity) -> (mechanic: GameEntity, distance: CGFloat)?
```

要求：

- 只统计同阵营、存活、`kind == .mechanic` 的实体。
- 本轮目标只要求玩家单选己方受损单位显示提示；helper 可保持 faction-generic，但 `mobileStatusLine` 中必须限制为玩家己方受损单位才显示。

- 不读取或修改 fog；己方 Mechanic 作为玩家资源信息可用于自己的选择面板。

### 3. 新增受损机动单位维修来源提示 helper

推荐新增：

```swift
private func damagedMobileRepairSourceLine(for entity: GameEntity) -> String? {
    guard entity.faction == .player,
          entity.isAlive,
          !entity.kind.isStructure,
          entity.kind != .mechanic,
          entity.hp < entity.kind.maxHP else { return nil }

    guard let distance = nearestFriendlyMechanicDistance(for: entity) else {
        return "Need MECH"
    }

    if distance < mechanicRepairRange {
        return "MECH in range"
    }
    return "Need MECH  \(Int(distance))"
}
```

上面只是参考写法，不要求逐字照抄；但必须满足：

- 只在玩家己方、存活、受损、非结构、非 Mechanic 机动单位上返回提示。
- 不对健康满血单位返回提示。
- 不对敌方单位、中立建筑、结构、Mechanic 自身返回该提示。
- 距离显示若采用 `Int(distance)`，可以向下取整；若采用 `ceil` 也可接受，但必须在提示词最终说明中写清实际格式。
- `MECH in range` 的判断必须是 `distance < mechanicRepairRange`，与 `updateRepair(dt:)` 保持一致。
- 如果没有存活己方 Mechanic，显示 `Need MECH` 或等价短提示。
- 文案要短，不新增 selection panel 行数。

### 4. 接入 `mobileStatusLine(for:)` 并保持优先级

在 `mobileStatusLine(for:)` 中接入维修来源提示时，必须保持高优先级状态先返回。

推荐顺序：

```swift
private func mobileStatusLine(for entity: GameEntity) -> String? {
    if entity.kind == .submarine {
        return submarineStatusLine(for: entity)
    }
    if entity.kind == .mechanic {
        return "Repair \(Int(mechanicRepairRange))  Damaged \(damagedRepairTargetCount(for: entity))"
    }
    if entity.kind == .carrier {
        return structureStatusLine(for: entity)
    }
    if entity.attackMoveDestination != nil {
        return "Attack-moving  Seek \(Int(attackMoveEngagementRadius))"
    }
    if entity.holdPosition != nil {
        return "Holding position  Guard \(Int(holdEngagementRadius))"
    }
    if let repairSource = damagedMobileRepairSourceLine(for: entity) {
        return repairSource
    }
    return nil
}
```

要求：

- Submarine 状态不得被维修来源提示覆盖，即使潜艇受损且附近有 Mechanic。
- Mechanic 自身仍显示 `Repair 95  Damaged n`，不得显示 `Need MECH`。
- Carrier 的既有专用单选面板逻辑不得改变；不要因为 Carrier 受损而覆盖 deck / queue / escort / rally 信息。
- `Attack-moving` 和 `Holding position` 状态优先于维修来源提示，避免玩家战术命令状态被遮掉。
- 只有没有上述状态时，受损己方机动单位才显示维修来源提示。
- 保持 `singleSelectionInfo(for:)` 现有行数和状态处理方式；不要扩展 HUD 行数或重排面板布局。
