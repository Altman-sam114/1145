# v4.16（支援目标面板资产状态）

## 版本号

v4.16

## 版本分配依据

- `update_log.md` 当前最新正式玩法版本为 v4.15「集结点 pending 面板摘要」。
- `md/prompt/v4（海军航母）/` 已连续围绕海军、航母、海岸资产、任务面板和 HUD 可读性做小步增强。
- v4.15 遗留事项中已把「支援目标面板资产状态」列为后续可推进方向。
- 本轮是一个很小的支援 pending selection info 文案增量：玩家点击 `SCAN`、`REPR`、`AIRS`、`BARR` 并成功进入 pending 后，selection info panel 的资产行按现有 `hasOperationalSupportAsset(for:faction:)` 口径显示资产 ready / need 状态，不改变支援技能费用、冷却、资产需求、执行效果、pending 行为、AI、迷雾、潜艇暴露或按钮布局。

## 背景

当前支援技能相关事实：

- `SupportPower` 包含 `reconSweep`、`fieldRepair`、`airStrike`、`navalBarrage`，HUD 标题分别是 `SCAN`、`REPR`、`AIRS`、`BARR`。
- `supportButtonSubtitle(for:)` 当前已经用 `supportCooldown(for:faction:power:)` 和 `hasOperationalSupportAsset(for:faction:)` 显示按钮 subtitle，优先级为冷却秒数、`need asset`、价格。
- `selectSupportPower(_:)` 进入 pending 前调用 `supportIssue(for:faction:)`；若有问题会清空 `pendingSupportPower` 并提示，若无问题才设置 `pendingSupportPower = power`、清空选择、刷新 HUD。
- `supportIssue(for:faction:)` 当前顺序是冷却、资金、资产需求；本轮不要改变这个顺序。
- `hasOperationalSupportAsset(for:faction:)` 当前只统计同阵营、存活、`isOperational` 的实体：
  - `SCAN` / `.reconSweep`：HQ 或 Radar Outpost。
  - `REPR` / `.fieldRepair`：HQ 或 Mechanic。
  - `AIRS` / `.airStrike`：Airfield 或 Carrier。
  - `BARR` / `.navalBarrage`：Battleship 或 Carrier。
- 建筑施工边界仍以 `isOperational` 为准；未完工 Radar Outpost、Airfield、Carrier、Shipyard 等不应提供支援资产。
- `selectionInfoContent(for:)` 当前在 `pendingSupportPower` 分支返回 4 行：
  - `$<cost>  CD <cooldown>s`
  - `Ready now` 或剩余冷却秒数
  - 半径 + Repair 或 Dmg
  - `supportAssetLine(for:)`
- `supportAssetLine(for:)` 当前固定返回 `Needs HQ or RAD` / `Needs HQ or Mechanic` / `Needs Airfield or CV` / `Needs Battleship or CV`，即使玩家已有 operational 支援资产，pending 面板仍显示 `Needs ...`，可读性不准确。

人工指定本轮目标：为 v4.16 实现「支援目标面板资产状态」。玩家点击 `SCAN`、`REPR`、`AIRS`、`BARR` 进入 pending 后，selection info panel 的资产行不再只显示 `Needs ...`，而是按现有 `hasOperationalSupportAsset` 口径显示 asset ready 或 need asset，例如 `Asset ready: HQ/RAD` / `Need HQ/RAD`。不得修改支援费用、冷却、资产需求、`supportIssue`、执行效果、pending 行为、AI、迷雾、潜艇暴露或按钮布局。人工要求不得用本地测试、本地 build、本地 static check 或 `git diff --check` 作为验收，只认 GitHub Actions artifact。

## 目标

1. 改善 pending 支援 selection info panel 的资产行。
   - 玩家点击 `SCAN`、`REPR`、`AIRS`、`BARR` 且成功进入 `pendingSupportPower` 后，selection info panel 第四行必须按资产真实状态显示。
   - 当 `hasOperationalSupportAsset(for: pendingSupportPower, faction: .player)` 为 true 时，显示 ready 状态，例如：
     - `Asset ready: HQ/RAD`
     - `Asset ready: HQ/MECH`
     - `Asset ready: AF/CV`
     - `Asset ready: BB/CV`
   - 当该函数为 false 时，显示 need 状态，例如：
     - `Need HQ/RAD`
     - `Need HQ/MECH`
     - `Need AF/CV`
     - `Need BB/CV`
   - 文案可以使用等价短格式，但必须清楚表达 ready / need，且不要再在资产已满足时显示纯 `Needs ...`。
2. 资产判断口径必须复用现有实现。
   - 必须调用或等价复用 `hasOperationalSupportAsset(for:faction:)`。
   - 不要在 UI helper 中重写一套不一致的 operational asset 搜索规则。
   - 不要把未完工建筑、死亡实体、敌方实体、中立实体或不在现有 switch 中的实体计入支援资产。
3. 资产标签必须短、稳定、适合现有 HUD 行宽。
   - 推荐新增只读 helper，例如 `supportAssetRequirementLabel(for:) -> String`：
     - `.reconSweep` -> `HQ/RAD`
     - `.fieldRepair` -> `HQ/MECH`
     - `.airStrike` -> `AF/CV`
     - `.navalBarrage` -> `BB/CV`
   - 也可以用 `HQ/Radar`、`HQ/Mechanic`、`Airfield/CV`、`Battleship/CV` 等等价文案，但要注意 selection row 宽度。
   - 本轮不需要展示具体是哪一个资产满足，也不需要展示数量。
4. 保持支援 pending 面板其他行不变。
   - `$<cost>  CD <cooldown>s` 行保持既有语义。
   - `Ready now` / 冷却剩余秒数行保持既有语义。
   - 半径 + Repair / Dmg 行保持既有语义。
   - 面板 title 仍使用 `pendingSupportPower.displayName.uppercased()` 或现有等价逻辑。
5. 保持支援执行链路完全不变。
   - 不改变 `SupportPower` 的 cost、cooldown、radius、damage、repairAmount。
   - 不改变 `supportIssue(for:faction:)` 的检查顺序、返回文案或条件。
   - 不改变 `selectSupportPower(_:)`、`executeSupportPower(_:at:faction:showFeedback:)`、`supportDeniedReason(for:)`。
   - 不改变 `revealSupportTargets(...)`、`revealSupportArea(...)`、`applySupportRepair(...)`、`applySupportDamage(...)`、`showSupportPowerEffect(...)`。
6. 文档同步。
   - `README.md`：同步支援 pending selection panel 会显示支援资产 ready / need 状态；不要写成按钮禁用、费用变化、资产需求变化或新支援能力。
   - `md/flow/flow.md`：在 `SupportPower` 或 `HUD / 小地图` 段落同步当前真实行为，强调 pending 面板资产行复用 `hasOperationalSupportAsset`，只读显示 ready / need，不改变支援约束或效果。
   - `md/flow/flowchart.md`：如 HUD 节点已有支援价格/冷却/资产提示，可短语同步为「支援按钮与目标面板资产状态」或等价说法；如果不改，Agent B 必须说明图中节点已足够抽象且流程结构未变。
7. `update_log.md` 由 Agent C 在云端 artifact 验收通过后追加正式 v4.16 记录；Agent B 不要提前写正式验收结论。

推荐实现方向：

- 将 `supportAssetLine(for:)` 改为可判断阵营资产状态的 helper，例如：

```swift
private func supportAssetLine(for power: SupportPower, faction: Faction) -> String {
    let requirement = supportAssetRequirementLabel(for: power)
    if hasOperationalSupportAsset(for: power, faction: faction) {
        return "Asset ready: \(requirement)"
    }
    return "Need \(requirement)"
}

private func supportAssetRequirementLabel(for power: SupportPower) -> String {
    switch power {
    case .reconSweep:
        return "HQ/RAD"
    case .fieldRepair:
        return "HQ/MECH"
    case .airStrike:
        return "AF/CV"
    case .navalBarrage:
        return "BB/CV"
    }
}
```

- 上面只是示例，不要求逐字照抄。也可以保留 `supportAssetLine(for:)` 签名并只在其中固定使用 `.player`，但推荐传入 `faction`，这样 helper 语义更明确。
- 在 `selectionInfoContent(for:)` 的 `pendingSupportPower` 分支中，将第四行从：

```swift
supportAssetLine(for: pendingSupportPower)
```

改为类似：

```swift
supportAssetLine(for: pendingSupportPower, faction: .player)
```

- 如果复用 `supportAssetRequirementLabel(for:)` 给 `supportButtonSubtitle(for:)`，必须确保按钮 subtitle 行为完全不变：冷却时仍显示秒数，未冷却但缺资产时仍显示 `need asset`，资产满足时仍显示价格。

- 不要让资产行受冷却影响。冷却中但资产满足时，资产行仍应显示 ready；冷却中的剩余时间仍由现有 `Ready <seconds>s` 行表达。
