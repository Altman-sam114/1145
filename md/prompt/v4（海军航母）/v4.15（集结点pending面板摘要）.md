# v4.15（集结点pending面板摘要）

## 版本号

v4.15

## 版本分配依据

- `update_log.md` 当前最新正式玩法版本为 v4.14「海岸资产职责面板」。
- `md/prompt/v4（海军航母）/` 已连续围绕海军、航母、海岸资产、任务面板和 HUD 可读性做小步增强。
- v4.14 遗留事项中已把「集结点 pending 摘要」列为后续可推进方向。
- 本轮是一个范围很小的 HUD selection info 增量：玩家点击 `RLY` 进入 pending 后，选择信息面板显示本次可设置 rally 的来源摘要和操作提示，不改变 rally 合法性、生产来源、出兵、AI、移动路径或按钮布局。

## 背景

当前 rally 相关事实：

- `EntityKind.supportsRallyPoint` 当前只允许 `.barracks`、`.airfield`、`.shipyard`、`.carrier` 作为 rally 来源。
- 当前 `shortCode` 分别是 War Factory / `.barracks` = `WF`，Airfield = `AF`，Shipyard = `SY`，Carrier = `CV`。
- `rallyDomain(for:)` 当前将 `.barracks` 映射到 `.land`，`.airfield` 和 `.carrier` 映射到 `.air`，`.shipyard` 映射到 `.naval`。
- `selectedPlayerRallyFactories()` 当前按 `selectedIDs` 过滤 Blue、存活、`supportsRallyPoint` 的实体；它不要求结构 `isOperational`，本轮不要改变这个既有口径。
- 玩家点击 `RLY` 时，`handleHudAction(.setRally)` 会清理建筑预览、清空 pending construction/support、关闭 attack-move，读取 `selectedPlayerRallyFactories()`；若没有来源则提示 `Select a factory or carrier first.`，否则设置 `isSettingRallyPoint = true`、刷新 HUD 并提示 `Tap the map to set rally point.`。
- 世界点击时，`handleWorldTap(at:)` 在支援、建筑之后优先处理 `isSettingRallyPoint`；若来源为空则显示 denied marker 和提示，否则调用 `setRallyPoint(to:for:)`，然后关闭 pending。
- `setRallyPoint(to:for:)` 对每个来源调用 `navigablePoint(for:rallyDomain(for:), near:)`，写入 `factory.rallyPoint`，更新 rally marker，并刷新 selection/HUD。
- 当前 `updateHUD()` 在 `isSettingRallyPoint` 时只把顶部 `selectedLabel` 设置为 `Set rally point: tap map for selected factory or carrier.`。
- 当前 `selectionInfoContent(for:)` 处理 pending construction、pending support、pending attack-move、空选择、单选、多选，但没有 `isSettingRallyPoint` 分支。
- 单选 Carrier 已有 `carrierDeckQueueAndRallyLine(for:)` 显示 `Rally set` / `Rally unset`，Shipyard 也已有 compact queue/rally 相关文案；本轮要做的是 pending 期间的选择信息面板摘要，而不是单选普通状态文案重做。

人工指定本轮目标：为 v4.15 实现「集结点 pending 面板摘要」。玩家点击 `RLY` 进入 pending 后，选择信息面板显示当前可设置 rally 来源数量/类型（land/air/naval 或来源 short codes）、当前已选来源的 rally set/unset 摘要，以及 `Tap map to set rally`。不得修改 rally 合法性、`setRallyPoint`、生产来源、出兵、AI、移动路径或按钮布局。人工要求不得用本地测试/build/static check/git diff --check 作为验收，只认 GitHub Actions artifact。

## 目标

1. 在 `RLY` pending 状态增强选择信息面板。
   - 当 `isSettingRallyPoint == true` 时，`selectionInfoContent(for:)` 应优先返回 rally pending 专用内容。
   - 面板 title 建议为 `RALLY POINT` 或等价短标题。
   - rows 建议保持现有 4 行限制，不新增 selection info row label、不改布局。
   - 必须包含以下信息：
     - 当前可设置 rally 的来源数量，例如 `Sources 2`。
     - 来源类型摘要，可用 domain 形式，例如 `Land 1  Air 1  Naval 0`；也可用来源 short codes，例如 `Sources WF AF`。若使用 short codes，要避免长选择溢出，可做 prefix/suffix 压缩。
     - 当前已选来源的 rally set/unset 摘要，例如 `Rally set 1  Unset 1`。
     - 固定操作提示 `Tap map to set rally`。
2. 来源口径必须复用现有选择逻辑。
   - pending 摘要中的来源列表必须来自 `selectedPlayerRallyFactories()` 或与其完全一致的条件：Blue、存活、`kind.supportsRallyPoint`。
   - 不要额外加入 `isOperational` 条件，不要改变未完工 factory/carrier 被 `RLY` 选中后的现有行为。
   - 不要把 HQ、Oil Derrick、Radar Outpost、Sonar Buoy、Guard Tower、SAM Site、Coastal Battery、普通机动单位或敌方实体计入来源。
3. 类型摘要必须与现有 rally domain 一致。
   - War Factory / `.barracks` 计入 land。
   - Airfield 和 Carrier 计入 air。
   - Shipyard 计入 naval。
   - 推荐通过 `rallyDomain(for:)` 计算，避免写出与真实 rally landing point 不一致的并行规则。
4. rally set/unset 摘要只读现有状态。
   - `rallyPoint != nil` 计入 set。
   - `rallyPoint == nil` 计入 unset。
   - 不要在 pending 面板里预设、清空或修改 rally 点。
5. 保持顶部 `selectedLabel` 和按钮高亮现有语义。
   - `RLY` 按钮高亮仍由 `isHudActionArmed(.setRally)` / `isSettingRallyPoint` 驱动。
   - 顶部 `selectedLabel` 可继续显示现有 `Set rally point: tap map for selected factory or carrier.`；如需微调只能是等价短文案，不作为本轮重点。
6. 文档同步。
   - `README.md`：同步 `RLY` pending 时 selection panel 会显示 rally 来源数量/类型、set/unset 摘要和 `Tap map to set rally` 提示。
   - `md/flow/flow.md`：在输入与命令或 HUD/小地图段落同步该 pending 面板真实行为，并强调不改变 rally 合法性、执行、生产来源、出兵、AI、移动路径。
   - `md/flow/flowchart.md`：如 HUD/输入图中已有 pending/HUD 描述，可短语同步为「集结点 pending 来源摘要」或等价说法；如果不改，Agent B 必须说明图中节点已足够抽象且流程结构未变。
7. `update_log.md` 由 Agent C 在云端 artifact 验收通过后追加正式 v4.15 记录；Agent B 不要提前写正式验收结论。

```swift
private func rallyPointPendingInfoContent() -> (title: String, rows: [String]) {
    let sources = selectedPlayerRallyFactories()
    let land = sources.filter { rallyDomain(for: $0.kind) == .land }.count
    let air = sources.filter { rallyDomain(for: $0.kind) == .air }.count
    let naval = sources.filter { rallyDomain(for: $0.kind) == .naval }.count
    let setCount = sources.filter { $0.rallyPoint != nil }.count
    return (
        "RALLY POINT",
        [
            "Sources \(sources.count)",
            "Land \(land)  Air \(air)  Naval \(naval)",
            "Rally set \(setCount)  Unset \(sources.count - setCount)",
            "Tap map to set rally"
        ]
    )
}
```

- Agent B 完成后直接 `git push origin main` 触发 GitHub Actions，不用本地测试、本地 build、本地静态检查或 `git diff --check` 作为验收。
- GitHub Actions 必须上传未加密 CI 结果包，至少包含：
  - `ci-artifact-manifest.json`
  - `junit.xml`
  - `xcodebuild.log`
  - `ci-failure-summary.md`
  - `static-checks.log`
  - `project-lint.log`
  - `DesertFrontline.xcresult`
- Agent C 必须用 `gh auth login` 后下载最新 `origin/main` run 的 artifact，默认缓存到：

```text
/private/tmp/desert-frontline-c-review-<run_id>/
```

- Agent C 必须核对：
  - GitHub run 的 `headBranch=main`。
  - GitHub run 的 `headSha` 等于最新 `origin/main` commit。
  - manifest 的 `branch`、`commitSha`、`runId`、`runAttempt` 与 GitHub run 完全一致。
  - `buildOutcome=success`，workflow conclusion 成功。
  - `xcodebuild.log` 对应该 run，并包含成功构建结论。
  - `testOutcome` 如仍为 `skipped`，不得说成 XCTest 通过。
- Agent C 验收失败时，输出退回清单；Agent B 在 `main` 上追加修复 commit 后继续 push。不得回滚远端 `main`，除非人工明确要求。
