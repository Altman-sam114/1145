# v4.11（航母甲板状态面板）

## 版本号

v4.11

## 版本分配依据

- `update_log.md` 与 `md/prompt/` 当前最新正式玩法版本为 v4.10「AMOV 终局目标面板」。
- v4 阶段持续围绕海军、航母、终局攻势和 HUD 可读性做小步增强。
- 本轮继续做一个范围很小的 HUD-only 增量：让玩家单选 Carrier 时，选择信息面板明确说明 Carrier 甲板可生产 Helicopter / Fighter，并显示 rally point 是否已设置。

## 背景

当前 Carrier 已经是移动海上空军生产平台：

- `EntityKind.carrier.canProduce(_:)` 与 Airfield 一样可生产 `.helicopter` 和 `.fighter`。
- `queueBuild(kind:faction:showFeedback:)` 对空军单位会从 Airfield 或 Carrier 中选择可用生产来源；玩家选中 Carrier 时，`productionSource(for:faction:)` 会优先使用选中的可生产来源。
- `updateBuildOrders(dt:)` 在 Carrier 完成空军生产时会从甲板附近出机，并调用 `showCarrierLaunch(...)` 显示短暂 deck-launch 反馈。
- `selectedPlayerRallyFactories()`、`setRallyPoint(to:for:)` 和 `rallyDomain(for:)` 已支持 Carrier，Carrier 的 rally domain 是 `.air`。
- 但 `singleSelectionInfo(for:)` 目前把 Carrier 当作移动战斗单位，再通过 `mobileStatusLine(for:) -> structureStatusLine(for:)` 只显示通用队列状态，例如 `Queue idle  Rally ready` 或 `Queue HEL 8s`。这不够明确：玩家单选 Carrier 时，面板没有直接告诉他 Carrier 可从甲板生产 Helicopter/Fighter，也没有清楚区分 rally 已设置 / 未设置。

本轮只补选择信息面板文案，不改变任何实际生产或 rally 行为。

## 目标

1. 增强 Carrier 单选选择信息面板。
   - 条件：玩家或 Red 的单个 `Carrier` 被选中并进入 `singleSelectionInfo(for:)`。
   - 面板必须明确显示 Carrier 可从甲板生产 `Helicopter/Fighter`。
   - 面板必须显示 rally point 是否已设置，例如 `Rally set` / `Rally unset`。
   - 如果 Carrier 当前有生产队列，仍应保留当前队列状态，例如 `Queue HEL 8s` 或 `Queue JET 9s +1`。
   - 如果 Carrier 无队列，应显示 `Queue idle`。
2. 保持选择面板紧凑。
   - 不新增面板行数，不新增滚动、分页或新 HUD 控件。
   - 文案要短，避免在现有 selection info panel 中溢出。
   - 推荐保持 4 行内表达：HP、攻防/ASW/sonar、甲板生产能力、rally/queue 状态。
3. 保持所有游戏语义不变。
   - 不改变 `EntityKind.canProduce(_:)`、`requiredFactory`、`productionSource(for:faction:)`、`queueBuild(...)`、`updateBuildOrders(dt:)`、`spawnPoint(for:from:)`、`showCarrierLaunch(...)`。
   - 不改变 `selectedPlayerRallyFactories()`、`setRallyPoint(to:for:)`、`rallyDomain(for:)`、`updateRallyMarker(for:)` 或 `rallyNode` 显隐规则。
   - 不改变 AI、战斗、战争迷雾、潜艇侦测、支援技能、任务、胜负、地图、Xcode project 或 workflow。
4. 文档同步。
   - `README.md`：同步 Carrier 单选面板会显示 deck 可生产 Helicopter/Fighter 以及 rally set/unset 状态。
   - `md/flow/flow.md`：同步 HUD / 生产相关说明，明确这是 selection info 文案增强，不是新生产系统。
   - `md/flow/flowchart.md`：如当前图中提到航母甲板或选择信息，可做一句短同步；若图无需变更，Agent B 必须说明原因。
5. `update_log.md` 由 Agent C 在云端 artifact 验收通过后追加正式记录。

## 非目标

- 不新增 Carrier 可生产单位；本轮只展示既有 Helicopter/Fighter 能力。
- 不改变空军生产来源选择、队列排序、费用、建造时间、出机点、deck-launch 视觉反馈或 ready pulse。
- 不改变 rally point 合法性、rally domain、rally marker、rally 点击流程、无效目标反馈或新单位出厂后前往 rally 的行为。
- 不改变 Carrier 战斗数值、射程、伤害、冷却、视野、声呐、反潜、wing-strike 视觉、老兵 XP 或 AI 使用。
- 不改变 Airfield、War Factory、Shipyard 或普通结构的选择面板，除非为了抽取通用 queue 文案 helper 做最小必要改动。
- 不泄露迷雾信息，不改变敌方单位可见性、`isKnownToFaction(...)` 或选择合法性。
- 不新增外部依赖、素材、XCTest target、workflow 或 Xcode project 配置。
- 不更新 `md/test/test.md`。
- 不引入 `smalldata_test`、`develop`、`codeb/...`、候选分支或 PR 流。

2. 修改 `DesertFrontline/GameScene.swift`。

推荐实现方向：

- 在 `singleSelectionInfo(for:)` 的非结构分支里，对 `entity.kind == .carrier` 做一个窄 special-case。
- 推荐新增一个小 helper，例如：

```swift
private func carrierDeckStatusRows(for entity: GameEntity) -> (deck: String, status: String)
```

或拆成两个短 helper：

```swift
private func carrierDeckCapabilityLine() -> String
private func carrierDeckQueueAndRallyLine(for entity: GameEntity) -> String
```

建议文案：

- 甲板生产能力行：`Deck builds Helicopter/Fighter`
- rally / queue 行：
  - 无队列、未设置 rally：`Rally unset  Queue idle`
  - 无队列、已设置 rally：`Rally set  Queue idle`
  - 有队列、未设置 rally：`Rally unset  Queue HEL 8s`
  - 有队列、已设置 rally：`Rally set  Queue JET 9s +1`

实现要点：

- 可复用现有 `buildOrders.filter { $0.sourceID == entity.id }` 逻辑，避免复制太多生产状态格式化代码。
- 也可以安全复用 / 调整 `structureStatusLine(for:)`，但不得让 War Factory / Airfield / Shipyard 的现有文案意外变化，除非 diff 非常小且文档同步。
- `entity.rallyPoint == nil` 表示 `Rally unset`，非空表示 `Rally set`。不需要在 selection info 里显示坐标；顶部 `selectedLabel` 已能在可取 tile 时显示坐标。
- 保留 `selectionCapabilityInfoLine(for:)` 对 Carrier 的 ASW attack / sonar 信息追加，不要因为 Carrier special-case 丢掉反潜和声呐信息。
- 保持选择面板最多 4 行。若需要取舍，优先保留：HP、攻击/射程/视野/ASW/sonar、Deck builds Helicopter/Fighter、Rally set/unset + Queue。

必须保持：

- 不修改 `queueBuild(...)`、`productionSource(...)`、`updateBuildOrders(dt:)` 和 rally 执行函数。
- 不修改 `HudAction`、底部按钮、pending 状态或触摸流程。
- 不修改 AI / combat / fog / mission / victory 逻辑。

## 风险和禁止项

- 风险：把 Carrier special-case 写在错误位置，导致 ASW / sonar info 被覆盖。必须确认 `selectionCapabilityInfoLine(for:)` 仍会追加到攻击信息行。

- 风险：把 `Rally ready` 当成 rally 已设置。`Rally ready` 只表示可设置 rally；本轮必须区分 `entity.rallyPoint == nil` 与非空。
