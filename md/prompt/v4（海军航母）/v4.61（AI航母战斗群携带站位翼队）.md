# v4.61 / AI 航母战斗群携带站位翼队

## 版本号

v4.61

## 版本分配依据

上一轮 v4.60 完成 Red 主攻波 HUD 摘要过期。v4.58 已让 Red Carrier 被接受进入 routine assault wave 时携带自身 bound guard wing，但当前同 anchor HEL/JET 若正在执行 Carrier guard station keeping，可能因为 `destination != nil` 被 `isAvailableEnemyCarrierGuardWing(_:)` 排除，导致 Carrier 出击时未带上正在回位的警戒翼队。本轮补齐这个窄行为，仍属于 v4 海军航母阶段的小步修正。

## 背景

当前相关事实：

- Red 空闲 Carrier 会绑定附近空闲 HEL/JET 作为 guard wing。
- 已绑定 Red guard wing 平时会被 `isEnemyCarrierGuardWingReservedForAnchor(_:)` 排除出 ordinary routine assault candidates。
- v4.58 的 `enemyAssaultCandidatesWithCarrierGuardWings(from:)` 只在 anchor Carrier 自身进入 provisional wave 时追加同 anchor bound wing。
- `isAvailableEnemyCarrierGuardWing(_:)` 要求 `destination == nil`，适合普通绑定候选，但会排除正在返回 Carrier guard station 的同 anchor wing。
- `issueFormationMove(... attackMove: true)` 会清理 included HEL/JET 的 old guard anchor，让它们随队进入 attack-move。

## 目标

- 当 Red Carrier 自身进入 routine provisional assault wave 时，允许该 Carrier 当前有效、同 anchor、无 attack target、无 attack-move、未撤退的 bound HEL/JET 随队参与本轮 assault candidates，即使这些 wing 正在向 guard station 移动。
- 继续阻止无 anchor、其他 Carrier anchor、低血撤退、占点 reservation、已有 attack target 或已有 attack-move 的 Red HEL/JET 被携带。
- 保持 ordinary routine wave 仍不会直接抽走 bound guard wing；只有 anchor Carrier 自身入波时才可携带。

## 非目标

- 不改变玩家 Carrier guard wing、玩家命令释放语义、HOLD station keeping、guard contact 优先级、范围圈、HUD 文案或顶部消息。
- 不改变 AI 选波目标、escort 门槛、低血撤退、Veteran / Elite 保护、生产、支援、迷雾、潜艇侦测、任务、胜负或 SKRM 重置语义。
- 不新增 CAP、巡逻、截击状态机、战斗加成、战报面板或外部依赖。

## 当前架构依据

- `enemyAssaultCandidatesWithCarrierGuardWings(from:)` 是 Red Carrier 入波携带 guard wing 的唯一追加入口。
- `isAvailableEnemyCarrierGuardWing(_:)` 仍适合普通 idle 绑定候选，不应为本轮放宽以免影响 AI 自动绑定或 ordinary candidates。
- `carrierGuardAnchor(for:)` 已要求 wing 是 HEL/JET、anchor Carrier 存活、同阵营且仍 HOLD。
- `shouldRetreatEnemyAssaultUnit(_:)` 和 `isEnemyUnitRetreating(_:)` 已提供低血撤退边界。

## 实现步骤

1. 在 `GameScene.swift` 增加一个窄 helper，例如 `canEnemyCarrierGuardWingJoinAssault(_ wing: GameEntity, with carrier: GameEntity) -> Bool`。
2. helper 必须要求：
   - `wing.faction == .enemy`
   - `wing.isAlive && wing.isOperational`
   - `wing.kind` 是 `.helicopter` 或 `.fighter`
   - `wing.attackTarget == nil`
   - `wing.attackMoveDestination == nil`
   - `!isEnemyCaptureReserved(wing)`
   - `!isEnemyUnitRetreating(wing)`
   - `!shouldRetreatEnemyAssaultUnit(wing)`
   - `carrierGuardAnchor(for: wing)?.id == carrier.id`
3. helper 不要求 `wing.destination == nil`，允许站位回位中的同 anchor wing 被 anchor Carrier 携带。
4. 修改 `enemyAssaultCandidatesWithCarrierGuardWings(from:)`，用该 helper 替代 `isAvailableEnemyCarrierGuardWing(wing)`、`!shouldRetreatEnemyAssaultUnit(wing)` 和重复 anchor 判断。
5. 保持 `isAvailableEnemyCarrierGuardWing(_:)` 原样，不改变 Red Carrier 自动绑定候选和普通 idle 条件。
6. 同步 README 的 Enemy AI 描述，说明 Carrier 被接受入波时可携带当前 bound guard aircraft，即使它们正在返回 guard station。
7. 同步 `md/flow/flow.md` 的 AI 段，写清该放宽只适用于 anchor Carrier 入波携带，不适用于 ordinary routine candidate。
8. `md/flow/flowchart.md` 当前未细分该行为，可不改。
