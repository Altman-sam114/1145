# v4.0（声呐浮标专职反潜再平衡）

## 版本号

v4.0

## 版本分配依据

- 当前 `md/prompt/` 与 `update_log.md` 最新版本为 v3.9「AI低血单位撤退回修」。
- v3 阶段已经围绕海战反潜、AI 认知、公平侦察、混编主攻和高价值单位保护做了多轮闭环。
- v4.0 开启海军 / 航母深化阶段，但本轮只做一个小而可验证的平衡增量：把 Sonar Buoy 明确成便宜、脆弱、覆盖稍大的沿海专职反潜传感器。

## 背景

当前 Sonar Buoy 是可建造海岸结构，完工后提供普通建筑视野和 sonar 检测，不攻击、不生产、不赚钱，也不作为 `SCAN` 资产。它当前成本较高、HP 不低，sonar range 是 300；README 和 flow 已说明它是 detector，但没有明确“脆弱专职传感器”的平衡定位。

本轮只调整 Sonar Buoy 静态数值和文档，让它更适合作为前沿海岸反潜传感器：部署成本更低、建造更快、HP 更低、普通视野更低、sonar range 稍高。不要改潜艇检测算法、迷雾合法性、AI 认知边界或支援资产规则。

## 目标

1. 调整 `EntityKind.sonarBuoy` 静态数值。
   - 建议 cost：`1150 -> 850`。
   - 建议 buildTime：`9.0 -> 7.0`。
   - 建议 maxHP：`540 -> 360`。
   - 建议 visionTiles：`6 -> 4`。
   - 建议 sonarRange：`300 -> 340`。
2. 保持 Sonar Buoy 专职传感器定位。
   - 仍然 `hasSonar == true`。
   - 仍然 `damage == 0`，不攻击。
   - 仍然不生产、不赚钱、不作为 `SCAN` 资产。
   - 仍然只有 `isOperational` 后才提供视野、sonar、覆盖圈和 AI 巡扫 anchor。
3. HUD / 选择信息必须自动或明确显示新数值。
   - `singleSelectionInfo(...)` 中 Sonar Buoy 文案仍应显示 `Sonar \(Int(sonarRange(for: entity.kind)))  No SCAN`，并随新 range 变成 340。
   - 选择态 sonar 覆盖圈仍由 `sonarRange(for:)` 派生，不硬编码第二份范围。
4. 文档同步。
   - `README.md`：Current Features 中说明 Sonar Buoy 是低成本、脆弱、较大 sonar coverage 的 coastal detector。
   - `md/flow/flow.md`：`EntityKind`、迷雾与 AI 小节同步成本/脆弱/覆盖定位，不写成会攻击或升级。
   - `md/flow/flowchart.md`：AI 或 Fog 节点可简短同步“专职声呐浮标覆盖”。
5. `update_log.md` 由 Agent C 在云端 artifact 验收通过后追加正式记录。

## 非目标

- 不新增 Sonar Buoy 升级、科技树、HUD 升级按钮、费用 UI、选择命令或外部素材。
- 不让 Sonar Buoy 攻击、生产、赚钱、维修、移动、提供 `SCAN` 资产或 ASW attacker 计数。
- 不改 `isSubmarineDetected(...)`、`isKnownToFaction(...)`、`revealedUntil`、`supportRevealTiles`、`SCAN` / `AIRS` / `BARR` 行为、潜艇开火暴露或支援命中暴露。
- 不改 AI 动态 ASW 生产、Red `SCAN` 目标选择公平性、AI 读取隐藏潜艇边界、routine assault、撤退回修或防守响应。
- 不改玩家命令、选择、建造合法性、海岸规则、碰撞规则、单位武器数值、Xcode project 或 GitHub Actions workflow。
- 不引入 `smalldata_test`、`develop`、`codeb/...`、候选分支或 PR 流。

关键源码事实：

- `EntityKind` 中 Sonar Buoy 的 cost / buildTime / maxHP / visionTiles 是静态数值来源。
- `sonarRange(for:)` 中 Sonar Buoy range 当前为 300；潜艇检测和选择态覆盖圈都读取该函数。
- `isActiveSonarSensor(_:)` 会要求结构 operational，因此未完工 Sonar Buoy 不显示覆盖圈，也不作为 active sonar sensor。
- `hasOperationalSupportAsset(for: .reconSweep)` 只接受 HQ 或 Radar Outpost；Sonar Buoy 不计入 `SCAN` 资产。
- AI `enemyReconPatrolAnchors()` 可把 operational Sonar Buoy 作为巡扫 anchor，但不读取隐藏玩家潜艇坐标。
- `enemyAntiSubmarineAssetCount()` 只统计非结构 ASW 单位；Sonar Buoy 不是 ASW attacker。
- `constructionIssue(...)` 中 Shipyard、Sonar Buoy 和 Coastal Battery 仍需要 coast tile；本轮不改。

## 验收标准

- Sonar Buoy cost/buildTime/maxHP/visionTiles/sonarRange 按本提示词调整。
- Sonar Buoy 仍只在 operational 后提供视野、sonar 检测和选择态覆盖圈。
- Sonar Buoy 仍不攻击、不生产、不赚钱、不作为 `SCAN` 资产、不计入 ASW attacker。
- HUD selection 中 Sonar Buoy range 与 `sonarRange(for:)` 一致；选择态覆盖圈也来自同一 range。
- Red AI `SCAN` 巡扫仍只用 Red-known 潜艇或己方 sonar / 旗点热点，不读取隐藏潜艇真实坐标。
- 动态 ASW 生产、玩家支援技能、潜艇暴露、迷雾合法性、建造海岸规则、单位战斗数值和 Xcode/workflow 配置无变化。
- README、flow、flowchart 与真实行为一致。
- 本轮提交已 push 到 `origin/main` 并触发 GitHub Actions。
- Agent C 能下载最新 run artifact 并核对 manifest / JUnit / xcodebuild.log / failure summary / xcresult。

## 风险和禁止项

- 风险：sonar range 增大后，如果文档暗示普通视野也增大，会造成玩家误解；必须明确普通视野降低、sonar coverage 增大。
- 风险：误把 Sonar Buoy 加入 `SCAN` 资产或 ASW attacker，破坏既有反潜边界。
- 风险：改 `isKnownToFaction(...)` 或 `isSubmarineDetected(...)` 会影响潜艇隐身和迷雾合法性，禁止。
- 风险：只调 range 不更新 README / flow，会让玩家可见事实和架构文档脱节。
- 禁止新增升级系统、科技树、UI 按钮或新资源。
- 禁止使用非 `Altman-sam114` 的 GitHub 账号伪装 push、CI 或 artifact 验收。
- 禁止下载大体积历史 artifact、模型、测试数据或无关产物。
