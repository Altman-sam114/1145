# v4.13（支援按钮资产提示）

## 版本号

v4.13

## 版本分配依据

- `update_log.md` 与 `md/prompt/` 当前最新正式玩法版本为 v4.12「海岸据点任务摘要」。
- v4 阶段持续围绕海军、航母、海岸目标、任务面板和 HUD 可读性做小步增强。
- 本轮是一个范围很小的 HUD subtitle 可读性增量：让玩家在不点击支援技能按钮的情况下，直接看出技能是否处于冷却、是否缺少 operational 支援资产，或是否已具备资产并显示费用。

## 背景

当前底部 HUD 已有四个支援技能按钮：

- `SCAN` / `SupportPower.reconSweep`
- `REPR` / `SupportPower.fieldRepair`
- `AIRS` / `SupportPower.airStrike`
- `BARR` / `SupportPower.navalBarrage`

现有源码事实：

- `HudAction.supportPower` 把四个 HUD action 映射到对应 `SupportPower`。
- `subtitle(for:)` 当前对支援技能的逻辑是：
  - 若 `supportCooldown(for: .player, power:) > 0`，显示 `ceil(cooldown)` 秒数。
  - 否则显示 `$cost`。
- `updateHUD()` 每帧刷新按钮 subtitle，并在技能冷却中把 subtitle 字色改成橙色。
- `supportIssue(for:faction:)` 是点击支援技能和执行支援技能时的合法性入口，按顺序检查冷却、资金、operational asset。
- `hasOperationalSupportAsset(for:faction:)` 是支援资产需求的真实口径：
  - `SCAN` 需要 operational HQ 或 Radar Outpost。
  - `REPR` 需要 operational HQ 或 Mechanic。
  - `AIRS` 需要 operational Airfield 或 Carrier。
  - `BARR` 需要 operational Battleship 或 Carrier。
- `supportAssetLine(for:)` 已在 pending support 选择信息中显示较长资产需求说明，但底部按钮在未点击前看不到缺资产状态。

人工指定本轮目标：增强支援技能 HUD 按钮 subtitle。冷却中仍显示秒数；非冷却且缺少所需 operational 支援资产时显示短提示 `need asset`（或同等短句）；资产满足时仍显示价格。只改 HUD 可读性，不改变支援技能执行、费用、冷却、资产需求、AI 或迷雾。

## 目标

1. 增强底部支援技能按钮 subtitle。
   - 对 `SCAN`、`REPR`、`AIRS`、`BARR` 生效。
   - 冷却中必须继续显示剩余秒数，例如 `17s`。
   - 冷却显示优先级最高；即使资产缺失，只要技能仍在冷却中，subtitle 仍显示秒数。
   - 非冷却且缺少对应 operational 支援资产时，subtitle 显示短提示，推荐固定为：

```text
need asset
```

   - 如果 HUD 按钮过窄导致文本风险较高，可使用同等短句，例如 `need src` 或 `need base`，但四个支援技能应保持一致。
   - 非冷却且 operational 支援资产满足时，subtitle 仍显示 `$cost`。
2. 复用现有资产判断口径。
   - 缺资产判断必须复用 `hasOperationalSupportAsset(for:faction:)`，或抽出等价 helper 后让执行逻辑和 HUD 共享同一口径。
   - 不得用新的并行规则判断资产。
   - 不得把未完工、死亡、敌方、非 operational 的实体算作可用支援资产。
3. 不改变支援技能执行语义。
   - `selectSupportPower(_:)`、`executeSupportPower(_:at:faction:showFeedback:)`、`supportIssue(for:faction:)` 的执行检查结果必须保持原语义。
   - 支援技能点击后仍按现有顺序检查冷却、资金、资产，并显示现有 message / denied marker。
   - 本轮只改变按钮 subtitle 的可读性，不改变按钮是否可点、pending support 状态、地图目标合法性、扣款、冷却写入或效果。
4. 保持 HUD 紧凑。
   - 不改 HUD 按钮数量、布局、命中区域、底栏高度、按钮标题或按钮颜色，除非为避免明显文本不可读做最小调整并在输出说明。
   - 不新增弹窗、教程、面板行、图标或地图反馈。
5. 文档同步。
   - `README.md`：同步支援技能按钮会在冷却时显示秒数、缺少 operational 支援资产时显示短提示、资产满足时显示价格。
   - `md/flow/flow.md`：在 `SupportPower` 或 `HUD / 小地图` 段落同步支援按钮 subtitle 的真实行为，并强调执行链路仍由 `supportIssue` / `hasOperationalSupportAsset` 约束。
   - `md/flow/flowchart.md`：如 HUD 节点已有命令/支援信息，可短语同步为“支援价格/冷却/资产提示”或等价说法；如果图已经足够概括且不改，Agent B 必须说明原因。
6. `update_log.md` 由 Agent C 在云端 artifact 验收通过后追加正式记录。

## 非目标

- 不改变 `SupportPower` 的 cost、cooldown、radius、damage、repairAmount。
- 不改变 `supportIssue(for:faction:)` 的执行语义、检查顺序、错误文案或 denied reason，除非为了复用 helper 做完全等价的内部整理。
- 不改变 `hasOperationalSupportAsset(for:faction:)` 对任一支援技能的资产需求。
- 不改变玩家或 AI 使用支援技能的逻辑、评分、阈值、保留资金、目标选择或效果。
- 不改变 `supportCooldowns`、`supportRevealTiles`、支援侦察、维修、伤害、潜艇暴露或战争迷雾规则。
- 不改变按钮可点击性；缺资产时按钮仍可点，但点击后仍走现有 `supportIssue` 提示。
- 不新增 disabled 状态、灰化状态、长按说明、教程文本、HUD 新行、图标或新音效。
- 不改变生产、建造、任务阶段、胜负、AMOV、rally、Carrier deck、老兵、控制组、小地图或 AI。

2. 修改 `DesertFrontline/GameScene.swift`。

推荐实现方向：

- 在 `subtitle(for:)` 的 `if let power = action.supportPower` 分支中加入资产缺失判断。
- 保持判断优先级：

```swift
if let power = action.supportPower {
    let cooldown = supportCooldown(for: .player, power: power)
    if cooldown > 0 {
        return "\(Int(ceil(cooldown)))s"
    }
    if !hasOperationalSupportAsset(for: power, faction: .player) {
        return "need asset"
    }
    return "$\(power.cost)"
}
```

- 如果担心 `subtitle(for:)` 直接承载太多分支，可抽出一个小 helper，例如：

private func supportButtonSubtitle(for power: SupportPower) -> String

## 验收标准

- `SCAN`、`REPR`、`AIRS`、`BARR` 冷却中 subtitle 显示剩余秒数，例如 `17s`。
- 冷却显示优先级最高：冷却中即使缺资产，也显示秒数而不是 `need asset`。
- 非冷却且缺少对应 operational 支援资产时，subtitle 显示短提示，推荐 `need asset` 或 Agent B 明确说明的同等短句。
- 非冷却且 operational 支援资产满足时，subtitle 仍显示 `$cost`。
- 缺资产判断与 `hasOperationalSupportAsset(for:faction:)` 同口径，只统计指定 faction、存活、`isOperational` 的支援资产。
- 资金不足但资产满足且不冷却时，subtitle 仍显示 `$cost`；点击后仍由现有 `supportIssue` 报资金不足。
- 支援技能执行、费用、冷却写入、资产需求、AI 支援选择、迷雾、潜艇暴露、维修、伤害、pending support 和 denied marker 行为不变。
- HUD 布局、按钮数量、按钮标题、命中区域和 pending 高亮不发生无关改动。
- README、flow、flowchart 与真实行为一致，不夸大为禁用按钮、新资源系统、新支援技能或 AI 新能力。
- 本轮提交已 push 到 `origin/main` 并触发 GitHub Actions。
- Agent C 能下载最新 run artifact 并核对 manifest / JUnit / xcodebuild.log / failure summary / xcresult。

## 风险和禁止项

风险：

- `need asset` 比价格长，可能在紧凑按钮上显得拥挤。若实际按钮宽度不足，可改为更短但语义清楚的统一短句，并在 Agent B 输出中说明。
- 如果 subtitle 直接复用 `supportIssue(for:)`，可能因为资金不足返回资金问题，违反“资产满足时仍显示价格”的要求；本轮 HUD subtitle 不应检查资金。
- 如果复制一套资产判断，后续资产需求变化可能导致 HUD 和执行链路不一致；必须复用或共享 `hasOperationalSupportAsset(for:faction:)` 口径。
- 冷却、资金和资产三者的优先级容易混淆；本轮 subtitle 优先级必须是冷却秒数 > 缺资产短提示 > 价格。

禁止：

- 禁止改变任何支援技能执行结果、费用、冷却、资产需求、AI 使用或迷雾效果。
- 禁止让缺资产按钮无法点击；现有点击反馈仍由 `supportIssue` 负责。
- 禁止把未完工建筑、死亡实体、敌方实体或非 operational 实体算作可用支援资产。
- 禁止新增 HUD 结构、弹窗、教程、图标、素材、依赖、测试 target、workflow 或 Xcode project 改动。
- 禁止更新 `md/test/test.md`。
- 禁止用本地测试、本地 build、本地静态检查或 `git diff --check` 作为验收。
- 禁止伪造 GitHub Actions、artifact、manifest、JUnit 或 xcodebuild 日志结果。
- 禁止使用非 `Altman-sam114` 的 GitHub 账号伪装完成 push、CI 或 artifact 验收。
