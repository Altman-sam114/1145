# v4.55 / 航母警戒翼队脱离组成反馈

## 版本号

v4.55

## 版本分配依据

上一轮 v4.54 已统一 Carrier guard wing 的 2 架规模口径。当前玩家移动、attack-move、直接攻击或 ordinary HOLD 让 bound Carrier guard aircraft 脱离 anchor 时，顶部反馈只显示 `CV guard released n.`，而 v4.49-v4.54 已经让 guard wing 的 HUD、分配和多选反馈都具备 `Hn` / `Jn` 组成口径。本轮把 release feedback 也补齐同一组成提示，属于 v4 海军 / 航母反馈一致性小步增量。

## 背景

当前相关事实：

- `carrierAirWingCompositionSuffix(for:)` 已提供 Helicopter / Fighter 组成后缀，例如 ` H1 J1`、` H2`、` J1`。
- `issueHoldPositionOrder(units:)` 在 Carrier HOLD 成功绑定 guard wing 时已经显示 `Guard wing n Hn/Jn.` 和每艘 Carrier 的 `GW Hn/Jn` deck cue。
- `groupSelectedCarrierGuardWingSummary(for:)` 和 `groupCarrierGuardWingSummary(for:)` 已在选择面板显示 bound guard wing 组成。
- `carrierGuardReleaseCount(for:)` 和 `carrierGuardReleaseSuffix(count:)` 目前只统计数量，释放反馈无法看出脱离的是 HEL、JET 还是混合翼队。

## 目标

- 玩家成功移动、attack-move、直接攻击或 ordinary HOLD 导致有效 bound Carrier guard HEL/JET 脱离 anchor 时，顶部成功消息从 `CV guard released n.` 扩展为 `CV guard released n Hn/Jn.`。
- 组成统计只针对本次实际会释放的有效 player HEL/JET guard wing。
- 复用现有 `carrierAirWingCompositionSuffix(for:)`，不要另写一套 HEL/JET 组成格式。
- 保持现有释放计数口径：只有 `carrierGuardAnchor(for:) != nil` 的玩家 Helicopter / Fighter 计入释放反馈。

## 非目标

- 不改变哪些命令会释放 Carrier guard wing。
- 不改变 `clearCarrierGuardAnchor(for:)`、`assignCarrierGuardWing(for:)`、`carrierGuardAnchor(for:)`、`boundCarrierGuardWing(for:)` 或 `nearbyCarrierAirWing(for:)` 的语义。
- 不改变 Carrier HOLD 绑定数量、候选排序、station keeping、target priority、guard ring、HUD 接触摘要、combat、fog、潜艇侦测、AI、生产、支援、任务、胜负或 SKRM 重置。
- 不新增 CAP、巡逻、截击、新按钮、新状态机或 AI 策略。

## 当前架构依据

- 直接攻击在世界点击 enemy 分支中先调用 `carrierGuardReleaseCount(for: attackers)`，再对 attackers 清理 `holdPosition` 和 guard anchor，最后调用 `carrierGuardReleaseSuffix(count:)`。
- 普通移动和 attack-move 在 `issueFormationMove(...)` 中先调用 `carrierGuardReleaseCount(for: mobileUnits)`，随后清理每个移动单位的 hold/anchor/attack 状态，最后调用 `carrierGuardReleaseSuffix(count:)`。
- ordinary HOLD 在 `issueHoldPositionOrder(units:)` 中先保存 `previouslyGuardedWingIDs`，执行 HOLD 和可能的 Carrier 重新绑定后，再统计最终 anchor 已清空的旧 wing，避免把同次重新绑定的 wing 误报为 released。
- `carrierAirWingCompositionSuffix(for:)` 已稳定用于 Carrier wing readiness、guard wing HUD、多选和分配反馈。

## 实现步骤

1. 新增一个轻量只读 helper，例如：
   - `carrierGuardReleaseWing(for units: [GameEntity]) -> [GameEntity]`
   - 过滤条件与当前 `carrierGuardReleaseCount(for:)` 相同：玩家、存活、Helicopter / Fighter、`carrierGuardAnchor(for:) != nil`。
2. 将 `carrierGuardReleaseCount(for:)` 改为复用该 helper 的 `count`，或在所有调用点改用 wing 列表后移除旧 count helper，避免口径分叉。
3. 将 `carrierGuardReleaseSuffix(count:)` 改为接收 wing 列表或新增替代 helper，例如 `carrierGuardReleaseSuffix(for wing: [GameEntity])`：
   - 空列表返回空字符串。
   - 非空返回 ` CV guard released \(wing.count)\(carrierAirWingCompositionSuffix(for: wing)).`
4. 修改直接攻击和 `issueFormationMove(...)`，在清理 anchor 前保存 release wing 列表，并用新 suffix 生成消息。
5. 修改 ordinary HOLD：
   - 继续先保存此前有效 bound wing 的 id。
   - HOLD / 重新绑定完成后，用这些 id 找回仍存在、且最终 `carrierGuardAnchor(for:) == nil` 的实际 released wing 列表。
   - 用同一个 release suffix 输出组成。
6. 同步 README 和 flow 中 release feedback 描述，明确脱离反馈现在包含 `Hn` / `Jn` 组成。
7. flowchart 如需同步，只做最小文字更新；若图中已有“脱离反馈”且不需要更细文案，可不改。

## 验收标准

- 玩家移动 bound guard HEL/JET 时，成功移动消息包含 `CV guard released n Hn/Jn.`。
- 玩家 attack-move bound guard HEL/JET 时，成功 AMOV 消息包含同一组成后缀。
- 玩家直接攻击导致 bound guard HEL/JET 脱离时，攻击成功消息包含同一组成后缀。
- 玩家 ordinary HOLD 让旧 bound guard HEL/JET 脱离且没有同次重新绑定时，HOLD 成功消息包含同一组成后缀；同次重新绑定的 wing 不计入 released。
- 没有实际释放 bound guard HEL/JET 时，不显示 release suffix。
- 现有 Carrier HOLD 分配组成反馈、deck cue、HUD `GW/Wing` 组成、guard ring、target priority、combat、fog、production、support、mission 和 victory 语义不变。
- 最新 `origin/main` GitHub Actions run 成功，artifact 对应本轮 commit。

## 风险和禁止项

- release feedback 可能出现在同一条成功消息后半段；本轮只补短组成，不扩大文案。
- 直接攻击、move、AMOV 的 release wing 列表必须在清理 anchor 前保存。
- ordinary HOLD 必须在重新绑定后再计算最终 released wing，保持 v4.50 后的“同次重新绑定不误报 release”语义。
- 禁止修改候选资格、半径、迷雾、潜艇侦测、AI 作弊边界、战斗目标合法性或生产来源。
- 禁止新增外部依赖、素材、测试 target 或大体积 artifact。
