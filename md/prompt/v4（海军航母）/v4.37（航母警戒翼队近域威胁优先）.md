## 2. 版本分配依据

- 当前 `md/prompt/v4（海军航母）/` 最新版本为 `v4.36（航母警戒翼队站位保持）.md`，本轮延续 v4 海军航母阶段。

- 下一步不直接实现完整 CAP / 巡逻 / 截击系统，而是在现有 HOLD 目标搜索前增加一个小范围目标优先级：绑定翼队优先处理 Carrier 近域内自己已知且可攻击的威胁。

## 3. 背景

当前 `GameScene.swift` 中：

- `GameEntity.guardAnchorCarrierID` 记录 HEL/JET 的 Carrier guard wing 归属。
- `updateCarrierGuardStation(for:)` 在移动更新中维护绑定翼队的 `holdPosition` 到 Carrier 附近稳定站位。
- `updateCombat(dt:)` 在 `entity.attackTarget == nil` 时按普通 HOLD / AMOV / 射程口径调用 `nearestTarget(for:maxDistance:)`。
- `nearestTarget(for:maxDistance:)` 已统一检查：目标存活、敌对、非中立、`canAttack`、距离、HOLD 半径和 `isKnownToFaction(...)`。
- 本轮必须复用这些既有边界，不能绕过迷雾、潜艇隐身、`canAttack` 或普通 HOLD 目标清理。

## 4. 本轮目标

实现一个小范围、可验收的 Carrier guard wing 战术增量：

1. 为已绑定到 Carrier 的 Helicopter / Fighter 增加轻量目标优先 helper，例如 `carrierGuardPriorityTarget(for:)`。
2. 该 helper 只在以下条件全部成立时生效：
   - 单位是 `.helicopter` 或 `.fighter`。
   - `guardAnchorCarrierID` 指向仍存活、同阵营、kind 为 `.carrier` 且 `holdPosition != nil` 的 Carrier。
   - 目标是敌方、存活、非中立、该翼队可攻击、对该翼队阵营已知。
   - 目标位于 Carrier 近域威胁半径内，同时仍满足该翼队当前 HOLD 站位的警戒半径约束，避免和既有 HOLD 清理逻辑打架。
3. 在 `updateCombat(dt:)` 的 `entity.attackTarget == nil` 分支中，优先尝试该 helper；没有合适目标时回落到原 `nearestTarget(for:maxDistance:)`。
4. 目标排序以“靠近被守护 Carrier”为主，而不是靠近翼队自身。这样 guard wing 会优先处理靠近航母的已知威胁。
5. 可选但建议：Fighter 对 air 目标、Helicopter 对 submarine / naval 目标可有轻量优先级；前提是仍通过 `canAttack` 和迷雾检查。
6. 不改变已有 `attackTarget` 的清理语义；如果目标离开 HOLD 半径、变未知、死亡或不可攻击，现有逻辑仍应清理。

## 7. 实现步骤

建议按以下小步实现：

1. 增加私有常量，例如 `carrierGuardThreatRadius`，数值应小于 `highValueNavalEscortRadius`，并与 `holdEngagementRadius` 兼容。
2. 增加 `carrierGuardAnchor(for:) -> GameEntity?` 或等价 helper，复用 v4.36 的 anchor 有效性判断，避免在目标优先 helper 里重复散落条件。
3. 增加 `carrierGuardPriorityTarget(for:) -> GameEntity?`：
   - 先确认翼队和 Carrier anchor 有效。
   - 从 `entities.values` 过滤合法敌方目标。
   - 必须检查 `wing.kind.canAttack(target.kind)`。
   - 必须检查 `isKnownToFaction(target, observer: wing.faction)`。
   - 必须检查 target 距离 Carrier 不超过 `carrierGuardThreatRadius + target.kind.footprint * 0.35`。
   - 必须检查 target 距离 wing 当前 `holdPosition` 不超过 `holdEngagementRadius + target.kind.footprint * 0.35`，以保持与既有 HOLD 清理一致。
   - 排序时优先靠近 Carrier 的目标；若加入类型优先级，仍要保持简单透明。
4. 修改 `updateCombat(dt:)`：
   - 在 `entity.attackTarget == nil` 时先计算原 acquisitionRange。
   - 如果是 HOLD 状态且有 guard anchor，可先尝试 `carrierGuardPriorityTarget(for:)`。
   - helper 返回 nil 时保持原 `nearestTarget(for:maxDistance:)` 行为。
5. 不改 `nearestTarget(for:maxDistance:)` 的既有语义，除非为了调用新 helper 做最小整理。
6. 更新 `README.md`：只写当前事实，即 Carrier HOLD guard wing 在保持站位时会优先处理 Carrier 近域内已知可攻击威胁；不要写完整 CAP / patrol / intercept。
7. 更新 `md/flow/flow.md`：同步移动与战斗、HUD / Carrier 段落，说明该优先目标仍复用 `canAttack`、迷雾已知和 HOLD 半径边界。
8. 必要时更新 `md/flow/flowchart.md` 的命令 / 战斗节点文字。
