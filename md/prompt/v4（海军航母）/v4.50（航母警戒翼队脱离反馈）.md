# v4.50（航母警戒翼队脱离反馈）

## 版本号

v4.50

## 版本分配依据

- 当前 `md/prompt/v4（海军航母）/` 已存在到 `v4.49（航母警戒翼队组成提示）.md`，本轮顺延为 v4.50。
- v4.34-v4.49 已建立 Carrier HOLD guard wing 的 anchor、站位、近域威胁优先、HUD、范围圈、AI 使用和组成提示。
- 当前 README / flow 已说明 bound HEL/JET 收到新移动、AMOV、攻击或 ordinary HOLD 后会脱离 guard，但玩家成功下达这些命令时没有明确短反馈。
- 本轮只补玩家命令成功反馈，不进入 CAP、巡逻、截击、新命令或 guard release 状态机。

## 背景

当前代码中 `clearCarrierGuardAnchor(for:)` 会清理 Helicopter / Fighter 的 `guardAnchorCarrierID`，但该 helper 同时被玩家命令、AI wave 接管、内部 anchor 失效维护等路径复用。如果直接在 helper 内显示消息，会让 AI 或内部状态维护产生玩家可见噪声。

适合的最小实现是在玩家命令入口清理前统计本次实际会脱离有效 Carrier guard anchor 的玩家 HEL/JET 数量，然后把现有成功消息追加短后缀，例如 `CV guard released 2.`。

## 目标

玩家对已被有效 Carrier guard anchor 绑定的 HEL/JET 成功下达以下命令时，顶部消息追加短反馈：

- 普通移动：`Move order issued. CV guard released n.` 或现有编队移动文案后追加同一后缀。
- AMOV：`Attack move issued. CV guard released n.` 或现有混合编队 AMOV 文案后追加同一后缀。
- 直接攻击可见敌人：`Attack order: Target. CV guard released n.`。
- ordinary HOLD：对 bound HEL/JET 自身下达普通 HOLD 且不是同次 Carrier HOLD 重新绑定时，`Hold position: ... CV guard released n.`。

反馈要求：

- 只统计本次命令实际处理的玩家 Helicopter / Fighter。
- 只统计清理前 `carrierGuardAnchor(for:)` 仍有效的 wing；不要直接信任裸 `guardAnchorCarrierID`。
- 成功命令才提示；无单位、不能攻击、无效目标等拒绝路径不显示 release 后缀。
- 多个 wing 绑定同一 Carrier 或多个 Carrier 都按 wing 数统计。
- 后缀尽量短，不覆盖原有命令消息。
- 本轮只改玩家反馈，不改变 guard anchor 清理条件、station、contact、combat、AI、迷雾、生产、支援、任务或胜负。

## 非目标

- 不新增 CAP、combat air patrol、巡逻、截击、拦截、新按钮、新 HUD action 或新玩家命令。
- 不新增持久 release 状态、时间戳、选择面板 transient 行或地图特效。
- 不使用 `showDeniedMarker` 表达脱离 guard；这些命令是成功命令，不是拒绝。
- 不让 `clearCarrierGuardAnchor(for:)` 直接调用 `showMessage(...)`。
- 不改变 `assignCarrierGuardWing(for:)`、`updateCarrierGuardStation(for:)`、`carrierGuardAnchor(for:)`、`isCarrierGuardContact(...)`、`carrierGuardPriorityTarget(for:)`。
- 不改变 Red AI guard wing、routine attack-move、低血撤退、占点 reservation 或 AI 侦察边界。
- 不新增外部依赖、素材、测试 target 或大体积 artifact。

## 当前架构依据

关键源码边界在 `DesertFrontline/GameScene.swift`：

- `handleWorldTap(at:tapCount:)`：直接攻击可见敌人的玩家入口，当前对实际可攻击单位清理 `holdPosition`、`clearCarrierGuardAnchor(for:)` 并设置 `attackTarget`。
- `issueAttackMoveOrder(to:units:)`：过滤可战斗单位后调用 `issueFormationMove(... attackMove: true)`。
- `issueMoveOrder(to:units:)` / `issueFormationMove(...)`：普通移动和 AMOV 共用的最终执行入口，当前在每个实际移动单位上清理 hold / guard / target 并设置 destination。
- `issueHoldPositionOrder(units:)`：ordinary HOLD 和 Carrier HOLD 共用入口，先对移动单位清理旧 anchor，再对本次 HOLD 中的 Carrier 调用 `assignCarrierGuardWing(for:)`。
- `carrierGuardAnchor(for:)`：有效 guard anchor 的权威判断，应复用它统计 release 数。
- `showMessage(...)`：顶部短消息出口。

推荐实现：

1. 新增只读 helper：
   - `carrierGuardReleaseCount(for units: [GameEntity]) -> Int`
   - 过滤玩家、存活、Helicopter / Fighter，并要求 `carrierGuardAnchor(for:) != nil`。
2. 新增字符串 helper：
   - `carrierGuardReleaseSuffix(count: Int) -> String`
   - `count > 0` 时返回 `" CV guard released \(count)."`，否则返回空字符串。
3. `issueFormationMove(...)`：在循环清理前对 `mobileUnits` 统计 releaseCount；`showFeedback` 文案追加 suffix。AI 路径 `showFeedback=false` 时不显示消息，即使统计存在也不外显。
4. 直接攻击：只对本次实际可攻击并会被 assigned 的玩家单位统计。可以在循环前先构造 `attackers = selected.filter { $0.kind.canAttack(tapped.kind) }`，对 attackers 统计 releaseCount，再执行原有赋值；成功消息追加 suffix。
5. `issueHoldPositionOrder(...)`：
   - 清理前记录本次移动单位中有效 bound wing 的 id 集合。
   - 原有 HOLD 与可能的 Carrier guard 重新绑定逻辑执行后，统计这些 id 中最终 `carrierGuardAnchor(for:) == nil` 的玩家 HEL/JET 数量。
   - 这样选中 Carrier + 其附近 wing 同时 HOLD 并被重新绑定时不会误报 release。
   - 原 `Guard wing n.` 后缀保留，release 后缀追加在消息末尾。
6. 不改 `clearCarrierGuardAnchor(for:)` 本身。

## 验收标准

Agent C 应按以下标准验收：

- 玩家对 bound HEL/JET 成功普通移动时，原成功消息后追加 `CV guard released n.`。
- 玩家对 bound HEL/JET 成功 AMOV 时，原成功消息后追加同一 release 后缀。
- 玩家对 bound HEL/JET 成功直接攻击可见且可攻击目标时，原攻击成功消息后追加同一 release 后缀。
- 玩家对 bound HEL/JET 下达 ordinary HOLD 且最终未重新绑定到有效 Carrier 时，HOLD 成功消息追加 release 后缀。
- 同时 HOLD Carrier 并重新绑定 wing 时，不应把最终仍有有效 anchor 的 wing 误报为 released。
- 无效命令、不能攻击、无移动单位、AI 命令、内部 anchor 失效清理、Red AI routine wave 不产生玩家 release 消息。
- `clearCarrierGuardAnchor(for:)` 未直接显示消息。
- `assignCarrierGuardWing(for:)`、`carrierGuardAnchor(for:)`、`isCarrierGuardContact(...)`、`carrierGuardPriorityTarget(for:)`、AI、combat、fog、sonar、production、support、mission、victory 行为未改变。
- README、flow、flowchart 与实际行为一致，且没有夸大为 CAP / 巡逻 / 截击。
- 最新 `origin/main` GitHub Actions run 成功，artifact 对应本轮 commit。

## 风险和禁止项

风险：

- 把消息放进 `clearCarrierGuardAnchor(for:)` 会导致 AI 或内部维护路径产生噪声，必须避免。
- HOLD 分支容易把同次 Carrier HOLD 重新绑定的 wing 误报为 released，必须按最终有效 anchor 再统计。
- 直接攻击必须只统计实际能攻击并被赋予 target 的单位，避免混合编队误报。

禁止项：

- 禁止扩大范围做完整 CAP、巡逻、截击、新按钮、新状态机或新玩家 UI。
