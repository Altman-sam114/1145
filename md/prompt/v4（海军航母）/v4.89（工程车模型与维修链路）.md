# v4.89 / 工程车模型与维修链路

## 版本号与分配依据

版本号为 `v4.89`。v4.86-v4.88 已依次完成 Tank、Artillery、Humvee 与沙地行进表现；本轮继续沿百度百科 lemma id `4042982` 和 Noble Master Games 官方 press kit 所确认的《沙漠风暴 Desert Stormfront》单位表，聚焦 Mechanic 与自动维修可读性。

## 参考依据

- 百度百科词条明确指向 Noble Master Games 的现代中东沙漠 RTS，不与其他同名作品混用。
- Noble Master 官方 press kit `https://www.noblemaster.com/press/sheet.php?p=desert_stormfront` 明确列出 Humvees、Tanks、Artillery、Mechanics、Helicopters、Planes、Ships、Submarines，并提供官方 trailer `https://www.youtube.com/watch?v=7oZC6DpIUhQ` 和原始截图。
- 官方截图以清晰俯视轮廓、阵营旗 / 血条、低遮挡反馈和紧凑触屏命令区维持移动端辨识；本项目借鉴这些原则，不复制原素材、模型或 UI 布局。

## 背景

当前 Mechanic 只有一个矩形车体和斜置色条，自动维修只偶发生成单点 spark。玩家能从选择面板看到范围和受损目标数，但战场上无法稳定辨认工程车，也不容易确认哪辆单位正在被哪台 Mechanic 维修。现有 `updateRepair(dt:)` 已是玩家与 AI 共享的自动维修事实链路，适合在不改变数值和目标选择的前提下补足模型与确定性视觉反馈。

## 目标

1. 在现有 footprint 内细化 Blue / Red Mechanic 工程车：四轮、底盘、后部工具舱 / 槽位、驾驶舱、阵营化风挡、警示灯、折叠吊臂 / 吊钩和工具色标。
2. 给 Mechanic 预创建 `mechanicRepairEffectNode`，包含阵营化外层维修束、白色核心束和带十字的目标脉冲。
3. `updateRepair(dt:)` 保持原最近受损友军目标、95 范围和每秒 22 HP 维修量，只在实际维修时更新预创建节点路径；没有合法目标时隐藏。
4. 玩家 Mechanic 的维修反馈始终可见；敌方反馈只有 Mechanic 和目标都通过玩家现有迷雾认知时显示，不能泄露未知目标位置。
5. land CI capture 加入双方 Mechanic、受损 Tank、双向维修束和玩家 Mechanic 选择范围圈，同时保留 Humvee / Tank / Artillery、扬尘、炮口、炮线和爆炸证据。

## 非目标

- 不修改 Mechanic HP、速度、价格、建造时间、占点速率、维修范围、维修量或自动目标优先级。
- 不新增手动维修命令、维修资源、维修冷却、无人机、拖车、科技树或新的 AI 状态机。
- 不修改 Field Repair 支援、经济、寻路、攻击、任务、胜负、HUD 页面或迷雾集合。
- 不新增外部依赖或素材，不运行本地 Xcode build、simulator 或玩法探针。
- 不修改或提交用户已有的 `project.pbxproj` DEVELOPMENT_TEAM 改动。

## 当前架构依据

- `GameEntity` 持有所有实体视觉子节点；新增维修节点必须属于 Mechanic 实体子树，随实体迷雾和销毁。
- `updateRepair(dt:)` 在统一帧循环内为玩家与 AI Mechanic 选择最近范围内受损友军并恢复 HP。
- `isKnownToFaction(..., observer: .player)` 是敌方视觉反馈是否可向玩家显示的边界。
- `repairCoverageNode` 已提供选中玩家 Mechanic 的范围圈；不得另建重复范围系统。
- `DESERT_CI_CAMERA_FOCUS=land` 只在 capture mode 编排确定性截图。

## 实现步骤

1. 为 `GameEntity` 增加维修 effect / beam / core / target 节点。
2. `configureMechanicRepairEffectNode(...)` 只为 Mechanic 预建双层束和十字目标脉冲，默认隐藏。
3. `updateMechanicRepairEffect(...)` 将世界目标转换为考虑 `xScale` 的实体局部坐标，截短到目标 footprint 边缘，并只更新 path / position / alpha / scale。
4. `updateRepair(dt:)` 每帧先隐藏所有 Mechanic 维修节点，找到原合法目标并完成原数值恢复后再显示对应链路；移除该链路原有随机单点 spark 调用，其他 construction / support spark 调用保持不变。
5. 重构 `addLandUnitBody(...)` 的 Mechanic shape 组合，保持 gameplay footprint 和朝向语义。
6. land capture 布置双方 Mechanic 和受损 Tank，刷新迷雾后冻结双向维修证据；玩家 Mechanic 加入选择以显示既有范围圈。
