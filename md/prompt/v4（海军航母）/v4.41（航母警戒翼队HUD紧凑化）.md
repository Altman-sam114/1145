# v4.41（航母警戒翼队HUD紧凑化）Agent B 实现提示词

## 1. 版本号

v4.41（航母警戒翼队HUD紧凑化）

## 2. 版本分配依据

- 当前 `md/prompt/v4（海军航母）/` 已存在 `v4.40（航母警戒翼队优先接触提示）.md`，本轮延续 v4 海军航母阶段并递增为 v4.41。
- v4.38 到 v4.40 已逐步完成 Carrier HOLD guard wing 的 `Ctc n`、Air / Sea / Sub / Ground / Mix 类型摘要和 `Tgt XXX` 优先接触短码。
- 当前单选 Carrier HOLD 时 HUD 行较长，例如 `Guard wing x/2 Need n  Ctc m Air Tgt XXX`，容易挤压选择面板。v4.41 只做只读 HUD 文案紧凑化，不改变任何玩法规则。

## 3. 背景

当前 `DesertFrontline/GameScene.swift` 中：

- `singleSelectionInfo(for:)` 在单选 Carrier 时，把 `carrierDeckQueueAndRallyLine(for:)` 和 `carrierAirWingLine(for:)` 拼到同一行。
- `carrierAirWingLine(for:)` 在 Carrier 未 HOLD 时显示 `Wing x/2 OK` 或 `Wing x/2 Need n`。
- `carrierAirWingLine(for:)` 在 Carrier HOLD 时改用绑定 guard wing 口径，显示 `Guard wing x/2 OK/Need n`，并追加 `Ctc n`、非零类型摘要和 `Tgt XXX`。
- `carrierGuardContactSummary(for:guardWing:)` 已返回 `(count: Int, type: String?, targetCode: String?)`，并复用 `isCarrierGuardContact(...)`、`carrierGuardTargetPriority(for:target:)`、Carrier 距离和 entity id 排序。
- `highValueNavalEscortLine(for:)` 和多选高价值海军护航摘要是另一套 escort HUD 口径，本轮不得改动。

## 4. 本轮目标

只实现一个小范围 HUD 文案增量：

1. 单选 Carrier 未 HOLD 时，保持现有 readiness 语义，可继续显示 `Wing x/2 OK` / `Wing x/2 Need n`，除非 Agent B 判断需要同步做同风格轻微缩短，但不得改变含义。
2. 单选 Carrier HOLD 时，把 guard wing HUD 行压缩为更短、更稳定的只读格式。
3. 具体格式由 Agent B 结合现有代码选择，但应明显短于当前 `Guard wing ...  Ctc ...` 文案，例如可采用类似：
   - `GW x/2 OK C0`
   - `GW x/2 OK C2 Air Tgt JET`
   - `GW x/2 Need 1 C0`
   - `GW x/2 Need 1 C2 Mix Tgt SUB`
4. 如果继续使用 `Ctc` 而不是 `C`，也可以，但必须整体明显紧凑，避免长 `Guard wing` 文案继续占用面板宽度。
5. 必须保留 `Ctc 0` 不显示类型 / 目标的语义，或使用等价紧凑语义：当 contact count 为 0 时，只显示 0 接触，不追加 Air / Sea / Sub / Ground / Mix，也不追加 `Tgt XXX`。
6. contact count 非零时，仍可显示类型摘要和 `Tgt XXX`，但必须来自现有 `carrierGuardContactSummary(for:guardWing:)` 的同一只读结果。
7. 本轮只改变显示字符串，不改变 guard wing 绑定、站位、接触判定、目标排序、命令状态、迷雾、潜艇侦测、AI、移动、战斗、生产或 `SKRM`。

- 不改 `carrierGuardPriorityTarget(for:)` 的战斗目标获取。
- 不改 `boundCarrierGuardWing(for:)`、`guardAnchorCarrierID`、guard wing 绑定或站位保持。
- 不改 `nearbyCarrierAirWing(for:)`、`highValueNavalEscortLine(for:)`、`nearbyNavalEscorts(for:)` 或多选高价值海军护航摘要。
- 不改战争迷雾、`isKnownToFaction(...)`、`visibleTiles`、`exploredTiles`、`supportRevealTiles`。
- 不改潜艇 `revealedUntil`、声呐检测、ASW 命中反馈或支援技能暴露逻辑。
- 不改 AI、移动、战斗、生产、支援技能、任务、胜负或 `SKRM` 重置。
- 不新增按钮、pending 模式、命令、状态机、依赖、素材、Xcode target、Xcode 工程配置或 workflow。

## 7. 实现步骤

建议按以下小步实现：

1. 从最新 `origin/main` 同步后，先阅读 `carrierAirWingLine(for:)` 当前实现，确认 HOLD 和非 HOLD 两条文案分支。
2. 只在 `carrierAirWingLine(for:)` 或同文件局部小 helper 内改 guard wing 文案组装。

4. 保持 `missing`、`count`、`requirement` 的计算不变。
5. 对 Carrier HOLD 分支使用紧凑 label，例如 `GW`，并压缩 contact 段，例如 `C0` / `C2` 或同等短格式。
6. 当 `contactSummary.count == 0` 时，不拼接 `contactSummary.type`，不拼接 `Tgt`。
7. 当 `contactSummary.count > 0` 时，可以继续拼接类型摘要和 `Tgt XXX`，但只能使用 `contactSummary.type` 和 `contactSummary.targetCode`。
8. 不要为了压缩文案去改 contact summary 的数据结构、排序或筛选；除非 Swift 编译需要极小的局部命名调整。
9. 阅读实际 diff，确认没有修改 `isCarrierGuardContact(...)`、`carrierGuardTargetPriority(...)`、AI、移动、战斗、生产、迷雾或 `SKRM`。
