# v4.66 / 单机警戒目标状态

## 版本号

v4.66

## 版本分配依据

v4.65 已完成 Carrier HOLD 后 guard wing 缺口反馈，并通过 `origin/main` 最新云端 artifact 验收。v4.65 遗留事项建议继续细化 Carrier guard wing 命令 UI。本轮选择单选 bound HEL/JET 的只读状态行作为最小可验证增量：当该飞机当前存在合法 Carrier guard contact 时，在 `CV GUARD` 状态行显示优先目标短码和类型，版本递增为 v4.66。

## 背景

当前 Carrier 单选和多选 HOLD 状态行已经能显示 `Tgt XXX`、Air / Sea / Sub / Ground / Mix 和 `Eng n`，但玩家单选被 Carrier guard anchor 绑定的 HEL/JET 时，只能看到 `CV GUARD Dn  Guard ...`。这说明它属于哪个 Carrier 近域 guard wing，却不能说明它当前是否正在处理某个合法 guard contact。本轮补上只读目标摘要，帮助玩家理解舰载机护航职责。

## 目标

- 单选玩家 bound Carrier guard HEL/JET 时，如果当前存在合法 Carrier guard contact，在状态行追加 `Tgt <shortCode> <type>`。
- 目标选择必须复用现有合法性：
  - `carrierGuardAnchor(for:)`
  - `wing.holdPosition`
  - `carrierGuardPriorityTarget(for:)`
  - `carrierGuardContactType(for:)`
- 没有合法 contact 时保持现有 `CV GUARD Dn  Guard <radius>` 文案。
- 该状态行只读展示，不写入 `attackTarget`、命令、迷雾或 AI 状态。
- 同步 README 和 `md/flow/flow.md` 当前事实说明。

## 非目标

- 不新增 CAP、巡逻、截击、护航命令或状态机。
- 不改变 `isCarrierGuardContact(...)`、`carrierGuardTargetPriority(...)`、`carrierGuardPriorityTarget(for:)` 的合法性和排序。
- 不改变攻击目标获取、`fire(attacker:target:)`、HOLD 回位、guard station、release feedback、Carrier 单选 / 多选摘要、AI、生产、支援、任务或胜负。
- 不更新 `md/test/test.md`，本轮不改变测试流程。

## 当前架构依据

- `carrierGuardWingStatusLine(for:)` 是单选玩家 bound HEL/JET 的状态行入口。
- `carrierGuardAnchor(for:)` 已确保 wing 当前仍绑定存活、同阵营、正在 HOLD 的 Carrier。
- `carrierGuardPriorityTarget(for:)` 已在战斗目标获取中复用 `isCarrierGuardContact(...)`，并按现有优先级、距 Carrier 距离和 id 稳定打平。
- `carrierGuardContactType(for:)` 已统一生成 Air / Sea / Sub / Ground 类型。
- 选择信息是 HUD 只读文案，不能改变实体字段或迷雾集合。

## 实现步骤

1. 同步 main：
   - `git fetch origin`
   - `git switch main`
   - `git pull --ff-only origin main`
   - 确认工作区没有无关改动。
2. 修改 `DesertFrontline/GameScene.swift`：
   - 在 `carrierGuardWingStatusLine(for:)` 中保留现有 guard 条件和距离计算。
   - 调用 `carrierGuardPriorityTarget(for: wing)` 获取当前合法优先 contact。
   - 如果有目标，生成 `targetSuffix = " Tgt \(target.kind.shortCode) \(carrierGuardContactType(for: target))"`。
   - 返回 `CV GUARD Dn\(targetSuffix)  Guard <radius>`；没有目标时 `targetSuffix` 为空，保持原有文案。
   - 不写入 `wing.attackTarget` 或任何实体 / 迷雾字段。
3. 修改 `README.md`：
   - 在 Carrier guard / selection panel 描述中补充：单选 bound HEL/JET 若有合法 guard contact，会显示 `Tgt XXX Air/Sea/Sub/Ground`。
4. 修改 `md/flow/flow.md`：
   - 在 `GameEntity` 或 HUD 选择信息段补充：单选 bound HEL/JET 的 `CV GUARD` 状态行只读显示当前合法优先 contact 的短码和类型。
5. 不需要修改 `md/flow/flowchart.md`，除非实际图中需要同步该短文案。

## 验收标准

- 单选未绑定 Carrier guard anchor 的 HEL/JET 不显示该状态行。
- 单选已绑定有效 Carrier guard anchor 的玩家 HEL/JET 且无合法 contact 时，仍显示现有 `CV GUARD Dn  Guard <radius>`。
- 单选已绑定有效 Carrier guard anchor 的玩家 HEL/JET 且存在合法 contact 时，显示 `CV GUARD Dn Tgt XXX Air/Sea/Sub/Ground  Guard <radius>`。
- 目标短码来自 `target.kind.shortCode`，类型来自 `carrierGuardContactType(for:)`。
- 新文案不得显示迷雾中未知目标；合法性必须继续由 `isCarrierGuardContact(...)` 和 `isKnownToFaction(...)` 间接约束。
- Carrier 单选 / 多选 `GW` 摘要、`Eng n`、release feedback、guard ring、AI、战斗、迷雾和生产语义不变。
- README 和 flow 与实现一致，不夸大能力。
- 最新 `origin/main` commit 的 GitHub Actions run 成功，artifact manifest、JUnit、xcodebuild.log、failure summary 均对应同一 run。
