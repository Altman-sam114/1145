# v4.48（AI航母警戒翼队）

## 版本号

v4.48

## 版本分配依据

- 当前 `md/prompt/v4（海军航母）/` 已存在到 `v4.47（选中航母显示警戒锚点范围圈）.md`，本轮顺延为 v4.48。
- v4.34-v4.47 已为玩家 Carrier HOLD guard wing 建立了 anchor、站位保持、近域威胁优先、HUD 和选择派生可视化。
- 当前 `carrierGuardAnchor(for:)`、`updateCarrierGuardStation(for:)`、`carrierGuardPriorityTarget(for:)` 和 `updateCombat(dt:)` 的核心 guard wing 链路不依赖玩家阵营，但 `assignCarrierGuardWing(for:)` 只在玩家 HUD `HOLD` 命令里使用。
- 本轮把已有 guard wing 机制有限地交给 Red AI 的空闲 Carrier 使用，作为航母护航行为的最小实装，不进入完整 CAP / 巡逻 / 截击系统。

## 背景

现有 Red AI 已经能：

- 生产混合陆海空部队，包含 Carrier、Helicopter 和 Fighter。
- 用护航门槛避免 Battleship / Carrier 孤立出击。
- 在 routine attack-move 波次中混入空军和海军支援。

现有 Carrier guard wing 机制已经能：

- 在 Carrier 仍 HOLD 时，让绑定 HEL/JET 维护 Carrier 附近稳定站位。
- 让绑定翼队优先处理 Carrier 近域内已知、可攻击且仍在 wing HOLD 警戒半径内的威胁。
- 在 `carrierGuardAnchor(for:)` 中要求 Carrier 存活、同阵营并仍有 `holdPosition`。
- 普通移动、AMOV、直接攻击、普通 HOLD 等命令会清理 HEL/JET 的旧 `guardAnchorCarrierID`。

## 目标

让 Red AI 在指挥周期内为满足条件的空闲 Carrier 绑定附近空闲 HEL/JET guard wing：

- Red Carrier 必须存活、operational、非结构、当前没有 `attackTarget`、没有 `attackMoveDestination`、没有普通 `destination`，且不是撤退中的单位。
- Red Carrier 附近存在同阵营、存活、operational、非结构、Helicopter / Fighter，且该 wing 当前没有 `attackTarget`、没有 `attackMoveDestination`、没有普通 `destination`，也不是占点 reservation 或撤退单位。
- AI 绑定时设置 Carrier 自身 `holdPosition = carrier.node.position`，让 `carrierGuardAnchor(for:)` 的有效条件成立。
- 绑定 wing 时复用现有 guard anchor 字段和站位链路：设置 wing `holdPosition`、`guardAnchorCarrierID`、清理 wing 的攻击 / attack-move / destination / path。
- 每个 AI 指挥周期只给每艘符合条件的 Carrier 补齐到最多 2 架 bound HEL/JET；多个 Carrier 不重复抢同一架 wing。
- 已经绑定到该 Carrier 且仍有效的 wing 应计入数量，不重复重写。
- 若 Carrier 后续进入 attack-move 波次、直接目标、防守响应或撤退路径，既有 `issueFormationMove(...)` / `retreatEnemyAssaultUnit(...)` 等清理逻辑应自然取消其普通 HOLD 或影响 guard anchor 有效性；本轮不新增复杂 release 状态机。
- 不给玩家显示新 HUD，不新增 AI 专属作弊侦察；guard wing 交战仍必须通过 `isKnownToFaction(...)` 和现有 `canAttack(...)`。

## 非目标

- 不新增 CAP、combat air patrol、巡逻路线、截击命令、新按钮、新 HUD action 或玩家新操作。
- 不改变玩家 Carrier HOLD 行为和 UI 文案。
- 不改变 Carrier / HEL / JET 数值、伤害、冷却、视野、生产成本或建造时间。
- 不改变 `carrierGuardAnchor(for:)`、`carrierGuardPriorityTarget(for:)`、`isCarrierGuardContact(...)` 的合法性口径。
- 不改变 Red routine attack-move 护航门槛、mixed provisional wave 规则、低血撤退或老兵保护规则。
- 不让 AI 读取玩家隐藏单位坐标；guard wing 目标仍必须 Red 已知。
- 不新增外部依赖、素材、测试 target 或大体积 artifact。

## 当前架构依据

关键源码边界在 `DesertFrontline/GameScene.swift`：

- `updateAI(dt:)`：Red AI 指挥周期入口，已执行生产、防空 / 反潜补单、占点、支援和 routine attack-move。
- `nearbyCarrierAirWing(for:)`：按同阵营、存活、operational、HEL/JET 和 `highValueNavalEscortRadius` 统计 Carrier 附近空军，当前可复用但需要对候选 wing 增加 idle / 非 reservation 过滤。
- `boundCarrierGuardWing(for:)`：统计仍 HOLD 且 `guardAnchorCarrierID` 等于 Carrier id 的 wing。
- `assignCarrierGuardWing(for:)`：当前只允许玩家 Carrier 使用；本轮可新增 AI 专用 helper，或把它参数化为按 carrier faction 工作，但不得改变玩家 HUD 行为。
- `updateMovement(dt:)` 每帧调用 `updateCarrierGuardStation(for:)`，因此 Red wing 一旦绑定也会走同一站位保持。
- `updateCombat(dt:)` 在目标获取阶段调用 `carrierGuardPriorityTarget(for:) ?? nearestTarget(...)`，因此 Red wing 一旦有效绑定，也会走同一近域威胁优先。

推荐实现：

1. 新增 `updateEnemyCarrierGuardWings()`，在 `updateAI(dt:)` 中 `commandEnemyAttackers(enemyUnits)` 之前调用，确保空闲 Carrier 能先建立 guard wing，但仍会被后续 routine wave 正常接管。
2. 新增 `isAvailableEnemyCarrierGuardWing(_:)` 或在 helper 内局部过滤：Red HEL/JET、存活、operational、非结构、无 attack target、无 attack-move、无 destination、非占点 reservation、非撤退。
3. 新增 `assignCarrierGuardWing(for carrier: GameEntity, using assignedWingIDs: inout Set<Int>, maxWing: Int = 2) -> Int` 这类 AI 专用 helper；不要破坏玩家 `assignCarrierGuardWing(for carriers:)` 的返回和消息语义。
4. 对每个符合条件的 Red Carrier：
   - 若 `boundCarrierGuardWing(for:)` 已有 2 架或以上，跳过。
   - 先设置 `carrier.holdPosition`，清理 Carrier 当前普通攻击 / attack-move / destination / path 只在它已被判定为空闲时执行。
   - 从 `nearbyCarrierAirWing(for:)` 里选取未被本周期其他 Carrier 使用、符合 idle 条件、且不是已经绑定到其他有效 Carrier 的 HEL/JET，补齐到 2 架。
   - 绑定 wing 的字段设置应与玩家 guard wing 一致。
5. 不新增玩家视觉 cue。Red Carrier / wing 的节点仍按 fog / `isKnownToFaction` 显示。

## 验收标准

Agent C 应按以下标准验收：

- Red 空闲 Carrier 在 AI 指挥周期内可把附近空闲 Red HEL/JET 绑定为 guard wing，最多补齐 2 架。
- Red Carrier 必须通过 `holdPosition` 成为有效 anchor；绑定 wing 的 `guardAnchorCarrierID` 指向该 Carrier。
- Red bound wing 后续走既有 `updateCarrierGuardStation(for:)` 与 `carrierGuardPriorityTarget(for:)`，只攻击 Red 已知且自身可攻击的近域目标。
- 正在 attack-move、已有 target、正在普通移动、占点 reservation、撤退中的 Red HEL/JET 不应被抢去当 guard wing。
- 已被其他有效 Carrier anchor 绑定的 wing 不应在同一周期被另一个 Carrier 抢走。
- 玩家 Carrier HOLD、HUD、selection ring、`CV GUARD` 文案、护航摘要和玩家命令不回退。
- Red routine attack-move 护航门槛、混编波次、低血撤退和老兵保护规则不被绕过。
- README、flow、flowchart 与实际行为一致，且没有夸大为 CAP / 巡逻 / 截击。
- 最新 `origin/main` GitHub Actions run 成功，artifact 对应本轮 commit。

## 风险和禁止项

风险：

- 如果 AI guard wing 过早 HOLD 住 Carrier，可能减少其加入 routine attack-move 的机会；必须只处理空闲 Carrier，并允许后续 `commandEnemyAttackers` 在条件满足时接管。
- `holdPosition` 被 Red 低血撤退也使用，AI guard helper 必须避开撤退单位。
- 绑定 wing 时不能抢占占点 reservation 或正在作战 / 移动的空军。

禁止项：

- 禁止扩大范围做完整 CAP、巡逻、截击、新按钮、新状态机或新玩家 UI。
- 禁止修改 guard contact predicate、目标排序、迷雾、潜艇侦测、单位数值、AI 生产规则或 routine wave 规则。
- 禁止新增外部依赖、素材或测试 target。
- 禁止无关重构 `GameScene.swift`。
- 禁止回滚或覆盖用户 / 其他 Agent 的无关改动。
- 禁止伪造本地或云端验证。
