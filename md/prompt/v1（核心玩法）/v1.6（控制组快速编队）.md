# v1.6（控制组快速编队）

## 版本号

v1.6

## 版本分配依据

- 人工明确指定本轮版本为 v1.6。
- v1.5“非法命令反馈”已由 Agent X 确认闭环通过，`update_log.md` 已记录 v1.5；当前本地 `HEAD` / `origin/main` 应为 `fa703b8fb6fa8f06580d8ff923e275d19db86f4f`。
- `md/flow/flow.md` 的未来扩展点仍包含“操作体验：控制编队”，本轮只实现两个触控 HUD 控制组按钮，是 v1.5 后的小型操作体验增量。

## 背景

`Desert Frontline` 当前已经具备选择、双击同类选择、拖框、多单位移动、`HOLD`、`AMOV`、生产、集结点、支援技能、命令按钮 pending 高亮和非法地图命令反馈。玩家在移动端没有键盘快捷键，常用部队组合需要重复框选或点选；v1.6 要增加两个小型控制组按钮，让玩家可以通过触控 HUD 快速保存和召回编队。

本轮只保存实体 ID 集，不保存单位快照。召回时从当前 `entities` 中过滤仍存活、玩家阵营、非结构的机动单位，避免死亡、被清理或不再合法的 ID 残留影响选择。保存控制组不下达单位命令，不改变生产、经济、战斗、AI、迷雾、支援、老兵、任务或胜负规则。

## 目标

1. 在底部 HUD 增加两个控制组按钮，建议标题为 `G1` / `G2`。
2. 单击 `G1` / `G2`：
   - 清理 pending 建筑、支援、集结点、attack-move 和 construction preview。
   - 从对应组的 ID 集中召回仍存活、玩家阵营、非结构的机动单位。
   - 过滤掉已死亡、已从 `entities` 移除、非玩家阵营或结构实体的 ID，并把过滤后的 live IDs 写回控制组。
   - 如果过滤后有单位：更新 `selectedIDs`、调用 `refreshSelection()`、调用 `updateHUD()`，显示类似 `Group 1 selected: N units.` 的 HUD message。
   - 如果过滤后为空：清空该控制组 ID 集；不要改变当前选择；调用 `updateHUD()`，显示类似 `Group 1 is empty.` 的克制文案。
3. 双击 `G1` / `G2`：
   - 清理 pending 建筑、支援、集结点、attack-move 和 construction preview。
   - 读取当前 `selectedMobilePlayerUnits()`，保存其 live IDs 到对应组，覆盖旧组。
   - 如果当前没有机动玩家单位：显示类似 `Select mobile units before saving Group 1.` 的错误文案；不保存空组，不覆盖旧组，不改变当前选择。
   - 如果保存成功：不改变单位命令、不改变 selection 本身；调用 `updateHUD()`，显示类似 `Saved N units to Group 1.` 的 HUD message。
4. `SKRM` 重开必须清空两个控制组状态。
5. HUD subtitle 要短，适配现有按钮尺寸：
   - 空组可显示 `empty`。
   - 非空组可显示 `N units`。
   - 也可以在空组显示 `save dbl`，但文字必须短，不能溢出按钮。
6. README Controls、`md/flow/flow.md` 输入流程必须同步；如流程图需要表达控制组 HUD 分支，也同步 `md/flow/flowchart.md`。
7. 不更新 `update_log.md`。正式 v1.6 记录由 Agent C 在最新 `origin/main` artifact 验收通过后追加。

## 非目标

- 不新增键盘快捷键、地图 ping、镜头跳转、自动编队、队伍图标浮层、声音、设置项或外部素材。
- 不自动把新生产单位加入控制组。
- 不保存单位快照、位置、命令、血量、等级、生产来源或阵营快照；只保存 ID 集。
- 不改变双击世界同类选择、拖框、`ARMY`、`HOLD`、`AMOV`、`RLY`、`BASE`、支援技能、生产按钮、HQ focus、AI difficulty 或 `SKRM` 的既有语义，除本轮明确要求的 `SKRM` 清空控制组。
- 不改变生产、经济、战斗、AI、迷雾、支援技能、老兵 XP、任务阶段、胜负、潜艇隐身或声呐检测。
- 不新增外部依赖、图片素材、音频素材、XCTest target 或大体积测试数据。
- 不拆分 `GameScene.swift`，不做大规模重构，不做全文件格式化。
- 不修改 `GameView.swift`、`DesertFrontlineApp.swift` 或 Xcode project，除非编译必须且能说明原因。按当前架构判断本轮不需要修改它们。
- 不默认跑本机完整 Xcode build 或模拟器；除非人工明确要求，本轮本地只跑轻量检查，构建以 GitHub Actions artifact 为准。

关键源码事实：

- `GameScene.swift` 是本轮唯一需要修改的源码入口。`GameView.swift` 只用 `SpriteView` 承载固定尺寸 `GameScene`，`DesertFrontlineApp.swift` 只创建 SwiftUI window。
- `HudAction` 当前枚举底部按钮：`selectArmy`、`holdPosition`、`attackMove`、生产、`setRally`、`buildBase`、支援技能、`cycleAI`、`focusHQ`、`newSkirmish`。新增控制组必须同步 `title`、`buttonColor(for:)`、`subtitle(for:)`、`layoutHUD()` action 列表和 `handleHudAction`。
- `layoutHUD()` 每次重建 HUD，会清空 `hudButtonFrames`、`hudButtonSubtitleLabels`、`hudButtonShapes`，按 `actions` 数组创建按钮；加入两个按钮后要确认紧凑布局仍可容纳，subtitle 不溢出。
- `makeButton(action:frame:)` 使用固定标题和 subtitle 字号；本轮不要改成复杂自适应 UI，优先保证 `G1` / `G2` 和短 subtitle 可读。
- `subtitle(for:)` 已用于生产价格、支援冷却和短状态文案；控制组 subtitle 应在这里从控制组 ID 集动态计算。
- 当前 `touchesBegan(_:,with:)` 在 HUD 命中时直接调用 `handleHudAction(action)` 并 return；`tapCount` 只在 `touchesEnded` 的世界点击路径传给 `handleWorldTap(at:tapCount:)`。本轮如要支持 HUD 双击保存，必须把 `touch.tapCount` 传入 HUD action 处理，例如改成 `handleHudAction(action, tapCount: touch.tapCount)`，或用等价方式显式处理 HUD 双击。不要留下“自行决定交互”的空洞实现。
- `selectedMobilePlayerUnits()` 已返回当前 `selectedIDs` 中玩家阵营、存活、非结构实体，适合作为保存控制组的数据源。
- `refreshSelection()` 只根据 `selectedIDs` 更新选择框和 rally marker 可见性；召回组后必须调用。
- `updateHUD()` 会刷新 subtitle、selection info、message 之外的 HUD 状态；保存或召回控制组后必须调用，保证 `G1` / `G2` subtitle 更新。
- pending 模式字段包括 `pendingConstructionKind`、`pendingSupportPower`、`isSettingRallyPoint`、`isSettingAttackMove`；进入或使用控制组时必须清理这些字段，并调用 `clearConstructionPreview()`。
- `restartSkirmish()` 当前清空实体、选择、生产、pending、支援、任务、迷雾、效果层并重建地图；本轮必须在这里清空控制组状态。
- v1.4 按钮高亮由 pending 状态推导；控制组按钮不需要 armed 高亮，不应新增第二份 active 状态。
- v1.5 的 `showDeniedMarker(at:reason:)` 是世界点击非法反馈；HUD 双击无可保存单位没有明确世界目标，保持 HUD message 即可，不需要 denied marker。

### 2. 新增控制组状态与小 helper

在 `GameScene` 状态字段附近新增两个控制组 ID 集。推荐形式之一：

```swift
private var controlGroups: [Int: Set<Int>] = [1: [], 2: []]
```

也可以用两个字段：

```swift
private var controlGroup1IDs: Set<Int> = []
private var controlGroup2IDs: Set<Int> = []
```

要求：

- 保存的是 `GameEntity.id`，不是 `GameEntity` 引用。
- helper 必须统一过滤 live IDs，避免每个分支复制过滤规则。推荐新增：

```swift
private func liveControlGroupIDs(for group: Int) -> Set<Int>
private func controlGroupUnits(for group: Int) -> [GameEntity]
private func pruneControlGroup(_ group: Int)
```

等价命名可以，但过滤条件必须是：

- `entities[id]` 存在。
- `entity.isAlive == true`。
- `entity.faction == .player`。
- `!entity.kind.isStructure`。

不要额外按战争迷雾、当前屏幕可见性、单位 kind、domain 或 combat damage 过滤；控制组可以包含 Mechanic、Carrier、Submarine 等所有玩家机动单位。

### 3. 新增 HudAction case 与标题

在 `HudAction` 中新增两个 case，建议：

```swift
case controlGroup1
case controlGroup2
```

`title`：

- `.controlGroup1` -> `G1`
- `.controlGroup2` -> `G2`

`buttonColor(for:)`：

- 给控制组一个和生产 / 支援 / AI 区分开的稳重颜色即可，例如偏青灰或橄榄色。
- 不要引入会让整体 HUD 变成单一色系的大面积重配色。

`isHudActionArmed(_:)`：

- 控制组默认 false，不参与 pending 高亮。

### 4. 调整 layoutHUD() action 列表

把 `G1` / `G2` 加入底部 HUD `actions` 数组。

建议位置：

- 放在 `.selectArmy` 后、`.holdPosition` 前，便于玩家把它们作为选择工具使用：

```swift
let actions: [HudAction] = [
    .selectArmy, .controlGroup1, .controlGroup2, .holdPosition, .attackMove, ...
]
```

注意当前 `actions.count` 会从 20 增加到 22：

- 非 compact HUD 下 `buttonWidth = min(86, ...)`，宽度会收窄；确认 `G1` / `G2`、`SKRM`、`empty`、`N units` 等标题和 subtitle 仍短。
- compact HUD 下 `commandRowCounts(total:rows:)` 会自动分成两行；不要硬编码行数或破坏现有按钮布局。
- 不要把按钮做成新的浮动面板，保持当前底部 command bar 体系。

### 5. 控制组 subtitle

在 `subtitle(for:)` 中增加控制组文案。

推荐规则：

- 空组：`empty` 或 `save dbl`。更推荐 `empty`，因为双击保存规则会在 README Controls 说明。
- 非空组：先 prune / 计算 live count，再显示 `1 unit` 或 `N units`。如果担心 `1 unit` 宽度，统一显示 `N units` 也可以。

要求：

- subtitle 不能修改控制组状态，除非你明确把 prune 放在安全 helper 中且不会造成 HUD update 期间意外副作用。更稳妥的做法是在保存 / 召回 / 死亡清理后 prune，subtitle 只计算 live count。
- 文案必须短，不要超过按钮宽度。

### 6. HUD 双击处理必须显式传 tapCount

修改 HUD 命中路径。当前代码形态类似：

```swift
if let action = hudAction(at: uiPoint) {
    handleHudAction(action)
    touchStartScene = nil
    touchStartWorld = nil
    return
}
```

本轮必须改为显式把 `tapCount` 传入，例如：

```swift
if let action = hudAction(at: uiPoint) {
    handleHudAction(action, tapCount: touch.tapCount)
    touchStartScene = nil
    touchStartWorld = nil
    return
}
```

并把函数签名改成：

```swift
private func handleHudAction(_ action: HudAction, tapCount: Int = 1)
```

要求：

- 非控制组按钮忽略 `tapCount`，保持原有单击行为。
- 控制组按钮用 `tapCount >= 2` 判断保存，用 `tapCount < 2` 判断召回。
- 不要把 HUD 双击改到 `touchesEnded` 后再处理，避免大范围重写输入链路。
- 不要影响世界双击同类选择；`handleWorldTap(at:tapCount:)` 仍保留现有路径。

### 7. 抽出 pending 清理 helper

为避免控制组和多个 HUD action 重复清理，可以新增一个小 helper，例如：

```swift
private func clearPendingCommandModes(clearSelection: Bool = false) {
    clearConstructionPreview()
    pendingConstructionKind = nil
    pendingSupportPower = nil
    isSettingRallyPoint = false
    isSettingAttackMove = false
    if clearSelection {
        selectedIDs.removeAll()
        refreshSelection()
    }
}
```

也可以只写专用于控制组的 helper：

```swift
private func clearPendingCommandModesForControlGroup()
```

要求：

- 不要大规模重构所有 `handleHudAction` 分支；本轮只需要确保控制组进入 / 使用时清理 pending 和 preview。
- 如果选择复用 helper 到少数已有分支，必须保证行为完全等价，避免扩大 diff。
- 控制组召回或保存不应清空 selection，除非召回成功后用组内 IDs 替换 selection。

### 8. 实现保存与召回

新增两个私有方法，推荐：

```swift
private func recallControlGroup(_ group: Int)
private func saveControlGroup(_ group: Int)
```

`recallControlGroup(_:)` 行为：

1. 清 pending 和 construction preview。
2. 过滤 live IDs，并写回控制组，清理无效 ID。
3. 如果 live IDs 为空：
   - 不改变 `selectedIDs`。
   - `updateHUD()`。
   - `showMessage("Group \(group) is empty.", color: .orange)` 或白色克制文案。
   - return。
4. 如果非空：
   - `selectedIDs = liveIDs`。
   - `refreshSelection()`。
   - `updateHUD()`。
   - `showMessage("Group \(group) selected: \(liveIDs.count) units.", ...)`。

`saveControlGroup(_:)` 行为：

1. 清 pending 和 construction preview。
2. 取 `selectedMobilePlayerUnits()`。
3. 如果为空：
   - 不覆盖旧控制组。
   - 不改变 `selectedIDs`。
   - `updateHUD()`。
   - `showMessage("Select mobile units before saving Group \(group).", color: .orange)`。
   - return。
4. 如果非空：
   - 覆盖对应组为 `Set(units.map(\.id))`。
   - 不改变单位命令：不得改 `destination`、`path`、`attackTarget`、`holdPosition`、`attackMoveDestination`、`rallyPoint`。
   - 不改变 selection 本身；可以调用 `refreshSelection()`，但不应改变 `selectedIDs`。
   - `updateHUD()`。
   - `showMessage("Saved \(units.count) units to Group \(group).", ...)`。

在 `handleHudAction(_:tapCount:)` 中新增：

```swift
case .controlGroup1:
    tapCount >= 2 ? saveControlGroup(1) : recallControlGroup(1)
case .controlGroup2:
    tapCount >= 2 ? saveControlGroup(2) : recallControlGroup(2)
```

注意：

- 控制组单击召回空组时不要清空当前选择。这能避免误触空组破坏玩家当前操作。
- 保存空组时不要覆盖旧组。玩家误双击空组按钮时，旧组仍保留。
- 召回非空组会替换当前选择，这是控制组的主要功能。

### 9. 死亡清理与 SKRM 重置

最低要求：

- `restartSkirmish()` 中清空两个控制组 ID 集。
- 召回时 prune 无效 ID，保证死亡单位不会被选中。

可选但推荐：

- 在现有死亡清理函数附近，如果已有集中删除实体的位置，例如 `cullDestroyedEntities()`，顺手从控制组中移除被清理的 ID，帮助 subtitle 及时反映数量。必须保持 diff 小，不要重写死亡清理。

无论是否做可选清理，验收标准以“召回时过滤、`SKRM` 重开清空、subtitle 不长期明显错误”为准。

## 风险和禁止项

- HUD 双击风险：当前 HUD action 在 `touchesBegan` 处理，必须显式传入 `touch.tapCount` 或等价方式；不要只在世界点击路径处理 tap count，否则控制组双击不会工作。
- 布局风险：按钮从 20 个增到 22 个，必须检查 title / subtitle 足够短；不要使用长 subtitle。
- 状态残留风险：控制组操作必须清理 pending 模式和 construction preview；`SKRM` 必须清空控制组。
- 数据 stale 风险：控制组只保存 ID，召回时必须过滤 live 实体并回写清理后的 ID 集。
- 误触风险：空组召回不要清空当前选择；空选择保存不要覆盖旧组。
- 命令污染风险：保存 / 召回控制组不得改单位移动、攻击、hold、attack-move、rally 或生产命令。
- 迷雾风险：控制组选择玩家自己的机动单位，不应改变可见性、探索、支援 reveal 或敌方显示规则。
- 禁止把控制组写成 AI 功能或影响 AI 行为。
- 禁止新增外部依赖、素材、音频、XCTest target 或大体积数据。
- 禁止使用非 `main` 长期分支、PR 流、`smalldata_test`、`develop`、`codeb/...` 或候选分支。
- 禁止伪造测试、隐藏失败命令、把未运行的 XCTest 说成通过。
- 禁止用旧 artifact、旧 output 或本地输出冒充本轮最新云端结果。
- 禁止使用非 `Altman-sam114` 的 GitHub 账号伪装完成 push、CI 或 artifact 验收。
