# v1.5（非法命令反馈）

## 版本号

v1.5

## 版本分配依据

- `update_log.md` 当前最新核心玩法版本是 v1.4，命令按钮状态高亮已由 Agent C 通过 `origin/main` 最新 GitHub Actions artifact 验收闭环。
- 本轮是在 v1.4 后的小型操作体验增量，只给已有非法命令 / 无效目标路径补充轻量世界视觉反馈。
- `md/flow/flow.md` 的未来扩展点列出“操作体验：双击同类选择、控制编队、命令按钮高亮、非法命令反馈”。v1.3 已完成双击同类选择，v1.4 已完成命令按钮高亮，本轮只实现“非法命令反馈”的最小可验收切片。

## 背景

`Desert Frontline` 当前已经覆盖经济、建造、Radar Outpost、Guard Tower、陆海空单位、海军、航母、AI、地图、战争迷雾、老兵经验、支援技能、HUD、任务阶段、双击同类选择、命令按钮 pending 高亮和 HQ 胜负条件。核心输入、命令、反馈和视觉效果都集中在 `DesertFrontline/GameScene.swift`。

当前已有很多失败路径会通过 `showMessage(...)` 给出 HUD 文案，例如建筑位置无效、支援目标未侦察、没有维修目标、没有可设置集结点的来源、没有可 attack-move 的单位等。问题是这些反馈大多只出现在 HUD 顶部，不指向玩家刚点击的地图位置。v1.5 要在不改变合法性和命令语义的前提下，在玩家刚点击的无效世界目标处显示短暂的拒绝标记，帮助玩家理解“这个点不能执行当前命令”。

本轮只补充视觉反馈，不改命令判定，不新增按钮，不做禁用态，不改资源、冷却、AI、迷雾、支援效果、建造规则或 pending 清理语义。

## 目标

1. 新增一个 `showDeniedMarker(at:reason:)` 或等价 helper，在 `effectsLayer` 上用 SpriteKit shape 显示短暂的红 / 橙色拒绝反馈。
2. 视觉建议：
   - 一个红橙色 ring 或斜向 X，持续约 0.45-0.7 秒后淡出并移除。
   - 可附一个很短的地图 label，例如 `NO BUILD`、`NO SCOUT`、`NO REPAIR`、`NO UNITS`、`NO ATK`。
   - label 必须简短，避免遮挡战场；不要把完整 HUD 错误文案塞进地图 label。
3. 只在已有无效路径上补视觉反馈，不改变合法性、不新增可执行命令、不扩大目标范围。
4. Agent A 评估后的最小安全接入集合如下：
   - 建筑放置失败：`placeStructure(kind:at:)` 中地图外、`constructionIssue(...)` 返回错误、资金不足。
   - 支援技能目标失败：`executeSupportPower(_:at:faction:showFeedback:)` 中玩家目标地图外、支援资产/资金/冷却在落点时失效、非 `SCAN` 目标未侦察、`REPR` 范围内没有损伤目标。
   - pending `RLY` 在地图点击时没有有效来源：`handleWorldTap(at:tapCount:)` 中 `isSettingRallyPoint` 为真但 `selectedPlayerRallyFactories()` 为空。
   - pending `AMOV` 在地图点击时没有可战斗单位：`handleWorldTap(at:tapCount:)` 中 `isSettingAttackMove` 为真但 selected combat units 为空。
   - 普通世界点击可见敌人但当前选中机动单位全部不能攻击该目标：现有逻辑没有实际分配 `attackTarget`，本轮应显示拒绝标记并避免误导性 “Attack order” 文案。
   - 选中了玩家实体但没有任何可移动单位，也没有可设置 rally 的生产来源时点击空地：保留现有清空选择行为，只补一个轻量 `NO UNITS` 标记；如果一开始就没有任何选中项，空地点击不需要反馈，避免正常浏览地图时噪音过多。
5. 保留已有 `showMessage(...)` 文案；denied marker 是补充视觉反馈。
6. 允许为“当前没有有效文案或现有文案明显误导”的分支新增短文案，但必须克制。例如可见敌人不可攻击时可以用 `Selected units cannot attack that target.` 替代错误的 `Attack order...`。
7. 无效建筑放置、无效支援目标等现有会保留 pending 的路径必须继续保留 pending；v1.4 按钮高亮继续由 pending 状态驱动。
8. `SKRM` 重开后不应残留任何 denied marker；现有 `effectsLayer.removeAllChildren()` 已覆盖，不能引入需要额外重置的新状态。
9. 更新 `README.md` Controls / feedback 说明，记录无效地图命令会显示短暂红橙标记。
10. 更新 `md/flow/flow.md` 的输入流程，记录非法命令反馈是 `GameScene` 在失败分支上的视觉补充。
11. 必要时更新 `md/flow/flowchart.md`，只做最小同步；如果现有流程图需要表达无效 pending 反馈，可在玩家输入图中补一个简短节点。
12. 不更新 `update_log.md`。正式 v1.5 记录由 Agent C 在最新 artifact 验收通过后追加。

## 非目标

- 不新增 HUD 按钮、禁用态、热键、控制编队、音效、设置项、菜单或大 UI。
- 不改变 `BASE`、`RLY`、`AMOV`、`SCAN`、`REPR`、`AIRS`、`BARR` 的 pending 进入和清理语义。
- 不改变建筑合法性、基地覆盖、可见性、地形规则、资金扣除或施工流程。
- 不改变支援技能费用、冷却、资产需求、目标合法性、伤害、维修、侦察或潜艇 reveal 链路。
- 不改变普通移动、攻击、attack-move、hold、路径、AI、经济、生产、任务、胜负、老兵 XP、战争迷雾或声呐检测。
- 不给 AI 的内部失败路径显示玩家 denied marker；本轮 marker 只面向玩家可见 / 玩家 showFeedback 分支。
- 不把建筑 preview 的红色预览改成持续报错，也不在拖动 preview 时连续刷 denied marker；只在玩家点击尝试执行时显示。
- 不在 HUD-only 无效操作上强行显示世界 marker，例如按钮点击时没有工厂、没有资金、支援冷却中、生产来源缺失、`HOLD` 无单位等。这些没有明确世界目标，保留现有 `showMessage(...)` 即可。
- 不把普通无选中空地点击都标成错误；没有选中项时允许玩家点击地图浏览 / 清空状态，不制造噪音。
- 不新增外部依赖、图片素材、音频素材、XCTest target 或大体积测试数据。
- 不拆分 `GameScene.swift`，不做大规模重构，不做全文件格式化。
- 不改 Xcode project，除非编译必须且能说明原因。
- 不默认跑本机完整 Xcode build 或模拟器；除非人工明确要求，本轮本地只跑轻量检查，构建以 GitHub Actions artifact 为准。

关键源码事实：

- SwiftUI 宿主很薄：`DesertFrontlineApp` 创建 `GameView`，`GameView` 用 `SpriteView` 承载 `SceneHolder.scene`，实际 HUD、输入、命令和视觉反馈都在 `GameScene`。
- `SceneHolder` 创建固定尺寸 `GameScene(size: 1366x1024)`，`scaleMode = .aspectFill`。
- `GameScene` 使用 `worldNode` 承载地图、实体、效果和迷雾；`effectsLayer` 当前用于移动 marker、attack-move marker、hold marker、受击 ping、施工 pulse、支援效果、生产 ready、弹道、爆炸、维修火花和声呐 ping。
- `fogLayer` 位于 `effectsLayer` 之上；本轮 denied marker 应作为世界反馈加入 `effectsLayer`，不要绕过战争迷雾另建 HUD 投影。`showMessage(...)` 仍是不可见 / 未侦察目标的主要说明。
- 当前 pending 状态字段是：
  - `pendingConstructionKind: EntityKind?`
  - `pendingSupportPower: SupportPower?`
  - `isSettingRallyPoint: Bool`
  - `isSettingAttackMove: Bool`
- v1.4 按钮高亮由 `pendingConstructionKind`、`pendingSupportPower`、`isSettingRallyPoint`、`isSettingAttackMove` 推导；本轮不得新增第二份“拒绝 / pending / active”状态。
- `handleWorldTap(at:tapCount:)` 的优先级是 pending 支援 -> pending 建筑 -> pending 集结点 -> pending `AMOV` -> 世界实体点击 -> 移动 / 清空选择。
- `placeStructure(kind:at:)` 当前失败路径会 `showMessage(...)` 后返回，成功后才扣钱、清 `pendingConstructionKind`、清 preview、创建结构并更新 HUD。
- `executeSupportPower(_:at:faction:showFeedback:)` 当前玩家失败路径包括地图外、`supportIssue(...)`、未侦察、无维修目标；其中部分失败保留 `pendingSupportPower`，`supportIssue(...)` 分支会按现有逻辑清理玩家 pending。
- `selectedPlayerRallyFactories()` 只返回玩家存活且 `supportsRallyPoint` 的来源；`setRallyPoint(to:for:)` 会为不同来源用 `navigablePoint(...)` 调整目标并显示 move marker。
- `issueAttackMoveOrder(to:units:)` 内部会过滤存活、非结构、`damage > 0` 的 combat units；有效时复用 `issueFormationMove(..., attackMove: true)`。
- 普通点击可见敌人时，当前代码只对 `unit.kind.canAttack(tapped.kind)` 的单位设置 `attackTarget`，但无论是否有单位可攻击都会显示 `Attack order: ...` 并 return。本轮应修正这个反馈，不改变不可攻击目标不会被攻击的事实。
- 普通点击空地时，若 `selectedMobilePlayerUnits()` 为空且 `selectedPlayerRallyFactories()` 为空，当前代码清空选择并返回；本轮只在之前确实有选中实体但不能执行地面命令时给轻量 marker。
- `restartSkirmish()` 会清空 entities、selectedIDs、buildOrders、pending、support cooldown/reveal、任务、经济、迷雾，并 `effectsLayer.removeAllChildren()`；本轮不得引入需要额外重置的持久 marker 状态。
- `GameView.swift`、`DesertFrontlineApp.swift` 和 Xcode project 当前不需要为本轮功能修改。

## 实现步骤

### 1. 开始前同步 main 并确认工作区归属

Agent B 必须先执行：

```sh
git fetch origin
git switch main
git pull --ff-only origin main
git status --short
```

要求：

- 当前工作区应干净。
- 如果 `git status --short` 显示任何未提交、未跟踪或冲突文件，B 必须停止并确认归属。
- 如果唯一未跟踪 / 修改文件是本提示词文件 `md/prompt/v1（核心玩法）/v1.5（非法命令反馈）.md`，B 必须明确确认它属于本轮 Agent A 产物后再继续；不要默认把无关脏改动混入。
- 如果 `git pull --ff-only` 因本地改动无法进行，B 不能回滚或覆盖，必须停止并报告阻塞。

### 2. 新增 denied marker helper

在 `GameScene.swift` 现有效果 helper 附近新增一个小 helper，推荐放在 `showMoveMarker(at:)` / `showAttackMoveMarker(at:)` / `showHoldMarker(at:)` 附近。

建议形态：

```swift
private func showDeniedMarker(at point: CGPoint, reason: String) {
    let color = UIColor(red: 1.0, green: 0.30, blue: 0.18, alpha: 1.0)
    let node = SKNode()
    node.position = point
    node.zPosition = 268

    let ring = SKShapeNode(ellipseOf: CGSize(width: 54, height: 25))
    ring.fillColor = color.withAlphaComponent(0.10)
    ring.strokeColor = color
    ring.lineWidth = 3
    ring.glowWidth = 1.5
    node.addChild(ring)

    let crossPath = CGMutablePath()
    crossPath.move(to: CGPoint(x: -14, y: -10))
    crossPath.addLine(to: CGPoint(x: 14, y: 10))
    crossPath.move(to: CGPoint(x: -14, y: 10))
    crossPath.addLine(to: CGPoint(x: 14, y: -10))

    let cross = SKShapeNode(path: crossPath)
    cross.strokeColor = color
    cross.lineWidth = 3
    cross.lineCap = .round
    node.addChild(cross)

    let label = SKLabelNode(fontNamed: "Menlo-Bold")
    label.text = reason
    label.fontSize = 10
    label.fontColor = color
    label.verticalAlignmentMode = .center
    label.horizontalAlignmentMode = .center
    label.position = CGPoint(x: 0, y: 27)
    node.addChild(label)

    effectsLayer.addChild(node)
    node.run(.sequence([
        .group([
            .scale(to: 1.35, duration: 0.52),
            .fadeOut(withDuration: 0.52)
        ]),
        .removeFromParent()
    ]))
}
```

可以微调尺寸、颜色、duration 和 zPosition，但必须满足：

- 使用 SpriteKit shape / label，不新增素材。
- 一次点击只添加短生命周期节点，自动 `removeFromParent()`。
- 不保存持久状态，不需要在 `update(_:)` 中维护。
- 不使用重复 forever action，不在每帧重复创建。
- `reason` 使用短码，不使用长句。
- 不遮挡 HUD，不把 marker 添加到 `hudNode`。

如需减少重复，可新增一个小 helper 计算 marker 落点：

```swift
private func deniedMarkerPoint(for point: CGPoint) -> CGPoint {
    tile(at: point).map(tileCenter) ?? point
}
```

该 helper 可选；不要为了它大规模重构输入链路。

### 3. 建筑放置失败补 marker

在 `placeStructure(kind:at:)` 的失败返回前补 `showDeniedMarker(...)`：

- `tile(at:) == nil`：在原始点击点显示 `NO MAP` 或 `NO BUILD`。
- `constructionIssue(...)` 返回错误：在 `tileCenter(tile)` 显示 `NO BUILD`。
- `playerMoney < kind.cost`：在 `tileCenter(tile)` 显示 `NO FUNDS`。

要求：

- 保留现有 `showMessage(...)` 文案。
- 不清理 `pendingConstructionKind`，不清理 preview，不调用 `clearConstructionPreview()`。
- 不扣钱，不创建结构，不更新 AI / 经济 / 迷雾。
- 建筑放置成功路径不显示 denied marker，继续显示现有施工 pulse 和 message。

### 4. 支援技能目标失败补 marker

只在玩家触发且 `showFeedback == true` 的失败路径显示 denied marker；敌方 AI 调用 `executeSupportPower(... faction: .enemy ...)` 不显示玩家 denied marker。

建议接入：

- 地图外：`tile(at:) == nil` 时，在原始点击点显示 `NO MAP`。
- `supportIssue(for:faction:)` 返回错误时：
  - 保持现有 `pendingSupportPower = nil` 语义。
  - 在 `targetPoint` 或原始点击点显示 `NO SUP` / `NO ASSET` / `NO FUNDS` 这类短码，具体不必细分太多。
- 玩家非 `reconSweep` 且 `!exploredTiles.contains(tile)`：在 `targetPoint` 显示 `NO SCOUT`。
- `fieldRepair` 无维修目标：在 `targetPoint` 显示 `NO REPAIR`。

要求：

- 保留现有 `showMessage(...)` 文案。
- 不改变现有 pending 清理语义：
  - 地图外、未侦察、无维修目标当前保留 `pendingSupportPower`，实现后仍保留。
  - `supportIssue(...)` 当前会清玩家 pending，仍保持。
- 不改变支援扣钱、冷却、reveal、伤害、维修或潜艇 reveal 行为。
- 支援成功路径不显示 denied marker，继续显示 `showSupportPowerEffect(...)`。

### 5. pending RLY / AMOV 失效补 marker

在 `handleWorldTap(at:tapCount:)` 中：

- `isSettingRallyPoint == true` 且 `selectedPlayerRallyFactories()` 为空时：
  - 在点击点或 snapping 后的点显示 `NO SOURCE`。
  - 保留现有 `isSettingRallyPoint = false`、`showMessage("No factory or carrier selected.", ...)`、`updateHUD()` 语义。
- `isSettingAttackMove == true` 且 selected combat units 为空时：
  - 在点击点显示 `NO UNITS`。
  - 保留现有 `isSettingAttackMove = false`、`showMessage("No combat units selected.", ...)`、`updateHUD()` 语义。

有效 `RLY` / `AMOV` 路径不显示 denied marker，继续显示 rally marker、move marker 或 attack-move marker。

### 6. 普通攻击无效目标补 marker

在 `handleWorldTap(at:tapCount:)` 的可见敌人点击分支中，当前代码对可攻击单位设置 `attackTarget` 后直接显示 `Attack order: ...`。本轮要让反馈匹配实际状态：

- 计算本次实际被分配攻击目标的单位数量，例如 `var assignedAttackers = 0`。
- 只有 `assignedAttackers > 0` 时，才显示现有 `Attack order: \(tapped.kind.displayName).`。
- 如果 selected mobile units 非空、目标是敌方、但 `assignedAttackers == 0`：
  - 不设置移动命令。
  - 不清空选择。
  - 在 `tapped.node.position` 显示 `NO ATK`。
  - 显示简短 HUD 文案，例如 `Selected units cannot attack that target.`。
  - return，保持现有“不可攻击目标不会被攻击，也不会退化为地面移动”的行为。

不要改变 `entity(at:)` 的迷雾过滤，不允许玩家可靠攻击迷雾中不可见敌人。

### 7. 选中非移动 / 非 rally 实体后点击空地补 marker

在普通世界点击落到“没有 selected mobile units”分支时：

- 如果 `selectedPlayerRallyFactories()` 非空，继续现有 `setRallyPoint(to:for:)` 行为，不显示 denied marker。
- 如果 `selectedPlayerRallyFactories()` 为空：
  - 如果 `selectedIDs` 在清空前非空，说明玩家当前选中了无法执行地面命令的实体，在点击点显示 `NO UNITS`。
  - 然后保留现有 `selectedIDs.removeAll()`、`refreshSelection()` 行为。
  - 如果 `selectedIDs` 本来就是空的，不显示 marker，避免普通空地点击产生噪音。

不要为单击己方实体选择、双击同类选择、框选空结果、小地图点击、双指平移缩放或 HUD button failure 添加世界 marker。

### 8. 保持 HUD / pending /重开行为

实现后检查并保持：

- v1.4 的 `BASE`、`RLY`、`AMOV`、支援按钮高亮仍完全由 pending 字段驱动。
- 无效建筑放置和多数无效支援目标如果现有逻辑保留 pending，高亮也继续保留。
- 有效执行或现有逻辑清 pending 的路径，高亮继续消失。
- `restartSkirmish()` 不需要新增 denied marker 清理逻辑；如果实现引入了额外状态，说明设计已经偏离本轮目标，应改回无状态短生命周期节点。

## 风险和禁止项

- 风险：把 denied marker 接到 `touchesMoved` 或建筑 preview 更新中会在拖动时连续刷节点。只在点击尝试执行的失败分支显示。
- 风险：改变 pending 清理会破坏 v1.4 高亮和现有支援 / 建筑体验。必须保留每个失败路径原有 pending 语义。
- 风险：把 marker 加到 `hudNode` 会变成屏幕提示而不是地图位置提示；本轮应放在世界效果层。
- 风险：为敌方 AI 支援失败显示 marker 会给玩家暴露不该知道的信息。只在玩家 showFeedback 分支显示。
- 风险：普通无选中空地点击如果都标红，会让地图浏览和取消选择变吵。只有“之前选中了不能执行地面命令的实体”时才标记。
- 风险：可见敌人不可攻击分支当前会误报 attack order；修正反馈时不要让单位改为移动到敌人脚下，也不要绕过 `canAttack(...)`。
- 风险：地图 label 太长会遮挡单位和战斗。使用短码，完整原因继续由 `showMessage(...)` 承担。
- 风险：marker zPosition 过高或挂到错误层可能绕过迷雾。默认作为世界效果加入 `effectsLayer`，不要另建绕雾显示系统。
- 禁止新增外部依赖或素材。
- 禁止改动 Xcode project，除非编译必须且能说明原因。
- 禁止扩大到控制编队、热键、按钮禁用态、音效、HUD 重排或 UI 大改。
- 禁止改变 HQ 胜负条件、经济、生产、AI、战斗、老兵 XP、迷雾、支援技能、任务阶段和潜艇侦测链路。
- 禁止绕过 `GameScene.swift` 当前状态流另建平行玩法系统。
- 禁止回滚、覆盖或混入用户 / 其他 Agent 的无关改动；归属不清时停止报告。
- 禁止伪造测试、隐藏失败命令、把未运行的本机 build 或未通过的云端结果说成已验证。
- 禁止使用非 `Altman-sam114` 的 GitHub 账号伪装完成 push、CI 或 artifact 验收。
