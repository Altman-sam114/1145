# v1.4（命令按钮状态高亮）

## 版本号

v1.4

## 版本分配依据

- `update_log.md` 当前最新核心玩法版本是 v1.3，双击同类选择已由 Agent C 通过 `origin/main` 最新 GitHub Actions artifact 验收闭环。
- 本轮是在 v1.3 后的小型操作体验增量，只增强 HUD 命令按钮的 armed / pending 状态反馈。
- `md/flow/flow.md` 的未来扩展点列出“操作体验：双击同类选择、控制编队、命令按钮高亮、非法命令反馈”。v1.3 已完成双击同类选择，本轮只实现其中最小可验收切片：命令按钮状态高亮。

## 背景

`Desert Frontline` 当前已经覆盖经济、建造、Radar Outpost、Guard Tower、陆海空单位、海军、航母、AI、地图、战争迷雾、老兵经验、支援技能、HUD、任务阶段、双击同类选择和 HQ 胜负条件。核心输入、选择、命令、HUD 和反馈都集中在 `DesertFrontline/GameScene.swift`。

现有 HUD 底部命令条支持 `BASE` 建筑放置、`RLY` 集结点、`AMOV` 攻击移动，以及 `SCAN` / `REPR` / `AIRS` / `BARR` 支援技能。进入这些模式后，`selectedLabel` 和选择信息面板会提示玩家下一步，但对应按钮本身没有明显 armed / pending 高亮。玩家需要更直接地看出“当前正在等待地图目标”的按钮状态。

本轮只给已有按钮增加状态高亮，不改变按钮数量、布局、命令语义、资源消耗、冷却、支援资产、触摸优先级或任何 RTS 规则。

## 目标

1. HUD 命令按钮能显示当前 armed / pending 状态：
   - `BASE`：`pendingConstructionKind != nil` 时高亮。
   - `RLY`：`isSettingRallyPoint == true` 时高亮。
   - `AMOV`：`isSettingAttackMove == true` 时高亮。
   - `SCAN` / `REPR` / `AIRS` / `BARR`：`pendingSupportPower` 等于对应 `HudAction.supportPower` 时高亮。
2. 高亮必须明显，建议使用按钮 `SKShapeNode` 的 `strokeColor`、`glowWidth`、`lineWidth`、`alpha` 或轻量 fill 调整；普通按钮仍保留现有 `buttonColor(for:)` 底色风格。
3. 高亮必须随状态实时变化：
   - 点击 `BASE` 进入建筑放置后，`BASE` 高亮。
   - 建筑放置成功、被其他 HUD 命令打断、`ARMY` / `HQ` / `AI` / 生产按钮清理 pending 后，高亮消失。
   - 点击 `RLY` 且选中可设置集结点的工厂或 Carrier 后，`RLY` 高亮；设置完成或被其他模式清理后消失。
   - 点击 `AMOV` 且有可战斗单位后，`AMOV` 高亮；下达目标、选择其他模式或清理后消失。
   - 点击可用支援技能后，对应支援按钮高亮；执行成功、因资产/资金/冷却失败而取消、或被其他模式清理后状态正确。
4. 高亮必须随 `layoutHUD()` 重建、`updateHUD()` 刷新、pending 模式切换和 `SKRM` 重置正确更新。
5. 保持现有 `buttonColor(for:)`、`subtitle(for:)` 和 support cooldown 文案行为；支援技能冷却倒计时继续显示并保持现有颜色提示。
6. 保持 pending 模式互斥清理不变，不改变现有触摸、HUD、小地图、双指平移缩放、框选、单击选择、双击同类选择、移动、攻击、集结点和支援目标链路。
7. 更新 `README.md` Controls，说明 armed / pending 命令按钮会高亮。
8. 更新 `md/flow/flow.md` 的 HUD / 输入说明，记录 HUD 按钮高亮由 pending 状态驱动。
9. 必要时更新 `md/flow/flowchart.md`，只做最小同步；如果现有流程图不需要新增节点，可不改。
10. 不更新 `update_log.md`。正式 v1.4 记录由 Agent C 在最新 artifact 验收通过后追加。

关键源码事实：

- SwiftUI 宿主很薄：`DesertFrontlineApp` 创建 `GameView`，`GameView` 用 `SpriteView` 承载 `SceneHolder.scene`，实际 HUD、输入和游戏状态都在 `GameScene`。
- `SceneHolder` 创建固定尺寸 `GameScene(size: 1366x1024)`，`scaleMode = .aspectFill`。
- `GameScene` 使用 `cameraRig` 作为 `SKCameraNode`，HUD 挂在 `cameraRig` 下，世界内容挂在 `worldNode` 下。
- `HudAction` 已定义底部命令条动作，`title` 提供按钮短码，`buildKind` 和 `supportPower` 提供生产/支援映射。
- 当前 pending 状态字段是：
  - `pendingConstructionKind: EntityKind?`
  - `pendingSupportPower: SupportPower?`
  - `isSettingRallyPoint: Bool`
  - `isSettingAttackMove: Bool`
- `layoutHUD()` 会 `hudNode.removeAllChildren()`，清空 `hudButtonFrames`、`hudButtonSubtitleLabels` 和选择信息 labels，然后重新创建所有 HUD 节点。
- `makeButton(action:frame:)` 当前创建按钮 `SKShapeNode`，设置 `shape.fillColor = buttonColor(for: action)`、黑色 stroke 和固定 lineWidth，但没有保存 shape 引用。
- `buttonColor(for:)` 是现有按钮底色来源，本轮应保留，不要把它改成只服务 active 状态的复杂系统。
- `subtitle(for:)` 会显示生产费用、支援费用/冷却、`BASE` 当前轮换建筑费用、AI 难度等；本轮必须保持这些文案。
- `updateHUD()` 每帧刷新金钱、任务、选择信息、小地图和 `hudButtonSubtitleLabels`；支援 cooldown 时会把 subtitle 字色改为橙色。
- `handleHudAction(_:)` 已在不同按钮分支中清理互斥 pending 状态，并在多处分支调用 `layoutHUD()` / `updateHUD()`。
- `selectSupportPower(_:)` 设置 `pendingSupportPower = power`，但正常只调用 `updateHUD()`，不一定重建 HUD。
- `handleWorldTap(at:tapCount:)` 的优先级是 pending 支援 -> pending 建筑 -> pending 集结点 -> pending `AMOV` -> 世界实体点击 -> 移动/清空选择。
- `placeStructure(kind:at:)` 成功后会清空 `pendingConstructionKind`、清除预览，并调用 `layoutHUD()` / `updateHUD()`。
- `executeSupportPower(_:at:faction:showFeedback:)` 成功后会清空玩家 `pendingSupportPower` 并调用 `updateHUD()`；部分无效目标会保留 pending，按钮应继续高亮。
- `restartSkirmish()` 会清空 `selectedIDs`、`pendingConstructionKind`、`pendingSupportPower`、`isSettingRallyPoint`、`isSettingAttackMove`、经济、任务、迷雾和实体，再调用 `layoutHUD()`、`updateFog(force:)`、`updateHUD()`。
- v1.3 已在 `touchesEnded` 传递 `tapCount`，并新增 `visibleWorldRect` / `selectVisiblePlayerUnits(matching:)`；本轮不得回归该双击同类选择链路。

### 2. 保存 HUD 按钮 shape 引用

在 `GameScene` 的 HUD 状态字段附近新增一个小型字典，例如：

```swift
private var hudButtonShapes: [HudAction: SKShapeNode] = [:]
```

实现要求：

- 在 `layoutHUD()` 开始处，和 `hudButtonFrames.removeAll()`、`hudButtonSubtitleLabels.removeAll()` 一起清空 `hudButtonShapes`。
- 在 `makeButton(action:frame:)` 创建 `shape` 后保存：

```swift
hudButtonShapes[action] = shape
```

- 不改变 `hudButtonFrames` 的 frame 坐标模型，不改变 `hudAction(at:)`。
- 不需要保存 title label，除非实现高亮确实需要；本轮推荐只改 shape 样式。

### 3. 新增 active 状态判定 helper

新增一个局部 helper，集中判断按钮是否处于 pending / armed 状态。建议形态：

```swift
private func isHudActionArmed(_ action: HudAction) -> Bool {
    switch action {
    case .buildBase:
        return pendingConstructionKind != nil
    case .setRally:
        return isSettingRallyPoint
    case .attackMove:
        return isSettingAttackMove
    case .reconSweep, .fieldRepair, .airStrike, .navalBarrage:
        return action.supportPower != nil && action.supportPower == pendingSupportPower
    default:
        return false
    }
}
```

可按 Swift 类型推断调整写法，但必须保持语义：

- 只有当前对应的 pending 模式高亮。
- 普通生产按钮不高亮。
- `HOLD` 不高亮，因为它是立即命令，不是等待地图目标的 pending 模式。
- `ARMY`、`HQ`、`AI`、`SKRM` 不高亮。
- 支援技能 cooldown 不等于 armed；只有被选中等待地图目标时才高亮。

### 4. 新增统一按钮样式刷新 helper

新增一个统一刷新函数，避免把样式逻辑散落在 `handleHudAction` 各分支里。建议形态：

```swift
private func refreshHudButtonStyles() {
    for (action, shape) in hudButtonShapes {
        shape.fillColor = buttonColor(for: action)
        if isHudActionArmed(action) {
            shape.strokeColor = UIColor(red: 1.0, green: 0.86, blue: 0.30, alpha: 1.0)
            shape.lineWidth = 5.5
            shape.glowWidth = 3
            shape.alpha = 1.0
        } else {
            shape.strokeColor = UIColor.black
            shape.lineWidth = 4
            shape.glowWidth = 0
            shape.alpha = 1.0
        }
    }
}
```

要求：

- 高亮必须清晰可见，且不遮挡按钮标题和 subtitle。
- 可以选择 amber、cyan 或与现有 HUD 调性一致的颜色；不要引入大面积闪烁或持续动画。
- 普通状态必须恢复到现有视觉基线：黑色 stroke、`lineWidth = 4`、无 glow、现有 fill。
- 不要因为 active 样式改变 `subtitle(for:)`、支援 cooldown 字色或按钮点击区域。
- 不要用 `SKAction` 循环动画实现高亮，避免按钮在每帧 `updateHUD()` 中重复添加 action。

### 5. 接入 `layoutHUD()` 和 `updateHUD()`

最小可靠接入方式：

- 在 `layoutHUD()` 创建所有按钮后调用一次 `refreshHudButtonStyles()`，保证 HUD 重建后立即拥有正确高亮。
- 在 `updateHUD()` 刷新 subtitle 后调用 `refreshHudButtonStyles()`，保证 pending 模式变化、支援执行失败保留 pending、支援执行成功清 pending、`SKRM` 重置等场景都能同步样式。

注意：

- `didChangeSize(_:)` 当前只调用 `layoutHUD()`，不直接调用 `updateHUD()`。因此 `layoutHUD()` 末尾的样式刷新很重要。
- `selectSupportPower(_:)` 正常只调用 `updateHUD()`，所以 `updateHUD()` 中也必须刷新样式。
- 现有多处分支已调用 `layoutHUD()` / `updateHUD()`，本轮不要为了高亮大规模重排这些调用。

### 6. 保持 pending 模式互斥清理

实现高亮时不要改变现有状态清理语义。特别保持：

- 点击 `ARMY`、`HQ`、`AI`、生产按钮会清理 pending 建筑、支援、集结点和 `AMOV`。
- 点击 `BASE` 会清理支援、集结点、`AMOV`，设置 `pendingConstructionKind` 并清空选择。
- 点击 `RLY` 会清理建筑、支援、`AMOV`，只在有选中工厂或 Carrier 时进入 `isSettingRallyPoint`。
- 点击 `AMOV` 会清理建筑、支援、集结点，只有有可战斗单位时进入 `isSettingAttackMove`。
- 点击支援技能会通过 `supportIssue(for:faction:)` 检查资金、冷却和资产；不可用时不进入 pending，不应高亮。
- pending 支援目标无效且现有逻辑保留 pending 时，支援按钮应继续高亮。
- `SKRM` 后所有 pending 状态清空，高亮全部消失。

## 验收标准

功能标准：

- 初始开局无 pending 模式时，`BASE`、`RLY`、`AMOV`、`SCAN`、`REPR`、`AIRS`、`BARR` 都不显示 armed 高亮。
- 点击 `BASE` 进入建筑放置后，`BASE` 明显高亮；建筑放置成功、点击 `ARMY` / `HQ` / `AI` / 生产按钮或 `SKRM` 后高亮消失。
- `BASE` 轮换到 War Factory、Airfield、Radar Outpost、Guard Tower、Shipyard、Oil Derrick 时，subtitle 继续显示当前费用和短码。
- 选中 War Factory、Airfield、Shipyard 或 Carrier 后点击 `RLY`，`RLY` 明显高亮；设置地图目标后高亮消失，rally marker 和选择信息仍工作。
- 未选中可设置集结点的来源时点击 `RLY`，不进入 pending，不高亮，并保留现有错误反馈。
- 选中可战斗单位后点击 `AMOV`，`AMOV` 明显高亮；点击地图发出 attack-move 后高亮消失。
- 无可战斗单位时点击 `AMOV`，不进入 pending，不高亮，并保留现有错误反馈。
- 点击可用 `SCAN` / `REPR` / `AIRS` / `BARR` 后，仅对应支援按钮高亮；执行成功后高亮消失。
- 支援技能冷却中或资产不足时点击按钮，不进入 pending，不高亮；现有 cooldown subtitle 和错误文案继续工作。
- pending 支援技能目标无效且现有逻辑保留 pending 时，对应支援按钮继续高亮。
- 任意时刻最多只应有一个等待地图目标的按钮高亮：`BASE`、`RLY`、`AMOV` 或一个支援按钮。
- `layoutHUD()` 重建后，高亮状态仍正确；`didChangeSize(_:)` 后不丢状态。
- `SKRM` 重开后所有高亮清空，金钱、任务、实体、迷雾、选择和 pending 模式仍重置。
- 单击选择、双击同类选择、框选、HUD 点击、小地图、双指平移缩放、普通移动、攻击命令、建筑预览、支援目标、集结点和 `AMOV` 不回归。

架构标准：

- 不新增平行 HUD 系统、输入系统、命令系统或状态系统。
- 高亮状态直接由现有 pending 字段推导，不保存第二份“当前高亮按钮”可变状态。
- `hudButtonShapes` 只缓存当前布局中的按钮 shape 引用，随 `layoutHUD()` 重建清空再填充。
- `buttonColor(for:)` 和 `subtitle(for:)` 继续作为底色和副标题事实来源。

## 风险和禁止项

- 风险：只在 `layoutHUD()` 设置高亮会漏掉 `selectSupportPower(_:)` 等只调用 `updateHUD()` 的状态变化。必须在 `updateHUD()` 也刷新样式。
- 风险：只在按钮点击瞬间保存“当前高亮按钮”会形成第二份状态，容易和 pending 字段不一致。高亮必须由现有 pending 字段推导。
- 风险：`layoutHUD()` 会移除并重建所有 HUD 节点；如果不清空并重填 shape 字典，可能引用已移除节点。
- 风险：支援 cooldown 不是 pending；不能把冷却中按钮误高亮。
- 风险：`executeSupportPower` 的某些失败路径会保留 pending；高亮应跟随真实 pending，而不是假设任何失败都清理。
- 风险：每帧添加 `SKAction` 动画可能积累 action 或造成闪烁。本轮推荐静态 stroke/glow 高亮。
- 风险：样式过重可能遮挡短码和 cooldown subtitle；保持高亮清晰但克制。
- 禁止新增外部依赖或素材。
- 禁止改动 Xcode project，除非编译必须且能说明原因。
- 禁止扩大到控制编队、热键、按钮禁用态、非法命令反馈、HUD 重排或 UI 大改。
- 禁止改变 HQ 胜负条件、经济、生产、AI、战斗、老兵 XP、迷雾、支援技能、任务阶段和潜艇侦测链路。
- 禁止绕过 `GameScene.swift` 当前状态流另建平行玩法系统。
- 禁止回滚、覆盖或混入用户/其他 Agent 的无关改动；归属不清时停止报告。
- 禁止伪造测试、隐藏失败命令、把未运行的本机 build 或未通过的云端结果说成已验证。
- 禁止使用非 `Altman-sam114` 的 GitHub 账号伪装完成 push、CI 或 artifact 验收。
