# v1.2（引入防御炮塔）

## 版本号

v1.2

## 版本分配依据

- `update_log.md` 当前最新核心玩法版本是 v1.1，已完成 Radar Outpost，并通过 `origin/main` 最新 GitHub Actions artifact 验收。
- v1.1 已经把建筑科技层的小增量接入 `EntityKind`、`BASE` 建造轮换、AI 缺失建筑补建、迷雾、HUD、战略价值和文档链路。
- 本轮沿用同一建筑科技层推进方式，在 Radar Outpost 之后增加单一基地防御建筑 Guard Tower，不进入完整科技树、防空专精、岸防炮或升级系统。
- 本轮同时修正一个通用架构边界：任何未完工攻击型结构都不能开火。这是 Guard Tower 以及未来防御建筑必须依赖的施工状态边界。

## 背景

`Desert Frontline` 当前已经覆盖经济、建造、Radar Outpost、陆海空单位、海军、航母、AI、地图、战争迷雾、老兵经验、支援技能、HUD、任务阶段和 HQ 胜负条件。当前核心逻辑集中在 `DesertFrontline/GameScene.swift`，玩家和 AI 共享建造、生产、移动、攻击、支援、迷雾和胜负循环。

v1.1 后可建造结构包含 War Factory、Airfield、Radar Outpost、Shipyard 和 Oil Derrick。当前 HQ 已有防御火力，但玩家/AI 还没有可在基地周边补建的专用防御结构。引入 Guard Tower 可以增强基地防守与目标优先级，同时保持本轮范围足够小：只新增一个普通陆地结构，复用现有 `canAttack`、`updateCombat`、`fire`、`nearestTarget`、fog、死亡清理、HUD 和 AI 评分链路。

## 目标

1. 新增 `EntityKind.guardTower`，显示名建议 `Guard Tower`，短码 `GT`。
2. Guard Tower 是普通陆地结构：
   - 可由玩家通过 `BASE` 轮换建造。
   - 可被攻击、可被摧毁、走现有血条、施工进度、选择框、迷雾隐藏和死亡清理。
   - 不生产单位、不赚钱、不占领油井/旗点、不设置 rally point、不作为支援技能资产。
   - 不复用 Oil Derrick 的油田规则，不复用 Shipyard 的海岸规则。
3. AI 必须能通过现有缺失建筑补建逻辑建造/重建 Guard Tower，形成“单一基地防御建筑”的最小行为闭环。
4. Guard Tower 施工未完成前不得开火、不得提供视野、不得扩展基地覆盖、不得提供任何其他效果。
5. Guard Tower 完工后自动攻击现有可见性规则允许的敌方 land / air / structure 目标。
6. Guard Tower 初版倾向不攻击 naval 和 submarine：
   - 不攻击普通海军目标。
   - 不攻击潜艇，也不加入声呐链路。
   - 不改变潜艇 `revealedUntil`、声呐检测或开火暴露逻辑。
7. 必须修正 `updateCombat(dt:)` 边界：任何 `kind.isStructure && !isOperational` 的攻击型结构都不能搜索目标、保持目标或开火。
8. Guard Tower 必须共享现有战斗链路：
   - `EntityKind.canAttack(_:)`
   - `updateCombat(dt:)`
   - `nearestTarget(for:maxDistance:)`
   - `fire(attacker:target:)`
   - `effectiveDamage` / `effectiveAttackCooldown`
   - `isKnownToFaction(...)`
   - `cullDestroyedEntities()`
9. 添加 SpriteKit 形状绘制，不新增外部素材或依赖。
10. HUD 选择信息、战略价值、AI 目标评分、README、`md/flow/flow.md` 和必要的 `md/flow/flowchart.md` 必须同步当前真实行为。
11. `SKRM` 重开后 Guard Tower、pending 建造、AI 状态、迷雾、支援冷却和选择状态必须随现有重开链路重置。

## 非目标

- 不新增 SAM、防空专精、岸防炮、反潜塔、科技中心、完整科技树、升级系统或任务阶段。
- 不新增新单位、弹药类型、护甲系统、射界系统、建造队列 UI 或独立防御建筑菜单。
- 不改变 HQ 胜负条件、任务阶段数量、基础经济、AI 难度参数或生产单位列表。
- 不让 Guard Tower 生产单位、设置 rally point、赚钱、占领、提供 `SCAN` / `REPR` / `AIRS` / `BARR` 资产。
- 不让 Guard Tower 攻击 naval / submarine，除非人工后续明确要求岸防或反潜能力。
- 不新增外部依赖、图片素材、音频素材、XCTest target 或大体积测试数据。
- 不拆分 `GameScene.swift`，不做大规模重构，不做全文件格式化。
- 不绕过战争迷雾：玩家方 Guard Tower 只能攻击 `isKnownToFaction(..., observer: .player)` 允许的目标。
- 不默认跑本机完整 Xcode build；除非人工明确要求，本轮本地只跑轻量检查，构建以 GitHub Actions artifact 为准。
- 不预写 `update_log.md` 的正式 v1.2 记录；Agent C 验收通过后再追加。

关键源码事实：

- `EntityKind` 集中定义单位/建筑的显示名、短码、domain、成本、建造时间、HP、速度、攻击射程、伤害、冷却、视野、占地、生产来源、可生产项和可攻击目标。
- 当前结构包含 HQ、War Factory、Airfield、Shipyard、Oil Derrick、Radar Outpost；`buildableStructures` 当前是 `[.barracks, .airfield, .radarOutpost, .shipyard, .oilDerrick]`。
- `GameEntity.isOperational` 由 `!kind.isStructure || buildProgressRemaining <= 0` 决定；未完工建筑可被攻击，但不能生产、赚钱、提供视野或扩展基地覆盖。
- `updateFog(force:dt:)` 已跳过 `entity.kind.isStructure && !entity.isOperational` 的玩家结构，因此未完工结构不会贡献玩家视野。
- `hasNearbyStructure(...)` 只统计 `entity.kind.isStructure && entity.isOperational` 的友方结构，因此未完工结构不会扩展基地覆盖。
- 玩家建造走 `handleHudAction(.buildBase)` -> `pendingConstructionKind` -> `updateConstructionPreview(...)` -> `placeStructure(...)` -> `startConstruction(...)`。
- `constructionIssue(...)` 已统一检查玩家视野、附近已完工友方基地覆盖、油井油田规则、Shipyard 海岸规则、普通陆地结构和占地。
- `selectedPlayerRallyFactories()` 只允许 `kind.supportsRallyPoint` 的实体设置 rally，Guard Tower 必须保持 `supportsRallyPoint == false`。
- `queueBuild(...)` 依赖 `requiredFactory`；Guard Tower 必须没有 `requiredFactory`，不能进入单位生产队列。
- `updateCombat(dt:)` 当前遍历 `entity.isAlive && entity.kind.damage > 0` 的实体并执行目标校验、目标搜索、射程判断和 `fire(...)`；本轮必须在这里加入未完工攻击结构不能开火的通用边界。
- `nearestTarget(for:maxDistance:)` 已使用 `entity.kind.canAttack(target.kind)` 和 `isKnownToFaction(target, observer: entity.faction)`，Guard Tower 必须复用这条目标合法性链路。
- `fire(attacker:target:)` 已负责有效伤害、潜艇开火暴露、视觉弹道、护甲倍率、受击警报、AI 防守拉动、击杀 XP 和爆炸。
- `awardVeterancyKill(...)` 当前只给非结构作战单位授 XP；Guard Tower 击杀不应获得老兵 XP，除非后续另开版本。
- `rebuildEnemyBaseIfNeeded()` 使用 `buildableStructures` 查找敌方缺失结构并通过 `enemyConstructionPosition(for:)` 补建；Guard Tower 加入后 AI 会自然维护一个该类结构。
- `enemyProductionCount()` 只统计 War Factory、Airfield、Shipyard、Carrier；Guard Tower 不应计入“Break Red Production”阶段。
- `strategicValue(of:)`、`enemyStrategicAssaultScore(...)`、`primaryTarget(for:)` 和 `targetScore(for:target:)` 共同影响 AI 目标偏好。
- `singleSelectionInfo(for:)` 和 `structureStatusLine(for:)` 负责建筑选择信息；Guard Tower 需要显示防御价值，而不是只显示泛泛的 `Base coverage`。
- `restartSkirmish()` 会清空实体、队列、pending 模式、支援冷却、支援揭示、迷雾、任务状态和 AI 计时器，再重建地图与初始实体。

## 实现步骤

### 1. 扩展 `EntityKind`

在 `GameScene.swift` 的 `EntityKind` 中新增 `.guardTower`，并补齐所有 `switch`。

建议数值：

```text
displayName: Guard Tower
shortCode: GT
domain: structure
cost: 1450
buildTime: 11.0
maxHP: 820
speed: 0
attackRange: 205
damage: 30
attackCooldown: 1.25
visionTiles: 6
footprint: 44 或 46
requiredFactory: nil
canProduce: false
canAttack: true
isFactory: false
supportsRallyPoint: false
hasSonar: false
```

数值原则：

- 火力强于 Humvee、弱于 Tank/Artillery 的持续压制组合。
- 射程略高于 HQ 或接近 HQ，但低于 Artillery / Battleship，避免变成岸防炮。
- 视野只按普通防御建筑处理，低于 Radar Outpost，不能替代侦察建筑。
- HP 低于生产建筑或接近 Radar Outpost，不应成为第二个 HQ。

### 2. 定义攻击目标

在 `EntityKind.canAttack(_:)` 中为 `.guardTower` 增加分支。

要求：

- 可以攻击 `target.domain == .land`。
- 可以攻击 `target.domain == .air`。
- 可以攻击 `target.isStructure`。
- 不可以攻击 `target.domain == .naval`。
- 不可以攻击 `.submarine`。当前 `target == .submarine` 的特殊分支只允许 submarine / battleship / helicopter / fighter / carrier，Guard Tower 不应加入该白名单。

建议实现形态：

```swift
case .guardTower:
    return target.domain == .land || target.domain == .air || target.isStructure
```

不要新增独立塔防战斗系统，不要绕过 `canAttack(_:)`。

### 3. 接入玩家 `BASE` 建造轮换

- 将 `.guardTower` 加入 `buildableStructures`。
- 建议顺序：`[.barracks, .airfield, .radarOutpost, .guardTower, .shipyard, .oilDerrick]`。
- `subtitle(for: .buildBase)` 会自动显示 `$cost shortCode`，确认 `$1450 GT` 不溢出。
- pending 建造面板会自动显示成本、HP、建造时间和 domain；确认普通陆地结构文案准确。

Guard Tower 必须走普通陆地结构放置：

- 需要玩家可见建造点。
- 需要附近已完工友方基地覆盖。
- 不能建在水、山脊或已占用位置。
- 不需要油田。
- 不需要海岸。

### 4. 接入 AI 缺失建筑补建

- 让 Guard Tower 通过 `buildableStructures` 进入 `rebuildEnemyBaseIfNeeded()` 的缺失建筑补建链路。
- 不新增专门 AI 建造函数。
- 不把 Guard Tower 放入 `aiBuildPattern()`，因为它不是生产单位。
- 不把 Guard Tower 放入 `enemyProductionCount()`，因为它不是生产建筑。
- `enemyConstructionPosition(for:)` 对 Guard Tower 应自然走普通陆地结构搜索。
- 若 AI 资金不足，跳过本轮补建即可；不要引入复杂资金预留系统。

默认不在 `spawnInitialForces()` 给双方初始 Guard Tower。保持 v1.2 为局内建造/补建增量；只有发现 AI 长期无法建造时，才考虑给双方都加初始 Guard Tower，并在最终说明平衡理由。

### 5. 绘制 Guard Tower

在 `configureEntityNode(_:)` 和 `addStructureBody(for:to:)` 中接入 `.guardTower`。

建议用现有 SpriteKit shape 生成：

- 低矮矩形或圆角混凝土基座。
- 竖直塔身或装甲柱。
- 小炮塔/双联机枪/雷达灯样式的顶部，但不要画成 SAM 或岸防炮。
- 使用阵营色作为顶部灯、装甲条或小旗，保持 Blue / Red 区分。

要求：

- 不遮挡血条、短码、施工进度、选择框、老兵徽章和生产/集结节点。
- 作为实体节点子节点渲染，敌方 Guard Tower 在迷雾中必须随父节点隐藏。
- 不引入持续高频动画；如添加炮塔灯光，必须是静态 shape。
- 不新增外部素材。

### 6. 修正 `updateCombat` 施工边界

本轮必须修正通用战斗边界：未完工攻击型结构不能开火。

建议做法之一：

```swift
for entity in entities.values where entity.isAlive && entity.kind.damage > 0 {
    if entity.kind.isStructure && !entity.isOperational {
        entity.attackTarget = nil
        entity.attackTimer = 0
        continue
    }
    ...
}
```

或将循环过滤为只处理 operational 结构和移动单位，但要保持移动单位、HQ、Carrier、潜艇等旧行为不变。

验收重点：

- 未完工 Guard Tower 不搜索目标。
- 未完工 Guard Tower 不保留攻击目标。
- 未完工 Guard Tower 不播放弹道、不扣血、不触发受击警报或 AI 防守拉动。
- 任何未来新增 `damage > 0` 的未完工结构也不会开火。
- 已完工 HQ 和 Guard Tower 正常开火。
- 移动单位不受该结构边界影响。

### 7. 保持迷雾与施工效果边界

确认并保留现有边界：

- `updateFog(force:dt:)` 跳过未完工玩家结构，Guard Tower 未完工不提供视野。
- `hasNearbyStructure(...)` 只统计 operational 结构，Guard Tower 未完工不扩展基地覆盖。
- `incomePerTick(for:)` 不把 Guard Tower 计入收入。
- `hasOperationalSupportAsset(...)` 不把 Guard Tower 计入任一支援资产。
- `selectedPlayerRallyFactories()` 不允许 Guard Tower 设置 rally。

Guard Tower 完工后：

- 通过 `effectiveVisionTiles(for:)` 提供普通结构视野。
- 通过 `updateCombat(dt:)` 自动寻找可见合法目标。
- 通过 `fire(...)` 产生现有弹道、伤害、警报和死亡清理。

### 8. HUD 和选择信息

同步玩家可见信息：

- `singleSelectionInfo(for:)` 对未完工 Guard Tower 应显示 `Atk 0  Rng 0  Vis 0` 和剩余建造时间，这可复用现有结构分支。
- `structureStatusLine(for:)` 对完工 Guard Tower 不要只显示 `Base coverage`。
- 建议文案：
  - `Defense tower  Land/Air`
  - 或 `Auto-defense  No naval`
- 单选标题自动显示 `Guard Tower GT`。
- `selectedLabel` 可保持通用建筑 HP 文案。
- 小地图会因为 `entity.kind.isStructure` 自动显示结构 blip，不需要新增图例。

不要增加新的 HUD 按钮；通过 `BASE` 轮换即可。

### 9. AI 目标价值和战略行为

Guard Tower 是防御威胁，应成为 AI 会考虑摧毁的目标，但不应高于 HQ 或核心生产建筑。

建议：

- `strategicValue(of: .guardTower)` 设为 660 左右。
- 该值建议高于 Radar Outpost / Oil Derrick，低于 War Factory / Airfield / Shipyard / Carrier。
- `primaryTarget(for:)` 的 priorities 中加入 `.guardTower`。建议顺序：

```swift
[.hq, .oilDerrick, .shipyard, .airfield, .barracks, .guardTower, .radarOutpost, .carrier, .battleship]
```

也可以把 Guard Tower 放在 Radar Outpost 前后，但必须保持它不比 HQ 和生产建筑更优先。

- `enemyStrategicAssaultScore(...)` 对 Guard Tower 加小额防御威胁分，例如 `+110`，鼓励 AI 清理防御点。
- `targetScore(for:target:)` 已有 `if target.kind.canAttack(attacker.kind) { score += 190 }`，Guard Tower 完成 `canAttack` 后会自然成为移动单位的战术威胁目标；不要再额外加过高分数。

## 验收标准

功能标准：

- `BASE` 可以轮换到 `GT`，玩家能在合法可见、基地覆盖内的陆地位置放置 Guard Tower。
- Guard Tower 建造会扣钱、显示预览、施工脚手架/进度、血条和完工提示。
- Guard Tower 未完工时可被攻击，但不提供视野、不扩展基地覆盖、不设置 rally、不提供支援资产、不赚钱、不生产单位、不搜索目标、不攻击、不造成伤害。
- Guard Tower 完工后可自动攻击当前可见性规则允许的敌方 land / air / structure。
- Guard Tower 不攻击 naval 和 submarine，不提供 sonar，不改变潜艇隐身链路。
- AI 能按缺失建筑逻辑建造或重建 Guard Tower，且不会影响普通单位生产和任务阶段生产计数。
- Guard Tower 可被玩家和 AI 攻击，死亡后从实体、小地图、迷雾、目标评分和选择状态中清理。
- `SKRM` 后 Guard Tower、pending 建造、支援冷却、迷雾、AI 计时器和选择状态无残留。

架构标准：

- 不新增平行建造、战斗、迷雾、AI 或 HUD 系统。
- 玩家和 AI 共享同一套 `EntityKind`、`GameEntity`、施工、攻击、迷雾、AI 目标评分和死亡清理逻辑。
- `isOperational` 是 Guard Tower 开火、视野和基地覆盖的统一生效边界。
- `updateCombat(dt:)` 对所有未完工攻击型结构通用禁火，不只对 Guard Tower 特判。
- 战争迷雾仍以 `visibleTiles`、`exploredTiles`、`supportRevealTiles` 和 `isKnownToFaction(...)` 为边界。
- 潜艇 `revealedUntil`、声呐检测和开火暴露不被 Guard Tower 改动破坏。
- HQ 胜负、任务阶段、经济收入、支援资产和单位生产列表不变。

文档标准：

- `README.md` 只记录已实现、可验证的 Guard Tower 行为。
- `md/flow/flow.md` 和必要的 `md/flow/flowchart.md` 与当前真实代码一致。
- `md/test/test.md` 只有测试规范实际变化时才修改。
- `update_log.md` 不由 Agent B 预写未验收结果；Agent C 验收通过后再追加 v1.2 正式记录。

验证标准：

- 本地 `git diff --check` 通过。
- commit 在 `main` 上，commit subject 为 `v1.2: 引入防御炮塔`。
- `git push origin main` 成功触发 GitHub Actions。
- Agent C 能下载最新 `origin/main` run 的未加密 artifact，并核对 manifest / JUnit / xcodebuild.log / failure summary / `.xcresult`。
- 云端 workflow conclusion 成功，manifest 中 `buildOutcome=success`，才算 Swift 代码最低验证通过。

建议人工/模拟器可用时补充 Stage Regression：

- 通过 `BASE` 轮换并建造 Guard Tower，确认未完工不扩视野、不攻击，完工后才开火。
- 让敌方 land / air / structure 进入射程，确认 Guard Tower 自动攻击。
- 让 naval / submarine 进入射程，确认 Guard Tower 不攻击且不破坏潜艇隐身/声呐。
- 摧毁 Guard Tower，确认死亡清理、迷雾、小地图、选择面板和 AI 目标状态正常。
- 观察 AI 是否会补建 Guard Tower，并继续生产混合单位、占点和进攻。
- 检查 `HOLD`、`AMOV`、普通移动、生产队列、支援技能、Radar Outpost 视野和 `SKRM` 重开没有回归。

## 风险和禁止项

- 风险：直接把 Guard Tower 加入 `buildableStructures` 后，AI 可能在补建防御塔时占用资金。数值和顺序要保守，不要引入复杂 AI 经济系统。
- 风险：未完工 Guard Tower 因 `damage > 0` 被 `updateCombat` 处理。本轮必须修正通用禁火边界，不能只靠 Guard Tower 特判规避。
- 风险：`canAttack(_:)` 如果写得过宽，会让 Guard Tower 变成岸防炮或反潜塔。必须明确不攻击 naval / submarine。
- 风险：新增 `EntityKind` case 会触发多个 Swift `switch` 编译错误。必须系统性补齐所有相关分支。
- 风险：HUD 空间有限，`GT` 文案必须短，选择面板不能溢出。
- 风险：AI 目标分值过高会让 Guard Tower 比 HQ 或生产建筑更优先，影响任务推进。保持中等价值即可。
- 禁止新增外部依赖或素材。
- 禁止改动 Xcode project，除非编译必须且能说明原因。
- 禁止扩大到 SAM、防空专精、岸防炮、科技树、升级系统、任务阶段或经济改动。
- 禁止改变 HQ 胜负条件、老兵 XP 授予边界、支援技能资产、任务阶段和基础经济。
- 禁止绕过 `GameScene.swift` 当前状态流另建平行玩法系统。
- 禁止回滚、覆盖或混入用户/其他 Agent 的无关改动；归属不清时停止报告。
- 禁止伪造测试、隐藏失败命令、把未运行的本机 build 或未通过的云端结果说成已验证。
- 禁止使用非 `Altman-sam114` 的 GitHub 账号伪装完成 push、CI 或 artifact 验收。
- 禁止使用 `smalldata_test`、`develop`、`codeb/...`、候选分支或 PR 合并流；本轮固定使用 `main` 直推和云端结果包验收。
