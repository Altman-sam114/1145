## 背景

当前 `updateAI(dt:)` 在每个 AI 指挥周期中会先补建、生产，然后挑选 `freeCaptureUnits` 去占领油井和前线旗点，最后在满足规模时调用 `commandEnemyAttackers(enemyUnits)` 发起 attack-move 主攻波次。

现有代码有两个任务冲突风险：

1. `freeCaptureUnits` 是一次性过滤出来的数组。AI 给油井 runner 调用 `setDestination(...)` 后，旗点 runner 仍从旧的 `freeCaptureUnits` 中选择，因此同一个 Humvee / Mechanic 可能在同一周期先被派去油井，又立刻被改派去旗点。
2. `commandEnemyAttackers(_:)` 当前筛选 `idleAttackers` 时检查 `attackTarget == nil`、`attackMoveDestination == nil`、`damage > 0` 和 `isOperational`，但不检查 `destination == nil`。因此刚被派去油井或旗点、仍只有普通 `destination` 的 Humvee / Tank 等作战单位，可能马上被主攻波次接管并改成 attack-move。

v1.9 要做一个有限增量：AI 已经分配给占油 / 占旗任务的单位应保留当前目的地，不被同一轮重复分配，也不被主攻波次抢走。这样 Red 的地图控制会更稳定，更接近 Desert Stormfront 风格 RTS 的抢点推进节奏。

## 目标

1. AI 每轮给油井 runner 设置目的地后，旗点 runner 必须从重新计算后的“仍空闲占点单位”中选择，避免同一个单位同轮被油井和旗点任务复用。
2. `commandEnemyAttackers(_:)` 的 `idleAttackers` 必须排除 `destination != nil` 的单位，避免正在执行占油、占旗或其他普通移动任务的作战单位被 attack-move 主攻波次覆盖。
3. 保持真正 idle 的作战单位仍能参加主攻波次：
   - `attackTarget == nil`
   - `attackMoveDestination == nil`
   - `destination == nil`
   - `kind.damage > 0`
   - `isOperational`
4. 保持 AI 占点候选的现有范围和优先级：
   - 油井优先 Mechanic，其次 Humvee。
   - 旗点优先 Humvee / Mechanic / Tank。
   - 不扩大到空军、海军、建筑或非存活单位。
5. 不改变 AI 波次目标选择、`enemyWaveObjective()`、`issueFormationMove(...)`、路径寻路、AI 难度参数、生产或经济。
6. README 的 Enemy AI 描述要同步体现 Red 会保留已派出的占油 / 占旗队，不会被主攻波次立即抢走。
7. `md/flow/flow.md` 的 AI 流程要同步当前事实；必要时最小更新 `md/flow/flowchart.md` 的 AI 节点文字。
8. 不更新 `update_log.md`。正式 v1.9 记录由 Agent C 在最新 `origin/main` artifact 验收通过后追加。

### 2. 抽出或复用 AI 占点候选过滤

推荐在 AI 函数附近新增小 helper：

```swift
private func isAvailableEnemyCaptureUnit(_ unit: GameEntity) -> Bool {
    unit.faction == .enemy &&
    unit.isAlive &&
    !unit.kind.isStructure &&
    unit.attackTarget == nil &&
    unit.attackMoveDestination == nil &&
    unit.destination == nil &&
    unit.kind.domain == .land &&
    unit.isOperational
}
```

也可以不新增 helper，直接在 `updateAI(dt:)` 中重新过滤两次。但如果新增 helper，要求：

- 不保存持久 reservation 状态。
- 不扩大可占点单位域。
- 不引入会影响玩家单位的通用行为。

### 3. 避免同一轮油井 / 旗点复用同一个 runner

推荐把当前一次性的 `freeCaptureUnits` 替换为重新计算：

```swift
let availableOilCaptureUnits = enemyUnits.filter(isAvailableEnemyCaptureUnit)
if let neutralOil = ...,
   let oilRunner = availableOilCaptureUnits.first(where: { $0.kind == .mechanic }) ?? availableOilCaptureUnits.first(where: { $0.kind == .humvee }) {
    setDestination(for: oilRunner, near: neutralOil.node.position)
}

let availableFlagCaptureUnits = enemyUnits.filter(isAvailableEnemyCaptureUnit)
if let targetFlag = enemyControlPointTarget(),
   let flagRunner = availableFlagCaptureUnits.first(where: { $0.kind == .humvee || $0.kind == .mechanic || $0.kind == .tank }) {
    setDestination(for: flagRunner, near: targetFlag.node.position)
}
```

等价实现也可以，例如用 `reservedCaptureUnitIDs` 排除刚分配的 runner。但不要新增跨帧状态。

要求：

- 油井和旗点目标选择保持现有逻辑。
- 如果油井 runner 被设置 `destination`，它不应再被同一周期选为旗点 runner。
- 如果没有油井目标或没有可用油井 runner，旗点仍可使用空闲单位。
- 如果同一周期油井和旗点各有不同空闲单位，两个任务都应能分配。

### 4. 主攻波次排除正在执行普通 destination 的单位

修改 `commandEnemyAttackers(_:)` 的 `idleAttackers` 过滤，加上：

```swift
unit.destination == nil &&
```

要求：

- 只影响 AI 主攻波次筛选，不改变玩家 attack-move。
- 不改变 `enemyWaveObjective()`、波次大小、排序、目标评分或 `issueFormationMove(...)`。
- 持有 `attackTarget`、`attackMoveDestination` 或 `destination` 的单位都不应被当成 idle 波次单位。
- 真正 idle 的作战单位仍能按原逻辑发起 attack-move。

## 风险和禁止项

- 风险：如果主攻波次过度过滤单位，AI 攻击频率会下降；本轮只过滤已有普通 `destination` 的单位，保持真正 idle 作战单位可参战。
- 风险：如果新增持久 reservation 状态，可能在死亡、卡路、`SKRM` 后残留；本轮禁止新增跨帧 reservation。
- 风险：如果改变 `issueFormationMove(...)`，会影响玩家和 AI 共享命令链路；本轮应只改 AI 调用前筛选。
- 风险：如果扩大占点单位候选到空军或海军，会改变占领系统语义；本轮保持陆地单位候选。
- 禁止伪造测试结果、隐藏失败命令、把本地轻量检查说成云端构建通过。
- 禁止让 Agent C 只看 Agent B 汇报；必须核对最新 artifact。
- 禁止把旧 run、旧 artifact 或本地输出冒充本轮云端结果。
- 禁止使用非 `Altman-sam114` 的 GitHub 账号伪装完成 push、CI 或 artifact 验收。
