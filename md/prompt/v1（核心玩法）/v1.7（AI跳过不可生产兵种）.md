## 背景

当前 `GameScene.updateAI(dt:)` 每个 AI 指挥周期会先补建缺失生产建筑，再按 `aiBuildPattern()` 和 `aiBuildCursor` 尝试排生产队列。现有逻辑大致是：

```swift
let desired: [EntityKind] = aiBuildPattern()
for _ in 0..<aiDifficulty.buildOrdersPerCycle {
    let buildKind = desired[aiBuildCursor % desired.count]
    aiBuildCursor += 1
    _ = queueBuild(kind: buildKind, faction: .enemy, showFeedback: false)
}
```

问题是：如果当前 `aiBuildPattern()` 指向空军、海军或航母等单位，而 Red 当前没有 operational Airfield / Shipyard / Carrier，或资金不够，`queueBuild(...)` 会失败，但 cursor 仍会推进。这会让 AI 在早期、生产建筑施工中、生产建筑被打断后，连续轮空多个生产槽，削弱 Red 的恢复和前压节奏。

v1.7 要保持 AI 的混合兵种意图，但让每个生产尝试槽从当前 cursor 开始，向后扫描 `aiBuildPattern()`，选出第一个“当前可生产且资金足够”的兵种再排队。若整个 pattern 当前都不可生产，则本槽不排队，并保持现有经济、补建、支援、占点和攻击逻辑。

## 目标

1. 修改 AI 生产选择：每个 `aiDifficulty.buildOrdersPerCycle` 槽位不再只尝试 `aiBuildCursor` 当前指向的 kind，而是从当前 cursor 开始，最多扫描 `aiBuildPattern().count` 次，寻找第一个当前可生产且资金足够的单位 kind。
2. “当前可生产且资金足够”的判定必须复用或严格对齐 `queueBuild(kind:faction:showFeedback:)` / `productionSource(for:faction:)` 的真实条件：
   - kind 必须是可生产单位，即 `kind.requiredFactory != nil`。
   - 对应 faction 必须有可生产该 kind 的 operational 来源：War Factory / Airfield / Shipyard / Carrier，且来源 `isAlive`、`isOperational`，并通过 `entity.kind.canProduce(kind)`。
   - faction 当前 money 必须大于等于 `kind.cost`。
   - 未完工建筑不能生产，不能被 helper 当作可用来源。
3. 保留 `aiBuildPattern()` 的 Easy / Normal / Hard 混合兵种序列，不把 AI 永久改成只造陆军。
4. 保留 cursor 轮转：
   - 每个成功找到并尝试排队的槽位，应把 `aiBuildCursor` 推进到被选中 kind 的下一个 pattern 位置，避免 AI 一直卡在同一个可生产项。
   - 如果整轮扫描都没有可生产项，建议仍让该槽位有界地推进一个位置或保持现状，但必须在实现注释或最终汇报中说明选择。更推荐“扫描失败不排队，cursor 只推进 1”，避免下一周期永久卡在同一个不可生产起点，同时不会跳完整个混合序列。
5. AI 内部失败不要刷玩家 HUD message 或 denied marker。调用 `queueBuild(... showFeedback: false)` 即可。
6. README 的 AI 描述要同步体现 Red 会跳过当前不可生产的兵种并保持生产节奏。
7. `md/flow/flow.md` 的 AI 流程要同步当前事实；如流程图中 AI 节点描述需要最小更新，也同步 `md/flow/flowchart.md`。
8. 不更新 `update_log.md`。正式 v1.7 记录由 Agent C 在最新 `origin/main` artifact 验收通过后追加。

### 2. 新增 AI 可生产判定 helper

在 `GameScene.swift` 的生产 / AI 相关函数附近新增小 helper。推荐形态：

```swift
private func canQueueBuild(kind: EntityKind, faction: Faction) -> Bool {
    kind.requiredFactory != nil &&
    productionSource(for: kind, faction: faction) != nil &&
    money(for: faction) >= kind.cost
}
```

等价命名可以，例如 `isBuildKindAvailableForAI(_:)` 或 `isCurrentlyBuildable(kind:faction:)`，但要求：

- 不复制一套新的工厂类型 switch 来替代 `productionSource`。
- 不把未完工建筑当作可用来源。
- 不绕过 Carrier 作为 air source 的现有 `canProduce` 规则。
- 不改变 `productionSource` 本身的玩家 selected factory 优先逻辑，除非只是新增无副作用 overload 并保持玩家行为完全一致。

如果担心 helper 名称被误用于玩家 HUD，可用 AI 专属 helper 包裹：

```swift
private func isAvailableAIBuildKind(_ kind: EntityKind) -> Bool {
    canQueueBuild(kind: kind, faction: .enemy)
}
```

### 3. 新增从 pattern 扫描可生产兵种的 helper

推荐新增：

```swift
private func nextAvailableAIBuildKind(in pattern: [EntityKind], startingAt cursor: Int) -> (kind: EntityKind, nextCursor: Int)?
```

或：

```swift
private func availableAIBuildKind(from pattern: [EntityKind], startingAt startIndex: Int) -> EntityKind?
```

要求：

- 最多扫描 `pattern.count` 次，不能无限循环。
- 每次取 `pattern[(cursor + offset) % pattern.count]`。
- 对候选调用第 2 步 helper，确认当前有来源且资金足够。
- 找到候选后返回该 kind，并让 caller 能把 `aiBuildCursor` 更新到候选之后的位置：`cursor + offset + 1`。
- 如果整个 pattern 都没有候选，返回 nil。
- 要处理空 pattern，虽然当前 `aiBuildPattern()` 不为空。空 pattern 时直接 nil，避免未来修改造成除零。

示例结构，仅供参考，Agent B 可按现有 Swift 风格微调：

```swift
private func nextAvailableAIBuildKind(in pattern: [EntityKind], startingAt cursor: Int) -> (kind: EntityKind, nextCursor: Int)? {
    guard !pattern.isEmpty else { return nil }
    for offset in 0..<pattern.count {
        let candidateIndex = (cursor + offset) % pattern.count
        let candidate = pattern[candidateIndex]
        if canQueueBuild(kind: candidate, faction: .enemy) {
            return (candidate, cursor + offset + 1)
        }
    }
    return nil
}
```

注意：`nextCursor` 可以保留为未取模的递增整数，因为现有代码通过 `% desired.count` 使用 cursor；也可以取模，只要行为稳定并避免溢出。不要每帧重置 cursor。

### 4. 修改 `updateAI(dt:)` 的生产槽逻辑

替换当前直接取 `desired[aiBuildCursor % desired.count]` 的循环。

推荐逻辑：

```swift
let desired = aiBuildPattern()
for _ in 0..<aiDifficulty.buildOrdersPerCycle {
    guard let nextBuild = nextAvailableAIBuildKind(in: desired, startingAt: aiBuildCursor) else {
        if !desired.isEmpty {
            aiBuildCursor += 1
        }
        continue
    }
    aiBuildCursor = nextBuild.nextCursor
    _ = queueBuild(kind: nextBuild.kind, faction: .enemy, showFeedback: false)
}
```

要求：

- 不要调用 `queueBuild` 之前扣钱；扣钱仍由 `queueBuild` 负责。
- `queueBuild` 仍可能因状态变化失败；本轮不需要为单线程场景做复杂重试，但如果 helper 和 queueBuild 条件对齐，正常不应失败。
- 不要在 AI 内部失败时 `showMessage(...)` 或 `showDeniedMarker(...)`。
- 不要移除 mechanic 少于 2 时的补 mechanic 分支。该分支仍可先执行；如果没有 War Factory 或钱不够，它会安静失败。
- 保持 `rebuildEnemyBaseIfNeeded()` 在生产选择之前，以便 AI 仍优先补生产建筑。

## 验收标准

实现必须满足：

- `updateAI(dt:)` 每个 AI 常规生产槽会从 `aiBuildPattern()` 当前 cursor 开始向后扫描最多 `pattern.count` 次。
- 当当前 cursor 指向缺少 operational Airfield / Shipyard / Carrier 的 air/naval/carrier 单位时，AI 能跳过它们，选择后续当前有来源且资金足够的单位。
- 当 War Factory 可用、资金足够，但 Airfield / Shipyard 不可用时，AI 仍能持续排 Humvee / Tank / Artillery / Mechanic，而不是在 air/naval 项上轮空。
- 当 Airfield、Shipyard 或 Carrier 完工后，对应 air/naval/carrier 项仍会重新进入可选范围；AI 没有被永久降级为只造陆军。
- 当所有 pattern 项都因为没有 operational 来源或资金不足而不可生产时，AI 不排队、不扣钱、不显示玩家 HUD message、不刷 denied marker，并继续执行补建、占点、支援、攻击等现有逻辑。
- `queueBuild` / `productionSource` 仍是实际排队和来源选择入口；未完工建筑不会生产。
- 玩家生产行为、玩家 HUD、生产队列执行、rally、出兵、经济、战斗、迷雾、潜艇、老兵、任务、胜负、控制组和 `SKRM` 行为不变。

## 风险和禁止项

风险：

- 如果 helper 手写工厂类型判断，容易遗漏 Carrier 作为 air source，或错误允许未完工建筑生产。
- 如果 cursor 在扫描失败时完全不推进，AI 可能长期从同一个不可生产项开始扫描；如果每次失败跳完整个 pattern，又可能过快扰乱混合节奏。实现必须说明 cursor 失败策略。
- 如果可生产判定只看来源不看资金，仍会把资金不足的高价单位交给 `queueBuild` 失败，不能解决“生产槽轮空”问题。
- 如果成功扫描后 cursor 更新到错误位置，AI 可能重复生产同一个便宜单位，破坏 `aiBuildPattern()` 混合意图。
- 如果为了修复 AI 改动 `queueBuild`，可能影响玩家生产和 HUD feedback。本轮优先新增无副作用 helper，不改共享入口语义。

禁止项：

- 禁止不读必读文档和相关源码直接改代码。
- 禁止把 AI 生产改成固定优先级、只造陆军、只造最便宜单位或按随机数绕开 `aiBuildPattern()`。
- 禁止绕过 `queueBuild` 直接扣钱或直接 append `BuildOrder`。
- 禁止让未完工 War Factory / Airfield / Shipyard / Carrier 生产单位。
- 禁止给 AI 内部生产失败显示玩家 HUD message 或 denied marker。
- 禁止修改玩家生产、单位属性、AI 补建建筑、攻击波次、支援技能、经济、迷雾、战斗、胜负、老兵或 HUD 交互。
- 禁止把半成品或未验证功能写进 README。
- 禁止更新 `update_log.md`，该文件由 Agent C 验收通过后追加。
- 禁止新增外部依赖、素材、大体积 artifact 或无关测试数据。
- 禁止伪造测试结果、隐藏失败命令或把环境失败说成源码失败。
