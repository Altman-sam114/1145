# v1.0（实现老兵经验系统）

## 版本号

v1.0

## 版本分配依据

- `update_log.md` 当前最新流程版本是 v0.3，业务原型已经覆盖经济、建造、陆海空、海军、航母、AI、战争迷雾、HUD、支援技能、任务阶段和 HQ 胜负。
- `update_log.md`、`md/flow/flow.md` 和 `md/prompt/README.md` 都明确把老兵/经验系统列为下一轮核心玩法扩展点。
- `GameEntity` 已有 `veterancyXP`、`killCount`、`veterancyNode` 字段，但未接入击杀结算、等级、加成、HUD 或 README。该轮属于第一个核心玩法正式版本，使用 `v1.0`。

## 背景

`Desert Frontline` 现在已经是可玩的 iOS SwiftUI + SpriteKit RTS 原型。玩家和 AI 共享生产、移动、攻击、支援技能、迷雾、任务和胜负循环，核心逻辑集中在 `DesertFrontline/GameScene.swift`。当前战斗只有静态 `EntityKind.damage`、`attackCooldown`、`visionTiles` 等数值，单位击杀后不会留下成长反馈。

下一步优化目标是把已经预留的老兵字段接入当前战斗主循环，让长期存活并完成击杀的作战单位获得等级、轻量战斗加成、头顶徽章和选择面板信息。该目标能增强 RTS 单局内的连续性，又不会引入新资源、外部依赖或大范围架构迁移。

## 目标

1. 为玩家和 AI 的作战单位共同实现老兵/经验系统。
2. 击杀结算时给击杀者增加 `killCount` 和 `veterancyXP`。
3. 根据 XP 计算等级，至少包含 Recruit / Hardened / Veteran / Elite 四档。
4. 等级提供轻量、可控的战斗加成，建议只影响：
   - 有效伤害。
   - 攻击冷却。
   - 视野格数。
5. 给获得等级的单位显示头顶老兵徽章，玩家可见敌军时也能看到敌军徽章。
6. 在单选和多选信息面板中显示老兵等级、XP、击杀数和简要等级分布。
7. 保持玩家与 AI 完全共用同一套成长、加成和渲染逻辑。
8. 保持 `SKRM` 重开后老兵状态随实体清空，不泄漏到下一局。
9. 更新 README 和核心流程文档，只写已经实现的行为。

## 非目标

- 不新增科技树、升级按钮、资源消耗型升级或永久战役存档。
- 不新增外部依赖、图片素材、音频素材或独立 XCTest target。
- 不拆分 `GameScene.swift`，不做大规模重构。
- 不改变单位生产列表、建筑体系、任务阶段或 HQ 胜负条件。
- 不把支援技能击杀默认计入任何单位 XP；本轮只做实体直接开火击杀的老兵成长。
- 不把半成品写入 README；只有实现并验证后才能加入 Current Features。

## 实现步骤

### 1. 增加等级模型

在 `GameScene.swift` 靠近 `EntityKind` / `SupportPower` / `MissionStage` 的模型区新增一个私有等级枚举或轻量结构，例如 `VeterancyRank`。

建议等级：

- Recruit：0 XP，默认，无徽章。
- Hardened：80 XP。
- Veteran：200 XP。
- Elite：420 XP。

建议属性：

- `displayName`
- `shortCode`
- `nextThreshold`
- `damageMultiplier`
- `cooldownMultiplier`
- `visionBonus`

建议平衡：

- Recruit：伤害 1.00，冷却 1.00，视野 +0。
- Hardened：伤害 1.10，冷却 0.96，视野 +0。
- Veteran：伤害 1.18，冷却 0.92，视野 +1。
- Elite：伤害 1.28，冷却 0.86，视野 +1。

加成必须轻量，避免一两个老兵单位破坏现有 AI 和任务节奏。

### 2. 在 `GameEntity` 暴露当前等级

给 `GameEntity` 增加只读计算属性或通过 `GameScene` helper 计算：

- 当前 `veterancyRank`。
- 下一级所需 XP。
- 是否有非 Recruit 等级。

不要把等级存在成第二份可变状态，避免 XP 和等级不同步。

### 3. 接入战斗数值

新增 helper，集中计算有效战斗数据：

- `effectiveDamage(for attacker: GameEntity) -> CGFloat`
- `effectiveAttackCooldown(for attacker: GameEntity) -> TimeInterval`
- `effectiveVisionTiles(for entity: GameEntity) -> Int`

替换以下位置：

- `updateCombat(dt:)` 设置 `attackTimer` 时使用 `effectiveAttackCooldown(for:)`。
- `fire(attacker:target:)` 扣血时使用 `effectiveDamage(for:)`，保留现有护甲倍率逻辑。
- `updateFog(force:dt:)` 计算玩家实体视野半径时使用 `effectiveVisionTiles(for:)`。
- `singleSelectionInfo(for:)` 显示有效伤害、基础射程、有效视野。

不要改变 `EntityKind.canAttack(_:)`、地形规则、路径规则或攻击合法性。

### 4. 接入击杀 XP

在 `fire(attacker:target:)` 中处理直接开火击杀：

1. 扣血前记录 `wasAlive = target.isAlive`。
2. 扣血后如果 `wasAlive && target.hp <= 0`，调用 `awardVeterancyKill(to:attacker,target:)`。
3. `awardVeterancyKill` 只允许可移动、可攻击、非建筑单位获得 XP：
   - `!attacker.kind.isStructure`
   - `attacker.kind.damage > 0`
   - `attacker.faction != target.faction`
   - `target.faction != .neutral`
4. 递增 `attacker.killCount`。
5. 按目标价值增加 XP。

建议 XP：

```text
基础 XP = max(24, target.kind.cost * 0.12)
若目标是建筑，基础 XP *= 0.65
若目标是 HQ，XP 可封顶到 180，避免最后一击直接造成过度跳级
```

如果击杀导致等级提升：

- 更新老兵徽章。
- 如果击杀者是玩家单位，显示简短 `showMessage`，例如 `Tank promoted to Veteran.`。
- 如果击杀者是敌军但当前对玩家可见，可以只更新徽章，不强制弹全局消息，避免 HUD 噪声。

支援技能 `applySupportDamage(...)` 击杀本轮不授予单位 XP；不要为支援伤害增加“虚拟击杀者”。

### 5. 添加头顶老兵徽章

在 `configureEntityNode(_:)` 中把 `entity.veterancyNode` 加入 `entity.node`，设置合适 zPosition 和位置，建议靠近血条上方，不遮挡单位短码和生产/施工进度。

新增 `updateVeterancyBadge(for:)`：

- Recruit 时 `removeAllChildren()` 或隐藏。
- Hardened / Veteran / Elite 显示不同数量的小条、小星或短码标签。
- 使用 SpriteKit shape / label 生成，不新增素材。
- 颜色应与现有 HUD/单位风格相容，例如铜、银、金。
- 敌方节点被迷雾隐藏时，徽章随父节点隐藏；不要绕过 `isKnownToFaction(...)`。

在以下时机调用：

- `configureEntityNode(_:)` 初始化后。
- `awardVeterancyKill(...)` 等级或 XP 改变后。
- 如有必要，在 `setFaction(_:,for:)` 后保持徽章颜色/可见性一致。

### 6. 更新 HUD 信息面板

单选 `singleSelectionInfo(for:)`：

- 对非建筑单位显示老兵等级、XP 和击杀数。
- 当前第四行已有状态用途，注意不要覆盖潜艇 detected、Mechanic repair、Carrier queue、HOLD、AMOV 等重要状态。可采用优先级：
  - 行 1：HP。
  - 行 2：有效 Atk / Rng / Vis。
  - 行 3：移动/domain 或建筑状态。
  - 行 4：老兵信息；若当前单位有特殊状态，则把老兵信息压缩进行 2 或行 3，或在不破坏现有状态的前提下调整文本。

多选 `groupSelectionInfo(for:)`：

- 保留 Land / Air / Sea、HP、Dmg / Max rng。
- 第四行可以在无 HOLD/AMOV 时显示等级分布，例如 `Vet H2 V1 E0  Kills 5`。
- 组内总伤害应使用有效伤害，而不是静态 `kind.damage`。

空选、建筑建造、支援技能、任务面板不需要新增老兵说明。

### 7. 保持 AI、迷雾和胜负边界

- AI 单位必须通过同一个 `fire`、`awardVeterancyKill`、`effectiveDamage`、`effectiveAttackCooldown` 获得成长。
- 玩家不能因为老兵系统攻击迷雾里不可见敌人。
- 潜艇 `revealedUntil`、声呐检测、开火暴露必须保持原行为。
- HQ 摧毁胜负逻辑不变。
- `SKRM` 重新开始必须清空所有老兵状态；因为实体重建即可，但要实际检查没有单独缓存残留。

## 验收标准

功能标准：

- 玩家和 AI 可移动作战单位都能通过直接击杀获得 XP 和 killCount。
- XP 达到阈值后等级变化生效，并影响有效伤害、攻击冷却和玩家视野。
- 头顶徽章在非 Recruit 单位上显示，且不绕过战争迷雾。
- 单选和多选面板能读出老兵状态。
- 支援技能击杀不会错误计入某个单位。
- 重开 skirmish 后没有老兵状态残留。

架构标准：

- 不新增平行战斗系统。
- 不绕过 `GameScene.swift` 现有 update / fire / cull / fog / HUD 数据流。
- 不破坏 `isOperational` 建筑边界。
- 不破坏潜艇 `revealedUntil` 与声呐检测。
- 不破坏玩家对迷雾中敌人的交互限制。
- 玩家和 AI 使用同一套 helper 和结算逻辑。

验证标准：

- 本地 `git diff --check` 通过。
- commit 在 `main` 上，commit subject 为 `v1.0: 实现老兵经验系统`。
- `git push origin main` 成功触发 GitHub Actions。
- Agent C 能下载最新 `origin/main` run 的未加密 artifact，并核对 manifest / JUnit / xcodebuild.log / failure summary。
- 云端 `buildOutcome=success` 且 workflow conclusion 成功。

## 风险和禁止项

- 风险：老兵加成过强会让存活单位滚雪球。加成必须轻量，初版优先选择伤害、冷却、视野，不改最大 HP。
- 风险：死亡清理在 `cullDestroyedEntities()` 中统一执行，击杀授 XP 必须在目标还可被识别时完成，避免 cull 后找不到 target。
- 风险：支援技能、AI 防守拉扯和多单位同帧开火可能造成重复授 XP。授 XP 时必须用扣血前后的 alive 状态判断，避免重复。
- 风险：HUD 行数有限，新增老兵文本不能遮挡生产、潜艇、HOLD、AMOV、集结点等已有关键状态。
- 风险：敌军徽章不能通过单独节点绕过迷雾显示，必须作为实体节点子节点或同步 `isKnownToFaction(...)`。
- 禁止新增外部依赖或素材。
- 禁止擅自改变单位列表、经济、AI 难度、任务阶段或胜负规则。
- 禁止把本机未跑或云端未过的结果写成已验证。
- 禁止使用 `smalldata_test`、`develop`、`codeb/...`、候选分支或 PR 合并流；本轮只使用 `main` 直推和云端结果包验收。
