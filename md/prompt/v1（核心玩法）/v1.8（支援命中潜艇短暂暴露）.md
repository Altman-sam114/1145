## 背景

当前潜艇隐身链路已经存在：

- `GameEntity.revealedUntil` 记录潜艇被临时暴露的截止时间。
- `fire(attacker:target:)` 中潜艇开火会把自身 `revealedUntil` 延长到 `lastUpdateTime + 2.4`，并在玩家可感知时显示 sonar ping。
- `revealSupportTargets(center:radius:duration:from:)` 当前由 `SCAN` 调用，会把范围内敌方潜艇暴露 8 秒。
- `isKnownToFaction(_:observer:)` 对潜艇调用 `isSubmarineDetected(_:by:)`，后者只在 `revealedUntil` 未过期或 observer 有声呐单位在距离内时返回 true。
- `AIRS` / `BARR` 的伤害走 `applySupportDamage(_:at:from:)`，当前能伤害范围内潜艇，但命中潜艇不会短暂暴露。结果是玩家或 AI 用支援打中潜艇时，潜艇可能扣血甚至死亡，但存活潜艇仍立即回到不可见状态，反馈和反潜追击都偏弱。

v1.8 要做一个有限增量：当 `AIRS` 或 `BARR` 的真实支援伤害命中存活潜艇时，短暂设置该潜艇的 `revealedUntil`，让命中有战术反馈，但不替代 `SCAN` 和声呐侦测。

## 目标

1. 让 `applySupportDamage(_:at:from:)` 在 `power.damage > 0` 且目标实体是敌方存活潜艇、实际落入伤害半径并扣血时，短暂延长该潜艇的 `revealedUntil`。
2. 推荐暴露时长为 `lastUpdateTime + 2.4` 秒，与潜艇开火暴露时长对齐；如 Agent B 选择 2.0 到 2.4 秒之间的值，必须说明理由。
3. 支援命中暴露必须只发生在真实命中并扣血之后：
   - 不能在目标未落入半径时暴露。
   - 不能在 `power.damage == 0` 的 `SCAN` / `REPR` 路径中重复处理。
   - 不能因为支援区域视觉、`revealSupportArea` 或 explored tile 而暴露未命中的潜艇。
4. 玩家和 AI 共享该规则：玩家 `AIRS` / `BARR` 命中 Red 潜艇会暴露；AI 支援命中 Blue 潜艇也会设置 `revealedUntil`，供 Red 自己的 `isKnownToFaction(..., observer: .enemy)` 和声呐链路使用。
5. 玩家可见反馈要遵守迷雾边界：
   - 依靠现有支援特效、短暂区域 reveal、血条变化和后续 `updateFog(force: true)` 提供反馈。
   - 本轮不新增 `showSonarPing(...)` 调用，避免把支援命中误表现为声呐扫描或在边界情况下泄露位置。
   - 如果是 AI 支援命中玩家潜艇，玩家本来就知道己方单位，允许保留现有支援效果和敌方来袭提示，但不要新增泄露 Red 视野的信息。
   - 不要把不可见、未命中的潜艇通过 HUD 文案或全局消息泄露给玩家。
6. 保持 `SCAN` 8 秒暴露逻辑不变，`SCAN` 仍是主动侦察技能；`AIRS` / `BARR` 只是命中后的短暴露。
7. 保持潜艇声呐链路不变：`isKnownToFaction`、`isSubmarineDetected`、`sonarRange`、`EntityKind.hasSonar` 不能被绕过或重写。
8. 保持支援技能费用、冷却、资产需求、半径、伤害、repair、支援视觉效果和胜负结算不变。
9. 更新 README 和 `md/flow/flow.md`，同步当前真实行为。必要时最小更新 `md/flow/flowchart.md` 的“战斗/迷雾”节点描述。
10. 不更新 `update_log.md`。正式 v1.8 记录由 Agent C 在最新 `origin/main` artifact 验收通过后追加。

  - `.reconSweep` 调用 `revealSupportTargets(... duration: 8.0 ...)`，玩家还会调用 `revealSupportArea(...)`。
  - `.fieldRepair` 调用 `applySupportRepair(...)`。

- `isKnownToFaction(_:, observer:)` 对玩家会先要求目标所在 tile 可见；对潜艇再调用 `isSubmarineDetected(...)`。
- `isSubmarineDetected(...)` 在 `submarine.revealedUntil > lastUpdateTime` 或 observer 阵营有声呐单位在 `sonarRange(...)` 内时返回 true。
- `showSonarPing(at:faction:)` 已有玩家/敌方颜色区分，但本轮不新增调用，避免支援命中和声呐扫描反馈混淆。

### 2. 新增小型 helper，集中处理支援命中潜艇暴露

推荐在支援技能函数附近新增 helper：

```swift
private func revealSubmarineHitBySupport(_ submarine: GameEntity) {
    guard submarine.kind == .submarine else { return }
    submarine.revealedUntil = max(submarine.revealedUntil, lastUpdateTime + 2.4)
    if submarine.faction == .enemy {
        submarine.node.isHidden = !isKnownToFaction(submarine, observer: .player)
    }
}
```

Agent B 可调整函数名或细节，但必须满足：

- 只处理潜艇实体。
- 使用 `max(...)`，不要缩短已有 `SCAN` 或声呐导致的更长暴露。
- 不直接改 `visibleTiles` 或 `exploredTiles`。
- 不绕过 `isKnownToFaction(...)` 长期显示敌方节点。
- 不调用 `showSonarPing(...)`。本轮视觉反馈依赖既有支援特效和 fog 刷新，后续若要专门做“命中潜艇提示”可另开小版本。

### 3. 在 `applySupportDamage(...)` 中只对真实命中的潜艇调用 helper

在距离检查、falloff 计算和扣血之后加入：

```swift
if entity.kind == .submarine {
    revealSubmarineHitBySupport(entity)
}
```

要求：

- 必须放在 `guard distance <= ...` 之后，确保未命中不暴露。
- 推荐放在扣血和 `updateHealthBar(entity)` 后、死亡判断前；这样“命中并造成伤害”的语义清楚。
- 不要在 `executeSupportPower(...)` 里按区域直接调用 `revealSupportTargets(...)` 来实现，否则会混淆 `SCAN` 和打击命中。
- 不要把 `revealSupportTargets(...)` 改成通用支援命中暴露，避免 `SCAN` 语义被污染。

### 4. 确认可见性和死亡清理顺序

检查 `.airStrike` / `.navalBarrage` 分支现有顺序：

```swift
applySupportDamage(...)
showSupportPowerEffect(...)

updateVictoryState()
...
updateFog(force: true)

```

本轮不需要改变这个顺序，除非编译或显示必须。重点确认：

- 存活潜艇在 `applySupportDamage` 后能依赖 `revealedUntil` 被 `updateFog(force: true)` 重新评估显示。
- 被支援打死的潜艇仍走现有 `explode(...)` 和 `cullDestroyedEntities()` 清理。
- 支援击杀仍不调用 `awardVeterancyKill(...)`。
