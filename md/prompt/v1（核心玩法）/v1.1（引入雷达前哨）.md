# v1.1（引入雷达前哨）

## 版本号

v1.1

## 版本分配依据

- `update_log.md` 当前最新核心玩法版本是 v1.0，已完成老兵经验系统，并通过 `origin/main` 最新 GitHub Actions artifact 验收。
- 本轮是在 v1.0 之后的小型核心玩法增量，目标是补强建筑科技层、基地侦察价值和支援技能资产，不进入完整科技树或大范围 AI 改造。
- `md/flow/flow.md` 的未来扩展点已经列出“建筑科技层：雷达、科技中心、防御塔、SAM、岸防炮、升级解锁”。本轮只取其中最小可验收切片：Radar Outpost。

## 背景

`Desert Frontline` 当前已经覆盖经济、建造、陆海空单位、海军、航母、AI、战争迷雾、支援技能、HUD、任务阶段、HQ 胜负和老兵经验。玩家和 AI 共享 `GameScene.swift` 中的建造、生产、战斗、迷雾和支援技能链路。

当前可建造结构只有 War Factory、Airfield、Shipyard 和 Oil Derrick。支援技能 `Recon Sweep` 只要求 operational HQ，基地科技层还不明显。新增一个可建造、可攻击、需要施工完成才生效的 Radar Outpost，可以在不引入完整科技树的前提下让玩家/AI拥有更清晰的侦察建筑目标，并给迷雾与支援技能资产提供下一步扩展基础。

## 目标

1. 新增一个结构建筑 `EntityKind.radarOutpost`，建议显示名 `Radar Outpost`，短码 `RAD`。
2. Radar Outpost 可由玩家通过 `BASE` 轮换建造，并走现有建筑预览、资金扣除、施工、血条、被攻击和死亡清理流程。
3. Radar Outpost 必须是普通陆地结构：
   - 需要玩家可见建造点。
   - 需要附近已完工友方基地覆盖。
   - 不能建在水上、山脊或已占用位置。
   - 不复用 Oil Derrick 的油田/占领特殊逻辑。
   - 不复用 Shipyard 的海岸特殊逻辑。
4. Radar Outpost 施工未完成前不得提供视野、支援技能资产或基地覆盖；只有 `isOperational` 后才生效。
5. Radar Outpost 提供较大的静态侦察视野，建议 `visionTiles = 9` 或 `10`，高于普通生产建筑但低于全图侦察。
6. Radar Outpost 可作为 `Recon Sweep` 资产：玩家或 AI 拥有 operational HQ 或 operational Radar Outpost 时可使用 `SCAN`。
7. 本轮倾向不降低 `SCAN` 成本或冷却。若 Agent B 认为必须给 Radar Outpost 体现轻量加成，只能选择非常保守、易测的行为，例如把 `supportAssetLine` 文案改为 `Needs HQ or RAD`；不要做动态冷却/成本折扣。
8. AI 必须能通过现有缺失建筑补建逻辑建造/重建 Radar Outpost，且不会因其不可生产单位而破坏生产队列。
9. HUD、选择信息、小地图、战略目标分值和文档必须反映已完成的 Radar Outpost 当前行为。
10. `SKRM` 重开后 Radar Outpost、支援冷却、迷雾、pending 建造和 AI 状态必须随现有重开链路重置。

## 非目标

- 不新增完整科技树、升级按钮、科技解锁、永久升级或战役持久化。
- 不新增防御塔、SAM、岸防炮、科技中心或新单位。
- 不改变 HQ 胜负条件、任务阶段数量或主要经济规则。
- 不让 Radar Outpost 生产单位、设置 rally point、攻击敌人、赚钱或占领油井/旗点。
- 不新增外部依赖、图片素材、音频素材、XCTest target 或大体积测试数据。
- 不拆分 `GameScene.swift`，不做大规模重构，不做全文件格式化。
- 不绕过战争迷雾：Radar Outpost 只能通过现有 `visibleTiles` / `exploredTiles` / `isKnownToFaction(...)` 链路产生侦察价值。
- 不默认跑本机完整 Xcode build；除非人工明确要求，本轮本地只跑轻量检查，构建以 GitHub Actions artifact 为准。

关键源码事实：

- `EntityKind` 集中定义单位/建筑名称、短码、domain、成本、建造时间、HP、攻击、视野、占地、生产来源、可生产项和可攻击目标。
- `GameEntity.isOperational` 由 `!kind.isStructure || buildProgressRemaining <= 0` 决定；未完工建筑已可被攻击，但不能生产、赚钱或扩展基地覆盖。
- 当前 `buildableStructures` 是 `[.barracks, .airfield, .shipyard, .oilDerrick]`，`BASE` 按该数组轮换 pending 结构。
- 玩家建造走 `handleHudAction(.buildBase)` -> `pendingConstructionKind` -> `updateConstructionPreview(...)` -> `placeStructure(...)` -> `startConstruction(...)`。
- `constructionIssue(...)` 已统一检查玩家视野、附近已完工友方结构、油井油田规则、Shipyard 海岸规则、普通陆地结构和占地。
- `hasNearbyStructure(...)` 只统计 `entity.kind.isStructure && entity.isOperational` 的友方建筑；Radar Outpost 完工后会自然扩展基地覆盖，未完工不会。
- `updateFog(force:dt:)` 当前对所有玩家存活实体使用 `effectiveVisionTiles(for:)` 加入 `visibleTiles`，但没有排除未完工结构；本轮必须修正为未完工结构不提供视野。
- `SupportPower.reconSweep` 当前 `supportIssue(...)` 文案是 `Recon requires an operational HQ.`，`hasOperationalSupportAsset(...)` 只允许 `.hq` 作为 Recon 资产。
- `supportAssetLine(for:)` 当前对 Recon 显示 `Needs HQ`。
- `rebuildEnemyBaseIfNeeded()` 使用 `buildableStructures` 检查敌方缺失建筑并通过 `enemyConstructionPosition(for:)` 补建。
- `enemyConstructionPosition(for:)` 对非油井结构从敌方 operational 结构附近搜索合法建造点。
- `strategicValue(of:)`、`enemyStrategicAssaultScore(...)`、`primaryTarget(for:)`、`targetScore(...)` 共同影响 AI 选择目标。
- `singleSelectionInfo(for:)` 和 `structureStatusLine(for:)` 负责建筑选择信息；Radar Outpost 需要显示侦察/支援价值，而不是默认只写 `Base coverage`。
- `restartSkirmish()` 会清空 `entities`、`buildOrders`、`pendingConstructionKind`、`pendingSupportPower`、`supportCooldowns`、`supportRevealTiles`、`visibleTiles` 和 `exploredTiles`，再重建地图与初始实体。

## 实现步骤

### 1. 扩展 `EntityKind`

在 `GameScene.swift` 的 `EntityKind` 中新增 `.radarOutpost`。

建议数值：

```text
displayName: Radar Outpost
shortCode: RAD
domain: structure
cost: 1300
buildTime: 10.0
maxHP: 720
speed: 0
attackRange: 0
damage: 0
attackCooldown: 1.0
visionTiles: 9 或 10
footprint: 44 或 46
requiredFactory: nil
canProduce: false
canAttack: false
isFactory: false
supportsRallyPoint: false
```

注意同步所有 `switch`，尤其是 `displayName`、`shortCode`、`domain`、`cost`、`buildTime`、`maxHP`、`visionTiles`、`footprint`、`strategicValue(of:)`、结构绘制和目标优先级相关列表。Swift `switch` 必须穷尽，不能留下编译错误。

### 2. 接入玩家 BASE 轮换和 AI 补建

- 将 `.radarOutpost` 加入 `buildableStructures`，建议顺序为 `[.barracks, .airfield, .radarOutpost, .shipyard, .oilDerrick]`，让玩家能通过 `BASE` 轮换到 `RAD`。
- AI 的 `rebuildEnemyBaseIfNeeded()` 会因为 `buildableStructures` 自动尝试补建缺失结构，但 Agent B 必须确认：
  - Radar Outpost 不进入 `queueBuild(...)` 生产链路。
  - Radar Outpost 不需要 `requiredFactory`。
  - `enemyConstructionPosition(for:)` 对 Radar Outpost 走普通陆地结构搜索。
  - AI 资金不足时只是跳过，不会阻塞后续生产。
- 不要求在 `spawnInitialForces()` 给双方初始 Radar Outpost；初版可以让玩家/AI在局内建造。若 Agent B 判断 AI 第一轮补建节奏过慢，可以给敌方初始 Radar Outpost，但必须同时给玩家初始 Radar Outpost，并说明平衡理由。默认建议不加初始建筑，保持小增量。

### 3. 绘制 Radar Outpost

在 `configureEntityNode(_:)` / `addStructureBody(for:to:)` 中接入 `.radarOutpost`。

建议使用现有 SpriteKit shape 生成，不新增素材：

- 小型矩形或圆角基座。
- 细塔架或天线杆。
- 雷达盘/半圆 dish 或扫掠线。
- 使用阵营色作为小面板或天线灯，保持 Blue / Red 区分。

要求：

- 不遮挡血条、短码、施工进度、选择框和老兵徽章。
- 作为实体节点子节点渲染，敌方 Radar Outpost 被迷雾隐藏时随父节点隐藏。
- 不引入持续高频动画；若加轻量扫掠动画，必须不影响性能且不依赖外部资源。

### 4. 修正建筑视野生效边界

当前 `updateFog(force:dt:)` 会遍历玩家所有存活实体并使用 `effectiveVisionTiles(for:)` 提供视野。本轮必须确保未完工结构不提供视野。

建议做法：

- 在 `updateFog(force:dt:)` 的玩家实体循环里跳过 `entity.kind.isStructure && !entity.isOperational`。
- 保留已完工 Radar Outpost、HQ、War Factory、Airfield、Shipyard、Oil Derrick 和移动单位的视野。
- 不改变控制点视野和 `supportRevealTiles`。
- 确认 `placeStructure(...)` 中创建未完工建筑后调用 `updateFog(force: true)` 不会因为新未完工结构直接扩展可见区域。
- `updateConstruction(dt:)` 已在玩家建筑完工时触发 `updateFog(force: true)`；Radar Outpost 完工后应自然刷新视野。

该修正符合既有架构规则“未完工建筑不得生产、赚钱或扩展基地覆盖”，并将视野也纳入同一施工边界。

### 5. 接入 Recon Sweep 资产

让 Radar Outpost 成为 `SCAN` 的 operational 支援资产。

需要同步：

- `hasOperationalSupportAsset(for: .reconSweep, faction:)`：允许 `.hq` 或 `.radarOutpost`。
- `supportIssue(for: .reconSweep, faction:)` 文案：改为类似 `Recon requires an operational HQ or Radar Outpost.`
- `supportAssetLine(for: .reconSweep)` 文案：改为 `Needs HQ or RAD`。
- 选择/目标面板中 pending `SCAN` 的资产说明应自动更新。

禁止：

- 不做动态 `SCAN` 冷却缩短、费用降低、半径扩大或多雷达叠加。
- 不让未完工 Radar Outpost 满足支援资产；必须依赖 `entity.isOperational`。

### 6. HUD 和选择信息

同步玩家可见信息：

- `selectionInfoContent(for:)` 的 pending 建造面板会自动显示 cost / HP / buildTime / domain；确认 Radar Outpost 的第 3/4 行文案适合普通陆地结构。
- `singleSelectionInfo(for:)` / `structureStatusLine(for:)` 对 Radar Outpost 不要只显示泛泛的 `Base coverage`。建议显示：
  - `Radar vision 9  SCAN asset`
  - 或 `Recon asset  Base coverage`
- `selectedLabel` 可保持通用建筑 HP 文案，不强制特殊化。
- `subtitle(for: .buildBase)` 会随着 `shortCode` 显示 `$1300 RAD`，确认按钮文本不溢出。
- 小地图会因为 `entity.kind.isStructure` 自动显示结构 blip；无需新增图例。

### 7. AI 目标价值和战略行为

Radar Outpost 是侦察/支援资产，应成为可攻击、有价值但不高于核心生产建筑的目标。

建议：

- `strategicValue(of: .radarOutpost)` 设为 620 左右，高于 Humvee/Mechanic，低于 War Factory/Airfield/Shipyard。
- `primaryTarget(for:)` 的 priorities 中加入 `.radarOutpost`，建议放在 `.oilDerrick` 之后、生产建筑之前或生产建筑之后均可，但必须说明理由。保守建议：`[.hq, .oilDerrick, .radarOutpost, .shipyard, .airfield, .barracks, .carrier, .battleship]`。
- `enemyStrategicAssaultScore(...)` 可对 Radar Outpost 加小额分数，例如 +90，鼓励 AI 在可见时打掉玩家侦察资产，但不要高于 HQ/生产建筑。
- 不改 `aiBuildPattern()`，因为 Radar Outpost 不是单位生产项。

  - Current Features 增加已完成的 Radar Outpost 行为：可建造、需施工完成才提供侦察和 `SCAN` 资产、可被攻击。
  - Controls 中 `BASE` 可建造结构列表加入 Radar Outpost。
  - 不写完整科技树、折扣、升级或未实现能力。
- `md/flow/flow.md`

  - 更新“建筑只有 isOperational 后才能生产、赚钱或提供基地覆盖”相关语句，补充未完工结构也不提供视野/支援资产。
  - 更新支援技能资产说明：Recon 可由 HQ 或 Radar Outpost 提供。

## 验收标准

功能标准：

- `BASE` 可以轮换到 `RAD`，玩家能在合法可见、基地覆盖内的陆地位置放置 Radar Outpost。
- Radar Outpost 建造会扣钱、显示预览、施工脚手架/进度、血条和完工提示。
- Radar Outpost 未完工时可被攻击，但不提供视野、不满足 `SCAN` 资产、不扩展基地覆盖。
- Radar Outpost 完工后提供较大静态视野，并可作为 `SCAN` 资产。
- 玩家没有 operational HQ 或 operational Radar Outpost 时不能使用 `SCAN`；有任一资产时可以。
- AI 能按缺失建筑逻辑建造或重建 Radar Outpost，且不会因 Radar Outpost 不生产单位而影响普通单位生产。
- Radar Outpost 可被玩家和 AI 攻击，死亡后从实体、小地图、迷雾和支援资产判断中移除。
- `SKRM` 后 Radar Outpost、pending 建造、支援冷却、迷雾和 AI 相关状态无残留。

架构标准：

- 不新增平行建造、迷雾、支援技能或 AI 系统。
- 玩家和 AI 共享同一套 `EntityKind`、`GameEntity`、施工、迷雾、支援资产和攻击逻辑。
- `isOperational` 是 Radar Outpost 生效边界。
- 战争迷雾仍以 `visibleTiles`、`exploredTiles`、`supportRevealTiles` 和 `isKnownToFaction(...)` 为边界。
- 潜艇 `revealedUntil`、声呐检测和开火暴露不被 Radar Outpost 改动破坏。
- HQ 胜负、任务阶段、经济收入和单位生产列表不变。

文档标准：

- `README.md` 只记录已实现、可验证的 Radar Outpost 行为。
- `md/flow/flow.md` 和必要的 `md/flow/flowchart.md` 与当前真实代码一致。
- `update_log.md` 不由 Agent B 预写未验收结果；Agent C 验收通过后再追加 v1.1 正式记录。

验证标准：

- 本地 `git diff --check` 通过。
- commit 在 `main` 上，commit subject 为 `v1.1: 引入雷达前哨`。
- `git push origin main` 成功触发 GitHub Actions。
- Agent C 能下载最新 `origin/main` run 的未加密 artifact，并核对 manifest / JUnit / xcodebuild.log / failure summary / `.xcresult`。
- 云端 workflow conclusion 成功，manifest 中 `buildOutcome=success`，才算 Swift 代码最低验证通过。

建议人工/模拟器可用时补充 Stage Regression：

- 通过 `BASE` 轮换并建造 Radar Outpost，确认未完工不扩视野、完工后扩视野。
- 在没有 HQ 但有 operational Radar Outpost 时确认 `SCAN` 可用；没有 HQ/RAD 时确认 `SCAN` 不可用。
- 摧毁 Radar Outpost 后确认视野和 `SCAN` 资产失效。
- 观察 AI 是否会补建 Radar Outpost，且仍继续生产混合单位和进攻。
